# 智能协同-第一层 Webhook 简单机器人-实现提示词

## 交接对象

请交给一个新的 Codex 全栈开发线程执行。

## 任务目标

在不破坏造价智算现有 Excel 填价、三数字匹配、结果预览、经验池预警、工作量抓取、问问智算、Word 报告、绿色版和 Tauri 桌面版能力的前提下，实现“智能协同”P1 的第一层：

1. 在左侧一级功能菜单中新增“智能协同”，固定放在“知识库问答”下方。
2. 实现一个独立的“智能协同”中间功能页，右侧问问智算继续保留。
3. 第一层只接入飞书群自定义机器人 Webhook，用于向指定群单向推送测试、开始、进度、失败、完成和结果摘要通知。
4. 完成通知可以提供“进入造价智算”URL，但卡片按钮只允许跳转，不实现服务端交互回调。
5. Webhook 失败、未配置或被禁用时，现有造价智算主流程必须继续正常运行。

本线程**只实现第一层 Webhook 简单机器人**。不要实现第二层企业自建应用机器人，不要接收飞书消息、文件和用户事件，不要开发多维表格台账、风险派单、审批、用户身份映射和全功能机器人。

## 已确认的产品口径

- 对外模块名：`智能协同`。
- 优先级：P1。
- 页面位置：左侧一级功能菜单“知识库问答”下方。
- 建设顺序：先做 Webhook 简单通知机器人，验收稳定后再另开线程做复杂全功能机器人。
- 飞书只承担组织协同与流程触达，不能复制或改变造价智算专业匹配逻辑。
- 第一层自定义机器人只能向所在群推送通知，不能响应用户消息、获取组织用户、接收文件或处理卡片回调。

## 推荐实现方案

推荐采用“独立后端适配模块 + 少量 FastAPI 路由 + 独立前端功能页 + 最小业务钩子”的方式。

### 一、后端独立模块

优先新建：

```text
backend/app/feishu_webhook.py
```

该模块只负责：

- Webhook 设置读取和校验。
- 飞书签名计算。
- 文本、富文本或卡片消息构造。
- 使用现有 `httpx` 依赖发送 HTTP POST。
- 超时、限流、非成功返回和网络异常转换。
- 发送记录和脱敏错误摘要。
- 测试消息、任务开始、任务完成、任务失败等通知模板。

该模块不得：

- 导入或修改 `fill_engine.py` 的匹配算法。
- 读取或改写二维知识库、规则表和经验池。
- 重新计算价格、系数、风险等级或报告金额。
- 调用大模型生成通知中的业务数字。
- 把 Webhook 地址、签名密钥或完整响应写入普通日志。

尽量让该模块可以通过依赖注入、MockTransport 或 monkeypatch 独立测试，避免测试真实请求飞书。

### 二、秘密配置与运行配置

Webhook 地址和签名密钥属于秘密，不能写入：

```text
frontend/
config/project-default-settings.json
README 示例中的真实值
Git 仓库
代码存档
发送日志
```

第一层推荐将秘密保存到被 Git 忽略的运行目录，例如：

```text
Codex-Temp/runtime/feishu-webhook-settings.json
```

或者复用项目已经存在且同样不会进入 Git 的安全本地配置机制，但要注意当前绿色版和 Tauri 打包会复制 `.env.local`：如果把群 Webhook 写入 `.env.local`，可能把特定群凭证带入外发包。因此除非明确修改打包策略并验证不会泄露，否则不要默认把飞书 Webhook 放进会随包复制的 `.env.local`。

保存规则：

- 后端保存，前端提交后不得再次读取完整 Webhook URL 和签名密钥。
- GET 状态接口只返回 `configured`、`enabled`、`security_enabled`、通知开关和必要的脱敏状态。
- 允许用户清空 / 替换配置，但更新接口的返回值也不能包含秘密。
- 配置文件缺失、损坏或字段非法时，按“未配置”安全降级，不能影响主程序启动。
- 不把该秘密配置放进 `config/project-default-settings.json`，因为该文件是项目级公开默认配置并会进入交付包。

### 三、建议 API

根据现有 FastAPI 风格确定最终路径，建议保持在独立命名空间：

```text
GET  /api/collaboration/feishu-webhook/status
POST /api/collaboration/feishu-webhook/settings
POST /api/collaboration/feishu-webhook/test
GET  /api/collaboration/feishu-webhook/history
```

要求：

- `status` 不返回 Webhook 地址和签名密钥。
- `settings` 校验 Webhook 必须为飞书官方 Webhook 地址格式；允许配置启用状态、签名密钥和通知类型。
- `test` 只发送一条明确标记为测试的消息。
- `history` 只返回发送时间、通知类型、成功 / 失败、HTTP 状态和脱敏错误，不返回请求秘密和完整敏感正文。
- 不提供“任意文本任意发送”的通用接口，避免把后端变成无约束消息转发器。
- 接口错误使用清晰的 4xx / 5xx 与中文说明，但真实 Webhook 和密钥不得进入异常正文。

### 四、消息签名与安全

按飞书官方自定义机器人签名规则实现：

- 使用当前 Unix 时间戳。
- 以 `timestamp + "\n" + secret` 组成签名字符串。
- 使用 HMAC-SHA256 和 Base64 生成签名。
- 有密钥时发送 `timestamp` 和 `sign`；无密钥时按未启用签名处理。

必须补确定性单元测试，冻结时间并验证签名结果。

同时支持以下防护口径：

- 页面提示用户在飞书端优先启用签名校验。
- 自定义关键词和 IP 白名单属于飞书端配置，页面可以提供说明，但本项目不伪造其状态。
- 通知正文默认不包含完整逐行价格、原始附件、知识库内容和大模型隐私上下文。

### 六、第一层通知接入点

先检查当前主流程真实路由和前端调用链，再选择最少、最稳定的高层接入点。

第一版至少支持：

- 测试通知。
- 任务 / 转换开始通知。
- 批量匹配或主处理完成通知。
- 主处理失败通知。

可选但不要为了凑功能扩大改动：

- 经验池预警完成。
- Word 报告就绪。

接入原则：

- 通知属于旁路、best-effort 能力。
- 先形成原业务响应，再触发通知。
- Webhook 超时、限流或失败不得改变原业务接口的成功 / 失败结论。
- 不得因为 Webhook 失败回滚 Excel、Word、预警或人工修改结果。
- 通知最好异步或通过 FastAPI `BackgroundTasks` 等现有可控机制执行，不能让飞书网络延迟长期阻塞主流程。
- 不引入 Redis、Celery、消息队列、数据库或新微服务来完成第一层。
- 如果当前架构无法安全地从后端统一挂钩，可以在现有前端成功回调后调用专用通知 API，但必须说明取舍，并保证所有业务数字仍来自后端现有结果，不能由前端重新计算。
