## 三、产品目标

知识记忆不是永久保存全部聊天，也不是让大模型自行训练或修改知识库。

本期只实现：

1. 用户主动将有价值的知识库回答、行级解释或复核结论保存为“知识候选”。
2. 候选未经人工确认，不能参与正式回答。
3. 经人工确认后，进入当前项目的“项目知识记忆”。
4. 后续相似问题可以检索当前项目内已确认的知识记忆。
5. 回答必须明确标注“项目记忆”、适用范围、来源和确认人。
6. 项目记忆可以修改候选、提交确认、确认、驳回、撤销或标记疑似失效。
7. 所有关键操作均保留版本和审计记录。

## 四、强制边界

1. 不新增左侧一级菜单。
2. 不新增第二个聊天窗口。
3. 继续使用现有右侧“问问智算”聊天流。
4. “知识记忆”管理使用轻量 Dialog、Drawer 或现有 Dock 内二级入口。
5. 不改变现有快捷指令、普通聊天、强制查库和行级问答的路由优先级。
6. 大模型只能解释和组织语言，不得裁决价格、基价或两个调整系数。
7. 项目记忆不得自动修改以下正式资产：
   - 正式知识库。
   - 结构化计价库。
   - 匹配规则表。
   - 经验池。
   - Excel 填价结果。
8. 不把 `knowledge-qa-index-【codex】.json` 当作知识记忆数据库。它仍然只是可重新生成的检索缓存。
9. 不自动保存普通闲聊，不由大模型自行判断并沉淀知识。
10. 本期不做 WeAct 消息内直接批准知识，只为以后“发送待确认提醒和网页安全链接”预留能力。
11. 知识记忆功能故障时，现有问答、填价、预警、Word、Excel 和智能协同主流程必须继续可用。

## 五、推荐后端结构

新增独立模块，例如：

不要把数据库、审核状态、项目隔离和审计逻辑全部堆入 `backend/app/main.py`。

### 5.1 知识数据字段

每条知识至少包含：

```text
id
project_key
scope_type
task_id
job_id
title
question
conclusion
conditions
exceptions
expires_at
source_type
source_reference
evidence_summary
submitter
confirmer
status
version
created_at
updated_at
confirmed_at
revoked_at
```

`scope_type` 至少支持：

```text
task
project
```

状态至少支持：

```text
candidate
pending
confirmed
rejected
revoked
suspected_stale
```

另建审计记录，保存：

- 创建。
- 编辑。
- 提交确认。
- 确认。
- 驳回。
- 撤销。
- 标记疑似失效。
- 版本变更。

已确认知识不能通过普通编辑直接覆盖历史。更新时创建新版本或保留完整变更记录。

### 5.2 项目范围

必须使用稳定的 `project_key`，不能把所有本地知识默认放入同一个无边界知识池。

优先复用当前任务中可以可靠获得的项目名称或项目标识。

如果现有处理结果不能可靠取得项目名称，则保存候选时必须要求用户确认或填写项目名称，并生成稳定的规范化 `project_key`。

`job_id` 只能作为来源任务定位，不能直接代替长期项目标识。

必须保证：

- 只有 `project_key` 一致的已确认知识可以参与当前项目问答。
- 查询参数不能绕过后端项目范围校验。
- 未绑定项目的知识默认不能参加正式项目问答。

### 5.3 身份和权限

先检查项目是否已有可靠登录或身份体系。

如果当前没有真实身份体系，不要伪造“生产级权限已经完成”。

P0 可以实现：

- 操作人和确认人留痕。
- 项目范围后端隔离。
- 确认角色校验接口。
- 未来企业身份授权钩子。
- 本地试点身份提示。

界面和文档必须明确：当前属于本地试点身份和审计能力，不等于完整的企业级身份认证。真正的企业用户授权后续可接 WeAct 或统一身份体系。

如果无法真正实现“未授权用户隔离”，不要把需求标成完全完成，应在 PRD 中保留真实状态和限制。

## 六、API 建议

可以根据现有 FastAPI 风格调整具体命名，但必须覆盖以下行为：

```text
POST   /api/knowledge-memory/candidates
GET    /api/knowledge-memory/items
GET    /api/knowledge-memory/items/{id}
PATCH  /api/knowledge-memory/items/{id}
POST   /api/knowledge-memory/items/{id}/submit
POST   /api/knowledge-memory/items/{id}/confirm
POST   /api/knowledge-memory/items/{id}/reject
POST   /api/knowledge-memory/items/{id}/revoke
POST   /api/knowledge-memory/items/{id}/mark-stale
GET    /api/knowledge-memory/items/{id}/audit
```

必须保证：

1. `candidate`、`pending`、`rejected`、`revoked`、`suspected_stale` 不参与正式问答。
2. 只有 `confirmed` 且 `project_key` 一致的知识可以检索。
3. 所有状态变更记录操作人、时间和原因。
4. 非法状态流转必须拒绝并返回明确错误。
5. 已确认知识的删除优先采用撤销和审计留痕，不直接物理删除。
6. API 不得返回其他项目的知识内容、标题、来源或是否存在等旁路信息。

## 七、接入现有知识问答

保留现有接口：

```text
POST /api/knowledge/search
POST /api/knowledge/ask
```

不要重写或替代现有正式知识库检索。

建议增加独立方法：

```python
search_confirmed_project_memory(...)
```

在 `/api/knowledge/ask` 中根据请求携带的 `project_key`，补充检索当前项目已确认的知识记忆。

回答依据优先级必须是：

```text
正式标准与规则
＞ 结构化计价库
＞ 已确认的项目知识记忆
＞ 仅作参考的历史案例
＞ 大模型通用解释
```

要求：

1. 正式知识和项目记忆分区展示。
2. 项目口径不得伪装成国家、行业或企业正式标准。
3. 只有项目记忆、没有正式知识时，可以基于项目记忆回答，但必须明显标注。
4. 正式依据和项目记忆同时命中时，正式依据排在前面。
5. 候选、待确认、驳回、撤销和疑似失效知识不能进入大模型提示词。
6. 不允许大模型自行修改知识状态。
7. 对过期、人工标记冲突或具有确定性失效证据的项目记忆，转为 `suspected_stale`。
8. 不要把“同时命中正式依据和项目记忆”简单等同于冲突。
9. 无法确定冲突时，只分区展示并提示人工复核，不让模型猜测。
10. 如果没有正式依据和项目记忆，继续使用固定无依据提示，并且不要调用大模型。

## 八、前端交互

继续使用现有右侧“问问智算”。

### 8.1 保存为知识候选

在以下回答下方增加“保存为知识候选”：

- 强制知识库回答。
- 自动知识库回答。
- 行级知识库复核结果。
- 有明确来源的复核结论。

普通自由聊天默认不显示该按钮。

保存时打开 Dialog，并尽量预填：

- 标题。
- 原问题。
- 待确认结论。
- 依据来源。
- 当前 Sheet 和行号。
- 当前任务 `job_id`。
- 当前项目名称或 `project_key`。
- 适用条件。
- 例外情况。
- 提交人。

保存前必须让用户看到即将沉淀的内容、来源和项目范围，禁止无提示自动保存。

### 8.2 知识记忆入口

在问问智算 Dock 中增加轻量“知识记忆”入口，至少支持：

- 已确认。
- 待确认。
- 疑似失效。
- 已驳回和已撤销历史。
- 关键词查询。
- 查看详情和来源。
- 查看审计记录。
- 编辑候选。
- 提交确认。
- 确认。
- 驳回。
- 撤销。

不要让管理面板长期挤占主表格宽度。

### 8.3 回答展示

命中项目知识记忆时，回答必须展示：

- “项目记忆”标签。
- 所属项目。
- 确认人。
- 确认时间。
- 适用条件。
- 来源入口。
- 纠错或撤销入口。

为 `ChatMessage` 增加必要的结构化元数据，不要从最终显示文本中反向解析问题、来源和项目范围。

保持现有右侧 Dock、大尾巴主题和 Navattic 极简 SaaS 视觉规范，不新增卡片套娃、常驻阴影或高饱和状态块。
