# 造价智算产品总览 PRD

## 产品定位

“造价智算”是面向工程造价业务的辅助智能体，当前核心场景是长输管道勘察测量最高投标限价编制。产品以结构化规则、结构化计价库、经验池预警和大模型解释能力为基础，帮助用户完成 Excel 填价、风险复核、报告生成和业务问答，并以“一套底座、多个专业 Skill”作为横向扩展架构。

产品不是单点填表脚本，也不是泛用聊天机器人，而是嵌入工程造价生产链条的专业智能体。当前将“勘察测量最高投标限价编制”作为首个已验证专业 Skill，验证“标准可调用、规则可执行、经验可预警、过程可留痕、成果可复核”的方法；后续通过新增专业能力包复制到通用服务类、估概算审核、结算审核、变更签证预算审核、数字与信息化项目投资计价等强规则、重表格、需解释、需留痕场景。

## 业务痛点

- 标准链长：收费标准、测绘定额、项目规则、历史说明分散，依据查找和口径统一成本高。
- 数据链散：多 sheet、多行表头、合并单元格、隐藏列、单位差异、公式缓存等表格问题频繁出现。
- 经验链断：历史经验沉淀在个人电脑、旧表批注、口头判断和项目样表中，难以组织复用。
- 审核链弱：填价依据、匹配过程、异常原因、经验偏离和复核意见不易留痕。

## AI 架构原则

造价智算采用“业务工作台 + 专业知识库 + 规则智能体 + 大模型协同 + 人工复核闭环”的混合 AI 架构。

- 凡是工程造价数字裁决，优先采用可验证的结构化规则。
- 凡是解释、追问、总结、提示和报告，交给大模型增强协同。
- 凡是规则不足、依据不充分、字段冲突或经验偏离，进入人工复核闭环。
- 当前原型先保障本地助手业务闭环；推广阶段再将价格库、知识库、规则 skill、AI 复核 skill、风险报告 skill 封装为服务能力。

## 双交互模式

造价智算保留两种互补的使用方式，共用同一专业能力、任务、规则和成果：

- 专业工作台模式：左侧选择模块，中间完成表格、预警、报告和精细复核，右侧问问智算随行解释。
- 对话式智能体模式：通过左侧一级入口“智算助手”进入保留左侧导航的宽屏会话区；右侧问问智算也可快捷进入同一工作台，在底部选择专业能力、上传文件并用自然语言驱动任务。

产品目标是“主流程可以只靠聊天完成，复杂核对可以随时进入专业页面”，而不是删除表格工作台或把计算交给大模型。会话中的上传、匹配、预警、报告和下载必须调用现有确定性 API，并显示真实动作、进度和成果。

正式方案见：

```text
00-PRD/01-模块PRD/16-对话式智能体工作台/PRD.md
```

## 专业 Skill 扩展原则

造价智算的横向扩展单元不是一张可替换的“库”，而是完整的“专业能力包（Skill）”：业务场景、输入规范、结构化计价库、规则包、字段映射、风险检查、知识资料、报告模板、标准流程和验收用例共同组成一个专业 Skill。

- 用户侧名称统一为“选择专业能力”，`Skill` 作为辅助标签，避免普通业务用户把它理解成纯技术概念。
- 当前勘察测量最高投标限价编制为首个已上线专业 Skill；通用服务类造价测算作为第二个横向验证方向，未真实跑通前只标记规划中或内测中；“更多造价专业”作为即将开放的扩展占位，表达专业能力将持续增加，但不代表具体业务已经可用。
- 通用底座负责 Excel、任务、规则执行、风险、问答、知识记忆、报告和协同；专业 Skill 只装载自己的专业数据、规则、模板和验收资产。
- 结构化计价库是数据资产，确定性匹配引擎是工具，`SKILL.md` 是工作说明；三者不得混为大模型自由裁决机制。
- v5.12.0 已完成专业能力选择、勘察测量 Skill 清单化、安全 Registry、任务级 `SkillRuntimeContext` 和处理 / 匹配 / 预警 / 知识 / 报告 / 下载运行时路由；测试专用第二 Skill 已证明资产隔离。当前仍只有勘察测量真实业务已上线，第二个真实 Skill 待独立资料和数据，不对外宣称已有多个专业业务上线。

正式方案和验收口径见：

```text
00-PRD/01-模块PRD/15-专业能力包与Skill管理/PRD.md
```

## 核心用户

- 造价业务人员：希望减少重复查价、手工填表、逐行复核和报告整理工作。
- 数字化建设人员：希望沉淀可复用的规则资产、知识库资产和智能体能力。
- 评审或参赛人员：希望快速看懂产品价值、业务流程、技术边界和可演示成果。

## 核心流程

1. 选择专业工作台或对话式智能体入口。
2. 选择并确认本次任务使用的专业能力（Skill）。
3. 上传该 Skill 支持的标准 Excel，或在会话中附加文件并说明任务目标。
4. 程序读取表2、表3、表4等关键 sheet。
5. 根据当前 Skill 运行时上下文的结构化计价库和第一层规则匹配基价、实物工作费调整系数、技术工作费调整系数。
6. 第一层未命中时，读取经验提示层并标识为经验参考。
7. 输出填价后的 Excel、表格预览、匹配摘要和 Word 报告，并保留 Skill ID 与版本追溯。
8. 用户可在工作台或会话中触发经验池预警、问问智算解释、行级 AI 复核和成果下载。

## 三链融合

| 链条 | 沉淀对象 | 智能体动作 | 管理价值 |
| --- | --- | --- | --- |
| 数据链 | 价格库、知识库、经验池、运行日志、风险案例 | 采集、清洗、匹配、检索、沉淀、迭代 | 让标准依据、历史数据和项目结果持续反哺当前业务 |
| 角色链 | 编制人、复核人、规则维护人、管理者、AI 助手 | 分工协同、人工兜底、权限分层、责任留痕 | 把人机协同关系讲清楚，避免 AI 越权或责任悬空 |
| 业务链 | 限价编制、依据说明、异常复核、成果报告、推广评价 | 流程编排、任务驱动、报告输出、指标评价 | 把 AI 嵌入真实造价流程，支撑过程遵从和闭环管理 |

## 产品边界

- 大模型只做解释、问答、报告润色和候选提示，不直接裁决最终单价或系数。
- 基价和系数匹配优先采用可审计的结构化规则。
- 原始资料、标准文件、测试输入和答案表不直接改动。
- 经验池当前主要用于预警、经验提示和复核，不应偷偷改变主匹配规则。

## 成功标准

- 用户能用一个标准 Excel 生成可打开、可复核、可报告的结果文件。
- 三个核心数字的来源清楚、颜色状态清楚、逐行说明清楚。
- 规则资产可以被业务人员和其他 AI 理解。
- 参赛展示时能体现“规则链、数据链、智能体链”协同，而不是单点脚本工具。

## 价值指标

- 提效率：减少查表、匹配、复核线索整理和报告生成的重复劳动。
- 提质量：减少因人员经验差异、表格格式差异和临时查找造成的错漏。
- 提感知：让用户看到“填了什么、从哪里来、按什么规则、为什么待复核、与经验值偏离多少”。
- 控风险：把无依据、多候选、单位冲突、经验偏离等问题前移到编制过程。
- 强创新：从高频、高规则、高责任的小切口沉淀数据资产、规则资产、能力资产和治理资产。

## 推广路线

1. 先把勘察测量最高投标限价编制沉淀为首个可选择、可追溯、可验证的专业 Skill。
2. 以通用服务类造价测算建设第二个真实专业 Skill，验证底座、专业资产和任务隔离可以横向复用。
3. 再扩展到估概算审核、结算审核、变更签证预算审核、数字与信息化项目投资计价等相邻场景。
4. 成熟后接入统一数字平台、大模型底座和数据中台，形成集团级造价 AI 能力组件、专业 Skill 和造价数字员工能力包。

推广原则为：先规则后模型，先助手后平台，先闭环后扩张，先可评估后规模化。
