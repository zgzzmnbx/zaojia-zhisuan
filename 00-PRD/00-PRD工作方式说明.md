# PRD 工作方式说明

## 目标

本目录用于让“造价智算”后续开发从聊天驱动，逐步转为轻量 PRD 驱动。PRD 只写当前要稳定传承的产品意图、模块边界和验收口径，不写完整历史流水账。

## 目录规则

- `00-产品总览.md`：写项目定位、用户、核心流程、能力边界。
- `01-模块PRD/`：按功能模块分文件夹，每个模块维护一个 `PRD.md`。
- `02-当前版本计划.md`：写当前版本准备做什么、暂不做什么、验收重点。
- `03-整体UI设计PRD.md`：写全局 UI、布局、主题、表格预览、AI 助手和设置入口原则。
- `03-UI设计规范.md`：写可执行的色彩、字号、间距、组件、表格、AI Dock 和验证规范。
- `PRD-Claude建议包-2026年7月3日/`：外部 Claude Code 建议包和吸收评估报告入口，只作为候选思路资料库，不是正式 PRD，也不是自动开发任务单。
- `../01-assets/01-UI参考图/`：放 UI 优化相关素材、截图、参考设计、技术笔记和实验代码，不放生产代码。
- `05-线程交接提示词/`：只放需要跨线程复制的任务提示词；实现完成后保留为历史交接记录，不作为当前待办。
- `archive/`：PRD 修改前的旧版备份，日常开发默认不读取。

## 使用规则

- 大功能、新模块、参赛展示优化、业务规则变化，先补 PRD 再开发。
- 外部 AI 建议包必须先经过人工 / Codex 评估，再按“近期吸收、需确认、远期储备、禁止或暂缓”进入正式 PRD；不得绕过本目录直接开发。
- 小修小补、纯 bug 修复、临时排查，可以只更新 README 或 CHANGELOG。
- PRD 备份改为手动触发：只有用户明确要求“备份 PRD”时，才运行 `python tools/archive_prd.py` 或双击 `备份PRD.bat`；日常修改 PRD 不自动备份。
- 日常接手默认先读 `AGENTS.md`、`README.md`、`00-PRD/02-当前版本计划.md`。
- 涉及具体模块时，再读对应模块的 `PRD.md`。
- 涉及整体界面、主题、布局、参赛展示和 AI 助手体验时，再读 `03-整体UI设计PRD.md`。
- 涉及 UI 参考图、Bug 截图、竞品截图、临时设计稿、界面分析和小样实验时，统一放入根目录 `01-assets/01-UI参考图/`：
  - `01-assets/design/`：设计稿、过程稿、界面方案图。
  - `01-assets/bug/`：报错截图、视觉问题截图、复现截图。
  - `00-reference/`：外部参考设计、竞品截图、灵感图。
  - `02-notes/`：技术经验、踩坑记录、参考图分析、UI 复盘。
  - `code/`：UI demo、静态原型、临时实验代码；正式生产代码仍放 `frontend/`、`backend/`、`src-tauri/`。

## 状态标记

PRD 中的需求清单统一使用以下状态，避免恢复复杂任务看板：

- `[已完成]`：已经实现，并经过基本验证。
- `[进行中]`：当前版本正在做，可能还没完全验收。
- `[待开发]`：明确要做，但还没开始。
- `[暂缓]`：方向认可，但当前版本暂不做。
- `[不做]`：明确排除，避免后续反复讨论。

每个模块 PRD 应保留一个精简 `## 需求清单`，只写当前仍有指导意义的 5 到 10 条需求。历史完成流水账进入 CHANGELOG，不写进 PRD。

## 关联资产

每个模块 PRD 应保留一个精简 `## 关联资产`，只引用本模块依赖的模板、规则、知识库、配置和关键代码路径。真实文件不移动进 PRD，避免破坏程序路径和代码存档边界。

关联资产表建议使用以下列：

| 类型 | 文件 | 用途 |
| --- | --- | --- |
| 知识库 | `相对路径` | 说明该文件在模块中的作用 |

## PRD 与其他文件分工

- PRD：写“想做成什么样”和“怎么验收”。
- README：写“当前项目是什么状态、入口在哪里、细节去哪里看”，不承载模块细节流水账。
- CHANGELOG：写“已经实际完成了什么”。
- 规则文档：写“匹配规则、知识库口径和业务依据”。
- AGENTS：只沉淀长期协作规则、长期项目约束和容易反复踩坑的经验，不写一次性过程记录。
