# 近期功能优先级与版本路线

**生成日期**：2026-07-03  
**基于版本**：造价智算 v5.3.7  
**性质**：外部建议，供 Codex 后续消化吸收，不直接污染当前 PRD

---

## 一、版本路线设计原则

1. **不超过 8 个近期功能**：避免功能膨胀，保证每个版本有清晰验收标准。
2. **不把所有功能都塞进近期**：远期储备功能明确标注"本阶段不做"。
3. **优先保证当前演示稳定、规则可信、输出可核验**：不因新功能破坏已有能力。
4. **新功能应复用现有能力**：不重新发明规则引擎、读表引擎、报告生成。
5. **遵守项目核心原则**：结构化规则裁决，大模型解释辅助，人工复核兜底。

---

## 二、V1：近期最小增强版（v5.4.x → v5.5.x）

### 版本目标

在不破坏当前核心流程的前提下，完成"质量底座强化 + 演示体验优化"，为后续辅助填价和结算审核助手打好基础。

### 建议功能清单

| 功能 | 优先级 | 复用度 | 风险 | 验收标准 |
|---|---|---|---|---|
| **知识库质量检测** | P1 | 高 | 低 | 输出质量检测报告（重复行、价格冲突、经验数格式错误、空值统计），命令行工具可跑通 |
| **经验池治理（数据质量检测）** | P1 | 高 | 低 | 输出经验池治理报告（重复记录、疑似异常值、单位不一致、空值统计），不自动删除 |
| **风险报告增强（结构化风险清单）** | P1 | 高 | 低 | 后端输出结构化风险 JSON（类型/等级/sheet/行号/说明/建议），前端风险卡片列表可展示与跳转 |
| **演示/评委模式优化** | P1 | 高 | 低 | 一键加载演示样例，按步骤引导提示（上传→转换→预警→问答→输出），关键结果高亮 |
| **标准依据追溯增强（MVP）** | P1 | 高 | 中 | 规则表增加"依据出处"列（手工维护），行级详情展示出处，缺失时显示"暂无标准出处映射" |

### 不建议做什么

- ❌ 不做辅助填价（V2 再做，V1 先打底座）
- ❌ 不做结算审核助手（V2 再做，V1 先做风险报告增强底座）
- ❌ 不做规则维护后台编辑态（只做只读查看器或不做）
- ❌ 不做多项目对比分析
- ❌ 不强行完成 Tauri 统信安装包

### 技术关键点

1. **知识库质量检测**：复用现有知识库读取、归一化、A/B 匹配判定逻辑，做成 `tools/check_knowledge_quality.py` 脚本。
2. **经验池治理**：复用经验池读取、要素匹配模式、IQR/标准差统计方法，输出治理报告 Excel。
3. **风险报告增强**：定义统一 schema（`RiskItem { type, level, sheet, row, column, description, suggestion, canJump }`），前端复用"跳到表格"定位能力。
4. **演示模式**：前端为主，新增"演示模式"开关，预置样例路径，步骤引导用 `localStorage` 记录进度。
5. **依据追溯**：`technical_fee_rules.xlsx` 新增"标准依据"列，行级详情 API 返回该列值，前端展示。

### 验收标准

- 质量检测脚本能检出测试库中的已知冲突行和异常值。
- 经验池治理报告能检出测试经验池中的已知重复和异常值。
- 风险报告结构化 JSON 能被前端解析并展示为卡片列表，点击可跳转。
- 演示模式能在 3 分钟内完成"上传→转换→预警→问答→输出"引导。
- 依据追溯能在行级详情中显示规则表的"标准依据"列内容（至少覆盖 50% 核心规则）。

### 对现有 PRD 的影响

- 新增 `00-PRD/01-模块PRD/09-质量检测与数据治理/PRD.md`（包含知识库质量检测、经验池治理）
- 更新 `00-PRD/01-模块PRD/04-问问智算AI助手/PRD.md`，补充风险报告结构化 schema
- 更新 `00-PRD/01-模块PRD/08-参赛展示与申报材料/PRD.md`，补充演示模式需求

### 给 Codex 的开发顺序建议

1. 先做知识库质量检测（最低风险，纯工具脚本）
2. 再做经验池治理（复用检测脚本模式）
3. 再做风险报告增强（为 V2 辅助填价和结算审核助手铺路）
4. 最后做演示模式和依据追溯（参赛节点前冲刺）

## 三、V2：业务能力扩展版（v5.6.x → v6.0.x）

### 版本目标

在 V1 质量底座和风险报告结构化基础上，完成"辅助填价 MVP + 结算审核助手 MVP"，验证"AI 建议、人工确认"的可信协同模式。

### 建议功能清单

| 功能 | 优先级 | 复用度 | 风险 | 验收标准 |
|---|---|---|---|---|
| **辅助填价（MVP）** | P1 | 高 | 中 | 用户选中待复核行，打开辅助填价窗口，显示候选价格/系数（来源/匹配理由/相似度/风险提示），用户确认后写入输出 Excel 副本，留存操作记录 |
| **结算审核助手（MVP-风险预警型）** | P1 | 高 | 中 | 输入结算 Excel + 原控制价 Excel，输出结构化风险清单、审核问题清单（复用经验池比对逻辑），不做自动审减 |
| **人工复核闭环（MVP）** | P1 | 高 | 中 | 待复核行可标记"已确认/已修改/挂起+备注"，显示复核进度（N/M），复核记录写入 JSON 留痕 |
| **审核意见草拟（配合结算审核助手）** | P2 | 高 | 中 | 基于结构化风险清单生成审核意见草稿 Word，每条意见含问题描述/依据/位置/建议，显式标注"草稿，需人工审定" |

### 不建议做什么

- ❌ 不做结算自动审减（只做风险预警和问题清单，不做裁决）
- ❌ 不做审核问题库（V3 再做，V2 先验证审核助手流程）
- ❌ 不做造价指标库（V3 再做，V2 先验证行级辅助能力）
- ❌ 不做多项目对比分析
- ❌ 不做规则维护后台编辑态

### 技术关键点

1. **辅助填价窗口**：前端新增弹窗组件，调用 `/api/fill-assist/candidates` 返回候选项（来源：二维库相似匹配 / 经验池均值 / 历史项目同类记录 / 第二层经验提示），后端复用归一化和 A/B 匹配模式并放宽到"相似度匹配"（不直接写值，只做候选）。
2. **结算审核助手**：复用经验池比对逻辑，输入变为"结算表 vs 控制价表"，输出结构化风险 JSON（单价超标 / 工程量异常 / 计算公式错误 / 合同外项），前端展示风险卡片。
3. **人工复核闭环**：新增 `preview-review-status.json`（结构：`{ sheet, row, status, note, timestamp }`），前端待复核列表显示复核状态，复用"跳到表格"定位。
4. **审核意见草拟**：复用 `python-docx` 报告生成，模板化为"审核意见书"模板，每条意见作为一个段落插入，大模型只润色措辞不裁决金额。

### 验收标准

- 辅助填价窗口能列出至少 3 种来源的候选项，相似度由高到低排序，用户确认后写入并留痕。
- 结算审核助手能输出至少 3 类风险（单价超标 / 工程量偏离 / 计算错误），每类至少 1 条明细可跳转。
- 人工复核闭环能显示"待复核 10 条，已确认 3 条，挂起 2 条"进度，重新转换后提示"复核状态已失效"。
- 审核意见草稿 Word 能生成至少 5 条意见，每条包含问题描述/依据/位置/建议，文档页眉标注"草稿"。

### 对现有 PRD 的影响

- 新增 `00-PRD/01-模块PRD/10-辅助填价/PRD.md`
- 新增 `00-PRD/01-模块PRD/11-结算审核助手/PRD.md`
- 更新 `00-PRD/01-模块PRD/02-填价结果预览与输出/PRD.md`，补充人工复核闭环需求

### 给 Codex 的开发顺序建议

1. 先做人工复核闭环（为辅助填价和审核助手的"确认写入"机制打样）
2. 再做辅助填价 MVP（验证"候选推荐 + 人工确认"模式）
3. 再做结算审核助手 MVP（复用经验池比对 + 结构化风险）
4. 最后做审核意见草拟（基于结构化风险，配合审核助手）

## 四、V3：可复制推广版（v6.1.x → v7.0.x）

### 版本目标

完成"横向场景验证 + 治理能力补全 + 平台化预备"，验证"从限价编制到估概算审核"的可复制性，补全经验池治理、审核问题库、造价指标库等数据资产治理能力。

### 建议功能清单

| 功能 | 优先级 | 复用度 | 风险 | 验收标准 |
|---|---|---|---|---|
| **估概算审核扩展（勘察测量费核对）** | P1 | 极高 | 中 | 输入估算/概算 Excel 勘察测量费部分，输出"应为多少 vs 报了多少"核对结果，复用现有知识库和规则表 |
| **审核问题库** | P1 | 高 | 中 | 人工录入或从审核意见草稿确认收录问题记录，审核时提示"历史同类问题"，复用要素匹配找同类 |
| **造价指标库** | P2 | 高 | 中 | 历史项目汇总为单位指标（每公里/每平方公里），复用经验池导入通道，提供指标查询界面 |
| **规则维护后台（只读查看器）** | P2 | 中 | 低 | 展示当前生效规则、来源文件、最近修改时间、规则间一致性检查结果，只读不写 |
| **项目复盘库** | P2 | 高 | 低 | 归档每次转换任务的运行结果（匹配统计/预警统计/人工修改记录/耗时），提供复盘列表和详情页 |
| **统信 UOS 本地网页版验证** | P2 | 中 | 高 | FastAPI 后端 + React 构建产物在统信 UOS 本地运行，完成主流程验证（上传/匹配/预览/下载/报告） |

### 不建议做什么

- ❌ 不做变更签证预算审核扩展（等估概算扩展验证后再做）
- ❌ 不做服务类项目扩展（知识库建设成本高）
- ❌ 不做数字化项目造价扩展（业务成熟度低）
- ❌ 不做多项目对比分析（复杂度高，价值释放有限）
- ❌ 不做 Tauri 统信安装包（本地网页版验证后再决定）

### 技术关键点

1. **估概算审核扩展**：新增"核对模式"（比对而非回填），输入两个 Excel（估算表 + 知识库计算参考值），输出偏差清单，复用现有知识库和规则表。
2. **审核问题库**：数据结构类似经验池（要素1-5 + 单位 + 问题类型 + 问题描述 + 处理方式），复用要素匹配找同类，审核时弹出"历史同类问题提示"。
3. **造价指标库**：经验池导入通道读取历史项目，按项目汇总为单位指标（每公里勘察费 = 总勘察费 / 公里数），存为指标库 Excel，提供查询界面。
4. **规则维护后台**：读取所有规则表文件，展示规则清单、字段结构、最近修改时间，运行一致性检查（类似 `check_prd_consistency.py`）。
5. **项目复盘库**：每次转换任务生成运行元数据 JSON（任务 ID / 输入文件 / 输出文件 / 匹配统计 / 预警统计 / 人工修改记录 / 耗时），归档到 `Codex-Temp/runtime/task-history/`，前端列表页展示。
6. **统信验证**：优先解决路径兼容（`pathlib.Path`）、依赖安装（`requirements.txt` / `package.json`）、启动脚本（`scripts/start-linux.sh`），不改核心匹配引擎。

### 验收标准

- 估概算审核能输出"应为 X，报了 Y，偏差 Z"核对结果，至少覆盖 10 行测试数据。
- 审核问题库能录入至少 5 条问题记录，审核时能提示"历史同类问题 2 条"。
- 造价指标库能汇总至少 3 个历史项目为指标（每公里勘察费 / 每平方公里测绘费），提供查询界面。
- 规则维护后台能展示所有生效规则表，显示字段结构和最近修改时间，运行一致性检查报告。
- 项目复盘库能归档至少 5 次转换任务，列表页展示任务 ID / 输入文件 / 匹配率 / 耗时，点击可查看详情。
- 统信本地网页版能完成主流程（上传/匹配/预览/下载/报告），后端和前端无崩溃。

### 对现有 PRD 的影响

- 新增 `00-PRD/01-模块PRD/12-估概算审核扩展/PRD.md`
- 更新 `00-PRD/01-模块PRD/03-经验池预警/PRD.md`，补充审核问题库和造价指标库需求
- 更新 `00-PRD/01-模块PRD/09-质量检测与数据治理/PRD.md`，补充规则维护后台和项目复盘库需求
- 更新 `00-PRD/01-模块PRD/07-桌面端与评委版/PRD.md`，补充统信本地网页版验证需求

### 给 Codex 的开发顺序建议

1. 先做估概算审核扩展（验证横向可复制性）
2. 再做审核问题库（配合结算审核助手）
3. 再做造价指标库（数据资产治理）
4. 最后做规则维护后台和项目复盘库（治理能力补全）
5. 统信验证作为独立技术任务，与业务功能并行推进

## 七、不能破坏的现有能力

每个版本开发前和完成后，必须验证以下能力仍然可用：

1. ✅ 上传标准 Excel → 识别表2/表3/表4 → 匹配三数字 → 输出 Excel
2. ✅ 第一层绿、第二层黄、未命中红的颜色状态
3. ✅ 经验池预警手动触发、预警参数和细节写回
4. ✅ 问问智算知识库检索、行级 AI 复核、风险报告
5. ✅ 原始工作量抓取
6. ✅ Word 报告生成、报告金额与费用汇总一致
7. ✅ 大尾巴主题唯一主界面、左中右三栏布局
8. ✅ 结果预览人工修改、重算公式
9. ✅ 评委绿色版、Tauri 桌面壳 MVP

## 九、给 Codex 的最终建议

1. **不要把所有建议都当作立即任务**。V1/V2/V3 是建议路线，不是强制路线。用户明确要求哪个功能，再启动对应版本。
2. **优先完成 V1 的质量底座**。知识库质量检测、经验池治理、风险报告增强是后续所有功能的基础，且风险低、复用度高。
3. **辅助填价和结算审核助手应分阶段**。不要一次做完整版，先做 MVP 验证"AI 建议、人工确认"模式可行，再做增强版。
4. **横向场景扩展应等结算审核助手验证后再做**。估概算审核、变更签证审核都是相邻场景,但知识库建设成本不低，先做一个验证可复制性，再批量扩展。
5. **每个版本完成后，运行回归测试验证现有能力未破坏**。新功能不能以牺牲已有能力为代价。
