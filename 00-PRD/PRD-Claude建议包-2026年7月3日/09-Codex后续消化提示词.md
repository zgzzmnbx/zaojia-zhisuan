## 二、阅读顺序

### 第一次接手建议包

1. **先读** `00-README-本次Claude建议包说明.md`（本建议包的边界与定位）
2. **再读** `01-项目理解与现状判断.md`（确认 Claude 是否理解项目核心）
3. **然后读** 本文件剩余部分（消化方法）

### 如果用户明确要求开发某功能

- 要求"开发辅助填价" → 读 `04-辅助填价功能PRD建议.md`
- 要求"开发结算审核助手" → 读 `05-结算审核助手PRD建议.md`
- 要求"准备统信迁移" → 读 `06-统信UOS迁移预备建议.md`

### 如果用户要求"看看有什么建议"

1. 读 `02-未来功能发散清单.md`，浏览功能机会地图（四象限）
2. 读 `03-近期功能优先级与版本路线.md`，了解 V1/V2/V3 路线
3. 与用户讨论：从高价值低复杂度象限挑 2-3 个功能确认需求

### 如果用户要求"优化 UI"

读 `08-UI优化PRD建议.md`，结合现有 `00-PRD/03-整体UI设计PRD.md` 和 `00-PRD/03-UI设计规范.md` 提炼优化清单。

## 三、如何判断哪些建议适合吸收

### 适合吸收的信号

1. **用户明确要求**："我想要辅助填价功能"——立即吸收对应 PRD。
2. **PRD 已有待开发条目**：如"经验池治理"在 `00-PRD/01-模块PRD/03-经验池预警/PRD.md` 已有 `[待开发]` 条目，Claude 建议与之吻合——可吸收。
3. **当前版本计划已提及**：如"演示模式"在 `00-PRD/02-当前版本计划.md` 的"本版待确认"中出现过——可讨论吸收。
4. **高价值低复杂度且用户认可**：如"知识库质量检测"属工具脚本、复用度高、风险低——可提议吸收。

### 不适合吸收的信号

1. **用户未要求且不在当前版本计划内**：如"多项目对比分析"——暂缓，不吸收。
2. **复杂度高且业务验证不足**：如"服务类项目扩展"——远期储备，不吸收。
3. **与项目核心原则冲突**：任何建议让大模型直接裁决价格/系数/审减金额——**明确拒绝**。
4. **需要大量新资产建设**：如横向场景扩展需要新知识库——等当前场景做透再考虑。

## 四、如何把 Claude 建议转化为正式 PRD

### 转化步骤

1. **确认需求真实性**：与用户确认"你真的需要这个功能吗？解决什么问题？"
2. **读 Claude 建议 PRD**：如 `04-辅助填价功能PRD建议.md`，理解功能定位、核心流程、边界。
3. **提炼 MVP 范围**：Claude 建议通常有 MVP 和进阶版，第一步只做 MVP。
4. **写入正式 PRD**：在 `00-PRD/01-模块PRD/` 中新建模块 PRD（如 `10-辅助填价/PRD.md`），结构沿用现有模块 PRD 模板：
   ```markdown
   # 模块名 PRD
   
   ## 模块目标
   
   ## 需求清单
   | 状态 | 需求 | 说明 | 验收口径 |
   
   ## 关联资产
   
   ## 功能边界
   
   ## 验收口径
   ```
5. **更新版本计划**：在 `00-PRD/02-当前版本计划.md` 的"本版待确认"或"下一版本计划"中列出该功能。
6. **开发**：按正式 PRD 开发，完成后更新状态为 `[已完成]`，并写入 `CHANGELOG.md`。

### 转化边界

- **不要把 Claude 建议原文复制到正式 PRD**：Claude 建议是外部顾问视角，正式 PRD 是项目内部需求，措辞和详细程度应调整。
- **不要把所有建议都纳入正式 PRD**：只把确认要做的功能写成正式 PRD，其他建议留在 `【Claude】/` 作为参考。
- **不要跳过用户确认**：即使 Claude 建议很详细，也必须与用户确认需求后再写正式 PRD。

## 五、哪些内容只能作为远期参考

以下功能建议作为远期储备，不纳入近期版本：

- 多项目对比分析（技术复杂度高）
- 造价指标库（需跨项目数据积累）
- 审核问题库（应在结算审核助手验证后再做）
- 变更签证预算审核扩展、估概算审核扩展、服务类项目扩展（新场景知识库建设成本高）
- 数字化项目造价扩展（业务成熟度低）

这些功能的 Claude 建议文档保留在 `【Claude】/02-未来功能发散清单.md` 中，作为未来版本规划的输入，但不在近期（V1/V2）开发。

## 七、哪些内容禁止直接执行

| 禁止内容 | 原因 |
|---|---|
| 让大模型直接裁决价格、单价、系数、审减金额 | 违反项目核心原则 |
| 推翻当前匹配引擎或规则体系 | 违反项目核心原则 |
| 修改 `00-PRD/`、`README.md`、`AGENTS.md`、`CHANGELOG.md`、现有代码、规则表、知识库 | Claude 建议包不修改现有项目文件 |
| 把主匹配 A/B 放宽到模式 C | 违反既有匹配口径，模式 C 只能用于候选生成 |
| 立即全面迁移到统信 | 当前主线在 Windows，统信只做逐步消除硬伤 |
| 批量重命名中文资产为英文 | 破坏真实路径引用，收益极小 |

## 八、如果要开发辅助填价，第一步怎么做

1. **与用户确认**：真的需要吗？基价/系数哪个字段优先？
2. **读** `04-辅助填价功能PRD建议.md`，重点读：
   - 功能定位（第一节）
   - 核心流程（第三节）
   - MVP 版本（第十五节）
   - 验收标准（第十七节）
3. **确认技术路径**：读 `07-核心技术路线建议-给Codex消化.md` 中"辅助填价"部分，确认：
   - 后端新增 `fill_assist.py`（不改 `fill_engine.py`）
   - 候选来源复用 `knowledge_base.py` 和 `experience_warning.py`
   - 写回复用现有 `/api/preview/cell`
4. **写正式 PRD**：在 `00-PRD/01-模块PRD/10-辅助填价/PRD.md` 新建模块 PRD，内容基于 Claude 建议提炼 MVP 范围。
5. **更新版本计划**：在 `00-PRD/02-当前版本计划.md` 中列入"本版待确认"或下一版本。
6. **开发**：
   - 后端：`backend/app/fill_assist.py` + `POST /api/fill-assist/candidates` + `POST /api/fill-assist/confirm`
   - 前端：`FillAssistDialog` 组件 + 结果预览行级"辅助填价"按钮
   - 测试：`backend/tests/test_fill_assist.py` 覆盖候选生成与写回
7. **验收**：按 `04-辅助填价功能PRD建议.md` 第十七节 MVP 验收 7 条逐一验收。
8. **收口**：更新模块 PRD 状态为 `[已完成]`，写入 `CHANGELOG.md`，运行 `python tools/archive_code.py` 存档。

## 九、如果要开发结算审核助手，第一步怎么做

1. **与用户确认**：真的需要吗？第一阶段只做风险预警（不做自动审减）可接受吗？能否提供 1-2 份结算表样例验证？
2. **读** `05-结算审核助手PRD建议.md`，重点读：
   - 核心判断（第一节）：审核为主，不做自动审减
   - 风险卡片结构（第七节）
   - 第一阶段 MVP（第十一节）
   - 验收标准（第十五节）
3. **确认技术路径**：读 `07-核心技术路线建议-给Codex消化.md` 中"结算审核"部分。
4. **前置依赖**：建议先做"风险报告增强（结构化风险清单）"（见 `02-未来功能发散清单.md` 功能 6），因为结算审核的风险卡片与风险报告共用 `RiskItem` schema。
5. **写正式 PRD**：在 `00-PRD/01-模块PRD/11-结算审核助手/PRD.md` 新建模块 PRD。
6. **开发**：
   - 后端：`backend/app/settlement_review.py` + `POST /api/settlement/review`
   - 前端：左侧导航新增"结算审核"模块 + 风险卡片列表页
   - 测试：`backend/tests/test_settlement_review.py` 构造已知问题样例验证检出
7. **验收**：按 `05-结算审核助手PRD建议.md` 第十五节 MVP 验收逐一验收。
8. **收口**：同上。

## 十、如果要做统信迁移预备，第一步怎么做

1. **与用户确认**：目标机型/架构（x86_64 还是 ARM/LoongArch）？目标环境是否联网？优先级是否低于业务功能？
2. **读** `06-统信UOS迁移预备建议.md`，重点读：
   - 当前阶段策略（第一节）：Windows 主线继续 + 逐步消除硬伤
   - 迁移风险清单（第二节）：10 大类风险
   - 阶段性路线（第四节）：阶段 0（跨平台纪律）→ 阶段 1（最小验证）
3. **阶段 0（现在起持续）**：新增代码守跨平台纪律：
   - 路径一律 `pathlib` + 相对项目根
   - `open()` 显式 `encoding="utf-8"`
   - 不引入 Windows 专属调用
4. **阶段 1（需统信实机）**：
   - 按 `docs/统信UOS兼容性开发说明.md` 命令跑通本地网页版
   - 执行 `06-统信UOS迁移预备建议.md` 第六节"最小验证方案"15 项
   - 产出《统信验证记录》，问题分级 FAIL/WARN
5. **阶段 2（按需）**：修复 FAIL 项 + 制作离线依赖包 + 完善 `scripts/start-linux.sh`。
6. **明确不做**：Tauri Linux / deb / AppImage 打包（暂缓，与现有 PRD 一致）。

## 十一、如何保证不破坏当前主流程

### 开发前

1. **读现有 PRD**：确认功能边界与现有能力的关系。
2. **读现有代码**：确认复用点的真实函数边界（Claude 建议基于代码合并文档，可能与最新代码有出入）。
3. **设计复用方案**：新业务逻辑放新文件，通过 `import` 复用旧模块；**不修改** `fill_engine.py`、`knowledge_base.py`、`experience_warning.py` 的既有函数签名与行为。

### 开发中

1. **只读不改核心模块**：主匹配、经验池预警、工作量抓取的核心逻辑只读不改。
2. **复用归一化与匹配模式**：所有找同类逻辑复用 `normalization.py` + A/B 匹配模式。
3. **复用写回与留痕**：所有写入输出 Excel 副本的操作复用 `/api/preview/cell`，留痕统一 schema。

### 开发后

1. **全量回归测试**：`python -m pytest backend/tests -v` 全部通过。
2. **答案表一致性**：100 行样例输出与答案表逐行一致仍是金标准。
3. **主流程验收**：上传 → 匹配 → 输出 Excel/Word 主流程与现有能力一致。
4. **颜色状态验收**：绿/黄/红三层颜色口径与逐行说明不变。
5. **经验池预警验收**：手动触发、A→B→D 口径、阈值行为不变。
6. **评委版与 Tauri 壳验收**：启动链和主流程不破坏。

## 十二、最后提醒

1. **Claude 建议不是命令**：Codex 应结合用户真实需求、当前版本计划和技术可行性，自主判断哪些建议值得纳入正式 PRD。
2. **不要把所有建议都做**：优先做高价值低复杂度的功能，远期功能作为储备。
3. **不要跳过用户确认**：每个功能立项前必须与用户确认需求、范围、优先级。
4. **不要污染当前 PRD**：Claude 建议保留在 `【Claude】/`，正式 PRD 在 `00-PRD/01-模块PRD/` 中新建或更新。
5. **守住项目核心原则**：结构化规则裁决，大模型解释辅助，人工复核兜底——所有新功能必须服从这一原则。
6. **与 Claude 建议冲突时以现状为准**：Claude 对模块内部结构的描述可能与最新代码有出入，以项目现状为准。
7. **记录差异与待确认项**：如发现 Claude 建议与现状冲突或基于错误假设，可在消化文档中记录差异，或与用户确认假设。
