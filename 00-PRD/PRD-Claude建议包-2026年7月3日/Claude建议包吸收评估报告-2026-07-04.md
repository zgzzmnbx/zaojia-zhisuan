# Claude 建议包吸收评估报告

生成日期：2026-07-04  
评估对象：`00-PRD/PRD-Claude建议包-2026年7月3日`  
评估口径：只评估、筛选和提出吸收方案；不把 Claude 建议直接转成正式 PRD，不执行开发。

## 一、Claude 是否理解当前项目

总体判断：**基本理解准确，可以作为外部建议参考，但不能原样作为版本路线执行。**

Claude 建议包对项目定位、技术栈、核心流程和边界的理解总体到位，尤其清楚认识到造价智算不是普通填表脚本，也不是泛用聊天机器人，而是“结构化规则裁决 + 大模型解释辅助 + 人工复核兜底”的工程造价辅助智能体。

### 关键边界核对

| 边界 | Claude 理解情况 | 评估 |
| --- | --- | --- |
| 造价智算是工程造价辅助智能体，不是普通填表脚本，也不是泛用聊天机器人 | 多次明确表述 | 准确 |
| 当前核心能力是 Excel 填价、三数字匹配、经验池预警、Word/Excel 输出、问问智算 AI 助手 | 文件 01 对现有能力梳理完整，也覆盖工作量抓取和预览人工修改 | 准确 |
| 关键数字由结构化规则、知识库、规则表、经验提示层和人工复核处理 | 明确写出“结构化规则裁决，大模型解释辅助，人工复核兜底” | 准确 |

| 不允许把经验池预警结果反向污染正式知识库 | 明确经验池、辅助填价确认值不自动写入正式知识库 | 准确 |
| 不允许绕过 PRD 直接开发 | 明确 Claude 建议不是正式 PRD，需要用户确认后再转正式 PRD | 准确 |

### 需要注意的偏差

1. Claude 的路线图偏“大而全”，把很多有价值但未确认的功能放入 V1/V2/V3，容易造成版本膨胀。
2. Claude 多次建议“新增模块 PRD”，但当前项目已经有 8 个模块 PRD，新增前必须判断是否真的需要独立模块，不能为了文档完整而膨胀 PRD。
3. 辅助填价建议中提到“模式 C 作为候选生成专用口径”，方向可以讨论，但必须严格隔离，不能外溢到主匹配、经验池预警和第二层经验提示。
4. 结算审核助手价值很高，但需要真实结算表样例验证；没有样例前不宜正式立项。
5. UI 建议的两张效果图有参考价值，但部分界面比当前项目更“业务中台化”，不能直接照图重做。

## 三、建议的下一步开发路线

我不建议照搬 Claude 的 V1/V2/V3 路线。当前项目已经到 v5.3.7，结果预览人工修改、重算公式、工作台 UI 都已经做了一轮；下一步最应该补的是“可信底座”和“低风险可验收能力”，而不是马上开大模块。

### 候选方向比较

| 候选方向 | 近期价值 | 风险 | 最小可验收范围 | 我的判断 |
| --- | --- | --- | --- | --- |
| 知识库质量检测 | 很高。正式知识库直接决定基价/单价和第二层经验提示可信度 | 低，只读检测 | 一个 `tools` 检查脚本，输出重复组合、同组合价格冲突、单位缺失、经验数格式错误、空值统计报告 | 最推荐近期做 |
| 经验池治理 | 高。预警均值和偏离率依赖经验池质量 | 低到中，只要不自动清洗 | 输出重复、异常值、单位不一致、样本离散、空值统计报告 | 第二梯队，可接在知识库质量检测后 |
| 风险报告增强 | 高。能支撑报告、AI、预警、未来结算审核 | 中，需要统一 schema | 从现有待复核 / 预警结果生成结构化风险项，并在前端展示简洁列表 | 推荐，但应在质量检测后做 |

| 辅助填价 MVP | 中高。能补齐待复核处理链 | 中，候选生成容易带偏 | 只做基价/单价待复核行，2 个候选来源，人工确认写入 | 暂缓到质量底座后 |

### 最推荐近期做的方向

**知识库质量检测。**

理由：

1. 它直接服务“规则裁决”的根基，是最符合项目核心原则的增强。
2. 它是只读工具，不会破坏匹配引擎、知识库、经验池和 UI。
3. 它对后续辅助填价、风险报告增强、依据追溯都有前置价值。
4. 它最容易形成清晰验收：给一个有问题的测试知识库，必须检出重复、冲突、格式错误和空值。
5. 它不依赖大模型，不引入新业务场景，不需要 UI 大改。

最小可验收范围：

- 新增一个只读检查能力，输入固定知识库 `03-知识库-二维数据库制作/【数据库】【导入】.xlsx`。
- 输出 Markdown 或 Excel 检测报告，至少包含：
  - 同一 `要素1-5 + 单位` 组合重复记录；
  - 同组合但 H 列价格冲突；
  - 单位为空或关键要素为空；
  - J-M 经验数格式不可解析；
  - 同组合经验数冲突；
  - 知识库总行数、有效行数、问题行数统计。
- 检查只读，不自动修复，不改知识库。
- 报告中给出 Excel 行号，方便人工回到知识库修正。

### 第二推荐方向

**经验池治理。**

理由：

1. 它与知识库质量检测同属数据质量底座，技术模式可以复用。
2. 经验池已经参与预警，治理能让风险提示更可信。
3. 它不改变主匹配，不污染正式知识库，边界清楚。

最小可验收范围：

- 读取现有经验池文件，只输出治理报告；
- 检出重复记录、单位不一致、样本不足、异常离散、空值统计；
- 不自动删除、不自动清洗。

### 暂缓方向

暂缓以下方向：

- **辅助填价 MVP**：方向正确，但要先把候选来源质量和知识库质量检测做稳。否则“候选推荐”可能把脏数据包装成建议。
- **结算审核助手 MVP**：没有真实样例前不立项。它是产品扩展，不是当前主流程小增强。
- **Tauri 统信安装包**：继续暂缓，只保留本地网页版兼容方向。
- **大规模 UI 信息架构重做**：当前 UI 基线健康，只做局部体验增强。
- **造价指标库、多项目对比、估概算/服务类/数字化项目扩展**：有价值，但当前不是最短路径。

## 四、Claude 技术路线风险检查

### 1. 过度重构风险

风险等级：中。

Claude 建议总体强调“新模块复用旧模块，不改核心函数”，这个方向是对的。但它列出的新增模块、API、schema、前端组件较多，如果一次性推进，会把项目从轻量本地助手推向复杂平台雏形。

控制建议：

- 一次只做一个方向；
- 新功能优先脚本化、只读化、旁路化；
- 不为了未来架构提前拆大规模模块；
- 不把 V1/V2/V3 当作正式路线图。

### 2. 大模型越权风险

风险等级：低到中。

Claude 文档明确不让模型裁决价格和审减金额，这是优点。但在“辅助填价”“结算审核意见草稿”里，如果 UI 表达不谨慎，用户可能误以为 AI 给出的候选或草稿就是结论。

控制建议：

- 所有候选值必须有结构化来源；
- 大模型只解释候选差异，不生成候选数字；
- 审核草稿必须标注“草稿 / 参考 / 待人工审定”；
- 金额字段由结构化数据填充，模型润色后必须校验数字一致。

### 3. 污染正式知识库风险

风险等级：中。

Claude 文档原则上不建议污染知识库，但辅助填价、历史人工确认、经验池治理、审核问题库都可能产生“看起来像知识”的数据。若未来处理不当，可能把人工确认值、经验池预警、审核问题直接混入正式库。

控制建议：

- 正式知识库、经验池、人工确认日志、审核问题库必须分库分文件；
- 辅助填价确认值只进入运行留痕或单独历史确认池，不自动进正式知识库；
- 经验池治理只输出报告，不自动改池；
- 任何“沉淀为知识”的动作必须人工确认。

### 4. 把候选建议当成最终裁决的风险

风险等级：中高。

辅助填价中“知识库-相似”“经验池均值”“历史人工确认”都只是候选。如果 UI 使用“推荐”“高可信”太强，用户可能直接点选而不复核。

控制建议：

- 候选按钮用“采用为人工确认值”，不要写“自动填入正确值”；
- 推荐标签只能表达排序，不表达裁决；
- 模式 C 只允许出现在候选生成模块内；
- 每次写入必须留痕，并保留原值与来源。

### 5. UI 改动过大影响演示稳定的风险

风险等级：中。

UI 建议中的任务上下文条、风险抽屉、AI 消息折叠、表格工具条、演示模式都合理，但如果一起做，会改变用户熟悉的工作流。

控制建议：

- UI 每次只改一个局部；
- 优先做 AI 过程消息折叠、表格状态筛选这类低风险点；
- 风险抽屉、演示模式要先确认交互；
- 不恢复旧主题，不做深色大屏，不做整页重构。

### 6. 统信迁移影响 Windows 主线的风险

风险等级：中。

Claude 对统信方向判断准确：先本地网页版，暂缓 Tauri Linux。风险在于如果把统信检查挂入日常流程，或为了兼容性重构大量路径，会拖慢 Windows 主线。

控制建议：

- 统信兼容保持专项任务；
- 日常开发仍以 Windows 主线稳定为优先；
- `check_platform_compat.py` 手动运行，不挂入默认构建；
- 不为统信批量重命名中文资产；
- 不动核心匹配引擎。
