**生成日期**：2026-07-03  
**基于版本**：造价智算 v5.3.7  
**性质**：外部建议，供 Codex 后续消化吸收，不直接污染当前 PRD  
**如需正式立项**：由 Codex 在 `00-PRD/01-模块PRD/` 中新建正式模块 PRD

## 一、核心判断：审核为主，还是填价为主？

### 结论

**审核为主。第一阶段不做自动裁决，不做自动审减定案。用户的倾向是合理的，且与项目现状高度吻合。**

### 判断依据

1. **业务性质决定**：结算审核的本质是"发现问题 + 组织证据 + 形成意见"，最终审减金额具有法律和经济责任属性，必须由人签字定案。工具的价值在于把审核人员从"人肉找问题"中解放出来，而不是替他签字。
2. **项目原则决定**：本项目核心原则是"结构化规则裁决，大模型解释辅助，人工复核兜底"。审减金额定案既不是可完全结构化裁决的对象（涉及合同解释、现场签证真实性等表格外因素），也绝不允许大模型裁决，因此只能"人工兜底"。
3. **资产现状决定**：当前项目已有的可复用资产——经验池预警（偏离比对）、待复核口径（异常标注）、风险报告（问题汇总）、跳转定位（问题定位）——全部是"审核发现"型能力，没有"重算定案"型能力。审核为主的路线复用度最高、风险最低。
4. **填价能力另有归属**：结算场景中如需"这项应该是多少钱"的参考，应复用**辅助填价**模块的候选组织能力（见 `04-辅助填价功能PRD建议.md`），而不是在审核助手里再造一套填价。

## 二、三种模式比较

| 维度 | 模式 A：风险预警型审核助手 | 模式 B：结算自动重算/自动审减助手 | 模式 C：审核工作台 + 风险卡片 + 审核意见草稿 |
|---|---|---|---|
| 核心动作 | 比对结算表与依据，输出风险预警清单 | 按规则重算"应审定价"，自动算审减额 | 模式 A + 问题组织 + 意见草稿 + 复核留痕 |
| 复用现有能力 | 高（经验池预警、待复核口径） | 中（匹配引擎可复用，但需完整重算链） | 高（预警 + 风险报告 + 报告生成 + 跳转定位） |
| 技术复杂度 | 低-中 | 高（公式链重算、取费程序、合同条款结构化） | 中 |
| 风险 | 低（只提示不定案） | **高**（自动审减金额 = 变相裁决，违反项目边界；重算错误将直接产生错误定案倾向） | 低-中（草稿仍需人工定稿） |
| 业务价值 | 中（发现问题） | 表面高、实际危险 | **高**（发现问题 + 组织证据 + 产出物直接可用） |
| 与项目原则契合度 | 契合 | **冲突**（审减金额属于关键判断，不允许自动定案） | 契合 |

### 推荐方案

**推荐模式 C（审核工作台 + 风险卡片 + 审核意见草稿），并以模式 A 作为其第一阶段 MVP。**

即：
- **第一阶段（MVP）**= 模式 A：输入结算资料 → 结构化比对 → 输出风险卡片清单（发现问题）。
- **第二阶段（增强）**= 补全模式 C：审核问题清单管理 + 审核意见草稿 Word（组织证据、形成草稿）。
- **模式 B 明确不做**：不做"自动审定价 / 自动审减金额定案"。允许做的最多是"依据 XX 标准计算的**参考值**"展示（复用知识库核对能力），且必须标注"参考值，非审定结论"。

## 三、业务定位

结算审核助手是造价智算在**结算审核场景**的审核工作台：把结算书与合同价格清单、控制价、知识库标准、经验池历史进行**结构化比对**，发现单价、工程量、计算和范围类问题，组织成**风险卡片**和**审核问题清单**，辅助审核人员生成**审核意见草稿**，最终**由人工审核确认定案**。

一句话：**它是审核员的"问题雷达 + 证据整理员 + 草稿秘书"，不是"审减计算器"。**

## 四、核心用户

| 用户 | 使用方式 | 关注点 |
|---|---|---|
| 审核人员（主用户） | 上传结算资料，逐卡片处理问题，导出意见草稿 | 别漏问题、依据可查、草稿省时 |
| 复核/管理人员 | 查看风险分布和处理进度 | 审核质量、留痕完整 |
| 送审方造价人员（间接） | 参考问题清单自查 | 问题描述清楚、依据明确 |

## 五、适用资料与输入资料清单

### 第一阶段适用范围（建议收窄）

**长输管道勘察测量类项目的结算审核**——与现有知识库、规则表、经验池同域，资产直接可用。其他专业结算暂不承诺。

### 输入资料清单

| 资料 | 必需性 | 用途 | 格式要求 |
|---|---|---|---|
| 结算书 Excel（送审结算表） | 必需 | 核查对象 | 结构化表格，要素列可映射（复用现有列映射机制） |
| 原控制价 / 合同价格清单 Excel | 强烈建议 | 单价一致性比对基准 | 同上 |
| 委托方确认工作量资料 | 可选 | 工程量核对参考 | 复用工作量抓取模块读取 |
| 二维知识库 / 规则表 | 内置 | 标准参考值计算 | 现有资产 |
| 经验池 | 内置 | 历史偏离比对 | 现有资产 |
| 合同条款关键参数（人工录入） | 可选 | 取费口径、下浮率等 | 第一阶段用简单表单录入，不做合同全文解析 |

## 六、核查对象与风险类型

### 核查对象

1. 结算单价（vs 合同清单单价 / 控制价单价 / 知识库标准参考值）
2. 结算工程量（vs 委托确认工作量 / 控制价工程量）
3. 行级金额计算（单价 × 数量 × 系数是否算对，公式兜底能力复用）
4. 汇总金额（分项之和、税率计算）
5. 项目范围（结算表中出现而合同清单中没有的"合同外新增项"）
6. 系数使用（实物/技术工作费调整系数 vs 规则表口径）

### 风险类型定义（结构化规则判定，非大模型判定）

| 风险类型 | 判定规则 | 默认等级 |
|---|---|---|
| 单价高于基准 | 结算单价 > 合同/控制价同类项单价（要素1-5+单位 A/B 匹配对齐） | 偏离≤5% 低 / >5% 高 |
| 单价偏离标准 | 结算单价 vs 知识库标准参考值偏离超阈值 | 同上（阈值可配） |
| 工程量增加 | 结算工程量 > 基准工程量 | 按偏离率分级 |
| 行级计算错误 | 程序重算行金额 ≠ 表内金额（超容差） | 高 |
| 汇总计算错误 | 程序兜底计算汇总 ≠ 表内汇总（超容差） | 高 |
| 合同外新增项 | 结算行在合同清单中无 A/B 同类记录 | 高（需人工判断签证依据） |
| 系数异常 | 系数与规则表口径不符或超经验池偏离阈值 | 按偏离率分级 |
| 经验偏离 | 复用经验池预警逻辑（5%/20% 阈值） | 低/高 |
| 无法比对 | 要素无法对齐、单位不一致 | 提示级（不是"没问题"） |

**重要口径**："未找到可比对基准"必须显式输出为一类提示，不得静默归为"无风险"——沿用经验池预警中"没有预警 ≠ 没有找到同类"的既有口径。

## 七、风险卡片结构

```text
┌──────────────────────────────────────────────┐
│ [高风险] 单价高于合同清单                        │
│ 位置: 表4 第18行  钻探/黏性土/30m以内  单位: m   │
│ 结算单价: 156.00   基准单价: 128.00 (合同清单)   │
│ 偏离: +21.9%      涉及金额影响: 约 +8,400 元     │
│ 依据: 合同清单第37行；知识库序号214 参考值128     │
│ 建议动作: 核对该项是否存在签证或价格调整依据        │
│ [跳到结算表] [跳到基准行] [标记处理] [写入意见]    │
└──────────────────────────────────────────────┘
```

字段 schema（与"风险报告增强"共用一套）：

```text
RiskCard {
  id, type, level(高/低/提示),
  sheet, row, elementSummary, unit,
  currentValue, baseValue, baseSource(合同/控制价/知识库/经验池),
  deviation, amountImpact(结构化计算，标注"影响估算"),
  evidenceRefs[](可跳转定位),
  suggestion(规则模板文案，大模型仅可润色),
  reviewStatus(未处理/已确认问题/已排除/挂起), reviewNote
}
```

**边界**：`amountImpact` 是"该行差额 × 数量"的算术展示，界面必须标注"影响估算，非审减定案"。

## 八、审核问题清单结构

风险卡片经人工"标记处理"后进入问题清单（人工确认问题才算问题）：

| 字段 | 说明 |
|---|---|
| 问题编号 | 自动生成（按类型分组编号） |
| 问题类型 | 同风险类型 |
| 位置与项目特征 | sheet/行/要素摘要/单位 |
| 问题描述 | 规则模板生成 + 人工可改 |
| 核对依据 | 证据引用（基准来源 + 行号 + 标准出处） |
| 涉及金额影响（估算） | 结构化计算，标注估算 |
| 送审值 / 核对参考值 | 只列事实，不写"审定值" |
| 处理建议 | 人工填写或从模板选择 |
| 状态 | 待沟通 / 已确认 / 已排除 |

问题清单可导出 Excel（复用 openpyxl 输出能力）。

## 九、审核意见草稿结构

基于问题清单生成 Word 草稿（复用 `report.py` 的模板生成方式）：

```text
一、审核概况：项目名称、送审金额、核对范围、资料清单（结构化填充）
二、核对方法说明：比对基准、匹配口径、阈值设置（固定模板文案）
三、发现的主要问题：按类型分组，逐条列出问题描述/位置/依据/金额影响估算
四、建议：逐条处理建议 + "以上意见为初步核对意见，最终审定结论以人工审核为准"
页眉/页脚：显著标注【审核意见草稿-待人工审定】
```

**硬性边界**：
- 草稿全文不得出现"审定金额为 X""审减金额为 X"的定案表述；只允许"核对差异金额估算合计 X（供参考）"。
- 大模型只润色问题描述的语句通顺度，金额、行号、依据引用全部由结构化数据填充，润色前后金额字段必须逐一校验一致。

## 十、与其他模块的复用关系与边界

### 与经验池预警模块

- **复用**：找同类逻辑（A→B→D）、偏离率分级、阈值配置模式、"跳到表格"定位。
- **边界**：审核助手的比对基准扩展为"合同清单/控制价"，这是新增的比对通道；经验池仍只做历史偏离参考，不作为审减依据单独成立。

### 与辅助填价模块

- **边界**：审核助手回答"这项**有没有问题**"；辅助填价回答"这项**该填多少的候选**"。审核工作台中如需参考值候选，跳转调用辅助填价的候选服务展示，**不在审核助手内重复实现候选生成**。
- 两者的"人工确认 + 留痕"机制应共用同一套 schema。

### 与右侧 AI 助手

- AI 可以：解释某风险卡片的判定规则和依据、总结当前审核进度、润色问题描述。
- AI 不可以：判定风险等级、计算审减金额、决定问题成立与否、代替用户标记处理。
- 审核场景的行级追问复用现有行级复核上下文机制（携带风险卡片结构化字段）。

## 十一、第一阶段 MVP（模式 A：风险预警型）

**范围**：

1. 新增"结算审核"模块入口（左侧导航新增项）。
2. 上传结算 Excel + 基准 Excel（合同清单/控制价），复用现有列映射设置。
3. 结构化比对产出风险卡片清单：先做 4 类风险——**单价高于基准、工程量增加、行级计算错误、合同外新增项**。
4. 风险卡片列表 + 等级筛选 + 跳转定位（复用现有定位能力）。
5. 风险清单导出 Excel。

**MVP 明确不含**：审核意见 Word 草稿、问题清单状态管理、合同参数录入、知识库标准参考值比对、AI 联动。

## 十二、第二阶段增强版（补全模式 C）

1. 风险卡片"标记处理"（确认问题/排除/挂起 + 备注）→ 审核问题清单。
2. 审核意见草稿 Word 生成。
3. 知识库标准参考值比对通道（复用匹配引擎"核对模式"——只比对不回填，该能力同时为未来估概算审核扩展打底）。
4. 系数异常和汇总计算核查。
5. 右侧 AI 联动：卡片解释、进度总结。
6. 审核问题库雏形（人工确认的问题沉淀为可检索记录，见 `02-未来功能发散清单.md` 功能 3）。

## 十三、暂缓能力

- ❌ 自动审定价 / 自动审减金额定案（**永久边界**，不是暂缓）
- ⏸ 合同全文解析与条款结构化（第一阶段用人工录入关键参数替代）
- ⏸ 签证资料影像识别 / PDF 解析
- ⏸ 多专业结算（先做勘察测量同域）
- ⏸ 多人审核协作与审批流

---

## 十四、风险边界汇总

1. **不定案**：所有金额输出均为"估算/参考"，界面与文档双重标注。
2. **不裁决**：风险等级由结构化阈值判定；大模型只解释和润色，且润色后金额校验一致。
3. **不静默**："无法比对"显式输出，不算"无风险"。
4. **不污染**：审核数据独立存放，不写入正式知识库、经验池模板和主填价规则。
5. **不越域**：第一阶段只承诺勘察测量类结算，表样不适配时明确报错，不强行匹配。

---

## 十五、验收标准

### MVP 验收

1. 用样例控制价表构造一份"结算表"（改高 3 行单价、加大 2 行工程量、新增 1 个合同外项、改错 1 处行金额），系统能全部检出且分类正确、无多报。
2. 每张风险卡片的位置、当前值、基准值、偏离率、金额影响估算与手工计算一致。
3. "跳到结算表/基准行"定位准确（复用既有定位验收口径：切 sheet、滚动、闪烁）。
4. 无法比对的行显式列出数量与原因。
5. 导出的风险清单 Excel 与页面卡片一致。
6. 全程不产生任何"审定/审减定案"字样；现有回归测试全部通过，主填价流程无回归。

### 第二阶段验收

7. 问题清单状态流转和留痕完整；意见草稿 Word 金额与结构化数据逐项一致，页眉含"草稿-待人工审定"。
8. 知识库参考值比对结果标注"参考值，非审定结论"。

## 十六、给 Codex 的技术关键路径

> 以下仅为建议路径，Codex 应结合现有代码结构自行判断。

1. **后端**：新增 `settlement_review.py`（独立模块，不改 `fill_engine.py`/`experience_warning.py` 的现有函数行为，通过导入复用其归一化、A/B 匹配、找同类和公式兜底能力）。
   - `POST /api/settlement/inspect`：读取结算表与基准表结构（复用现有 inspect 模式与合并单元格默认口径）。
   - `POST /api/settlement/review`：执行比对，返回 `RiskCard[]` JSON。
   - `POST /api/settlement/export`：导出风险清单 Excel。
2. **对齐逻辑**：结算行 ↔ 基准行按"要素1-5+单位"A→B 对齐；一对多时列为"无法唯一对齐"提示，不强行选一条。
3. **schema 先行**：`RiskCard` schema 与"风险报告增强"（见 `02-未来功能发散清单.md` 功能 6）统一定义在 `schemas.py`，两个功能共用。
4. **前端**：左侧导航新增"结算审核"模块；中间工作区 = 上传区 + 风险卡片列表（筛选 chip 按等级/类型）；样式走 `daweiba-*`，遵守扁平化规范。
5. **配置**：审核阈值独立 JSON（不与经验池预警阈值混用，允许分别调整）。
6. **测试**：`backend/tests/test_settlement_review.py` 覆盖上述验收场景 1 的构造样例；同时全量回归。
7. **开发顺序**：schema 定义 → 对齐与比对引擎（含测试）→ 风险卡片 API → 前端列表与跳转 → 导出 → （二阶段）问题清单与草稿。

## 十七、假设与待确认项

| 假设 | 待确认 |
|---|---|
| 假设第一阶段限定勘察测量类结算 | 需用户确认首个结算试点项目的资料形态 |
