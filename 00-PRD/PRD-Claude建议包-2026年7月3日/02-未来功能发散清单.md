# 未来功能发散清单

**生成日期**：2026-07-03  
**基于版本**：造价智算 v5.3.7  
**性质**：外部建议，供 Codex 后续消化吸收，不直接污染当前 PRD

---

## 一、发散原则

1. 所有功能必须遵守项目核心原则：**结构化规则裁决，大模型解释辅助，人工复核兜底**。
2. 所有功能不允许让大模型直接裁决价格、单价、系数、审减金额或审核结论。
3. 优先复用现有能力（填价引擎、经验池预警、知识库问答、Word 报告、要素匹配模式），而不是重新发明规则引擎。
4. 每个功能都给出优先级、落地路径、风险边界和最小可行版本判断。

## 三、发散功能清单（13 个新方向）

### 功能 1：造价指标库

| 维度 | 内容 |
|---|---|
| 功能名称 | 造价指标库 |
| 解决什么业务问题 | 历史项目的单位造价指标（如每公里勘察费、每平方公里测绘费）散落在各项目文件中，无法快速调取"同类项目大概花多少钱"的宏观参考 |
| 适合哪个用户 | 造价业务人员（编制前估算参考）、管理者（快速判断报价合理性） |
| 输入是什么 | 历史控制价 / 结算 Excel（复用经验池导入通道）、项目元信息（项目类型、规模、地区、年份） |
| 输出是什么 | 指标库 Excel（单位指标、项目特征、来源项目）、指标查询界面、指标对比视图 |
| 是否复用现有能力 | 高度复用：经验池导入通道、要素1-5+单位匹配模式、二维 Excel 数据库存储方式 |
| 技术复杂度 | 中（需要新增"项目级汇总"逻辑，当前经验池是行级记录；需要定义指标口径） |
| 业务价值 | 高（从"行级预警"升级到"项目级参考"，是造价数字资产的自然演进） |
| 风险边界 | 指标只做参考展示，不参与三数字裁决；指标口径必须显式写明（含税/不含税、是否含系数）；样本量不足时必须提示"样本不足，仅供参考" |
| 建议优先级 | P2（中期） |
| 是否适合近期交给 Codex | 否。需要先积累足够历史项目数据，且需要用户先确认指标口径定义 |

### 功能 2：经验池治理

| 维度 | 内容 |
|---|---|
| 功能名称 | 经验池治理（数据质量检测与清洗辅助） |
| 解决什么业务问题 | 经验池导入的历史数据可能有重复记录、异常值、单位错误、脏数据，污染预警平均值，导致偏离率失真 |
| 适合哪个用户 | 造价业务人员（维护经验池）、规则维护人员 |
| 输入是什么 | 当前经验池 Excel |
| 输出是什么 | 治理报告（重复记录清单、疑似异常值清单、单位不一致清单、空值统计）、清洗建议（不自动删除，人工确认后处理） |
| 是否复用现有能力 | 高度复用：经验池读取逻辑、要素匹配模式、归一化规则 |
| 技术复杂度 | 低-中（重复检测用现有 A/B 匹配模式；异常值检测用统计方法如 IQR / 标准差，纯结构化计算） |
| 业务价值 | 高（经验池是预警的根基，脏数据直接导致预警不可信；且 `00-PRD/01-模块PRD/03-经验池预警/PRD.md` 已有 `[待开发] 经验池数据质量治理` 条目） |
| 风险边界 | 只输出治理报告和建议，不自动删除或修改经验池数据；清洗操作必须人工确认；模板文件仍然只读 |
| 建议优先级 | **P1（近期）**——PRD 已有待开发条目，属于计划内工作 |
| 是否适合近期交给 Codex | **是**。逻辑清晰、复用度高、不碰核心匹配、PRD 已有入口 |

### 功能 3：审核问题库

| 维度 | 内容 |
|---|---|
| 功能名称 | 审核问题库 |
| 解决什么业务问题 | 审核中发现的典型问题（多算工程量、重复计费、系数用错、单价超标）散落在各项目审核意见中，新审核无法快速比对"以前有没有见过类似问题" |
| 适合哪个用户 | 审核人员、复核人员 |
| 输入是什么 | 人工录入或从审核意见草稿中确认收录的问题记录（问题类型、要素特征、问题描述、处理方式、涉及金额） |
| 输出是什么 | 问题库 Excel（按模板维护，类似经验池）、审核时的"历史同类问题提示" |
| 是否复用现有能力 | 复用：二维 Excel 数据库存储方式、要素匹配模式找同类、经验池模板管理方式 |
| 技术复杂度 | 中（数据结构简单，但"问题特征如何结构化"需要业务定义；与结算审核助手强耦合） |
| 业务价值 | 高（是结算审核助手的配套数据资产，审核经验从个人走向组织） |
| 风险边界 | 问题库只做提示，不自动生成审核结论；问题收录必须人工确认；不与正式知识库混淆 |
| 建议优先级 | P2（中期，跟随结算审核助手一起规划） |
| 是否适合近期交给 Codex | 否。应先做结算审核助手 MVP，问题库作为其第二阶段配套 |

### 功能 4：项目复盘库

| 维度 | 内容 |
|---|---|
| 功能名称 | 项目复盘库 |
| 解决什么业务问题 | 每个项目做完后，"这个项目匹配率如何、哪些待复核、人工改了什么、预警了什么"没有归档复盘，无法评估工具效果和沉淀改进方向 |
| 适合哪个用户 | 管理者（评估工具价值）、数字化建设人员（持续改进）、造价业务人员（回看历史项目） |
| 输入是什么 | 每次转换任务的运行结果（匹配统计、预警统计、人工修改记录、输出文件路径） |
| 输出是什么 | 项目复盘记录（JSON / Excel）、复盘列表页、单项目复盘详情（匹配率、待复核数、修改数、预警分布、耗时） |
| 是否复用现有能力 | 高度复用：当前转换已产出匹配统计和摘要，人工修改已有 JSON 留痕，只差"任务级归档与展示" |
| 技术复杂度 | 低-中（主要是把已有运行数据结构化归档，新增一个列表/详情页） |
| 业务价值 | 中-高（对参赛展示很有价值——"用了多少次、省了多少复核时间"是最硬的价值证据） |
| 风险边界 | 只记录运行元数据，不存原始敏感数据副本；不改变任何运行逻辑 |
| 建议优先级 | P2（中期；若参赛需要"使用效果数据"可提前） |
| 是否适合近期交给 Codex | 可以（低风险，但优先级低于经验池治理和辅助填价） |

### 功能 5：标准依据追溯增强

| 维度 | 内容 |
|---|---|
| 功能名称 | 标准依据追溯增强（一键看依据） |
| 解决什么业务问题 | 当前逐行匹配说明能写"命中知识库第 X 行 / 命中规则 Y"，但用户想看"这条规则对应哪份标准文件的哪一条"仍需要自己去翻标准资料 |
| 适合哪个用户 | 造价业务人员、复核人员、评委（演示时展示可追溯性） |
| 输入是什么 | 当前行的匹配结果（命中知识库行号 / 规则编号） |
| 输出是什么 | 依据链视图：知识库行 → 规则条目 → 标准文件原文段落（`计价格[2002]10号` 等 Markdown 已在检索范围内） |
| 是否复用现有能力 | 高度复用：知识库问答的本地检索（`/api/knowledge/search`）、行级复核上下文、标准资料 Markdown |
| 技术复杂度 | 中（需要在知识库 / 规则表中补"依据出处"字段，或建立规则→标准段落的映射表；展示层复用现有证据展示） |
| 业务价值 | 高（"每个数字都能追溯到标准原文"是本产品最核心的可信卖点，演示效果极强） |
| 风险边界 | 追溯链只做展示，不改变匹配结果；映射表人工维护，缺失时明确显示"暂无标准出处映射"，不让大模型编造出处 |
| 建议优先级 | **P1（近期）** |
| 是否适合近期交给 Codex | **是**。第一步可只做"规则表增加依据出处列 + 行级详情展示出处"，工作量可控 |

| 输出是什么 | 结构化风险清单（JSON：风险类型 / 等级 / sheet / 行号 / 说明 / 建议动作 / 跳转定位）+ Word 报告中的结构化风险章节 + 前端风险卡片列表 |

| 建议优先级 | **P1（近期）** |

### 功能 8：多项目对比分析

| 维度 | 内容 |
|---|---|
| 功能名称 | 多项目对比分析 |
| 解决什么业务问题 | 无法把 2-3 个同类项目的控制价放在一起横向对比（同类记录单价差异、总价构成差异），审核和编制都靠人工开多个 Excel 对照 |
| 适合哪个用户 | 审核人员、管理者 |

| 业务价值 | 中-高（审核场景有用，但单项目流程尚未做透时价值释放有限） |

| 建议优先级 | P3（远期） |

## 五、结论

1. **近期最值得做的组合**是：辅助填价（MVP）+ 风险报告增强 + 知识库质量检测 + 演示/评委模式优化。这个组合复用度高、风险低、且直接强化"可信、可核验、可演示"三个当前最重要的目标。
2. **结算审核助手**价值最高但应分阶段：先用"风险报告增强"打底座（结构化风险清单），再做审核工作台。
3. **横向场景扩展**（估概算、变更签证、服务类、数字化）全部建议放在结算审核助手验证之后，避免知识库建设成本失控。
4. 所有功能均不改变"三个数字由结构化规则裁决"的核心边界。
