# 统信 UOS 迁移预备建议

**生成日期**：2026-07-03  
**基于版本**：造价智算 v5.3.7  
**性质**：外部建议，供 Codex 后续消化吸收，不直接污染当前 PRD  
**现有基础**：项目已有 `docs/统信UOS兼容性开发说明.md`、`tools/check_platform_compat.py`、`backend/app/paths.py`、`scripts/start-linux.sh`，本文在其基础上补充，不推翻既有预案

---

## 一、当前阶段策略判断

### 三个候选策略

| 策略 | 内容 | 判断 |
|---|---|---|
| A. 立即全面迁移 | 主线切到统信，Windows 退居兼容 | ❌ 不建议：参赛主线在 Windows，评委版/Tauri 壳/演示环境都是 Windows，切换会拖垮当前节奏 |
| B. Windows 主线继续 + 逐步消除跨平台硬伤 | 日常开发仍在 Windows，新代码守跨平台纪律，按里程碑做统信抽查验证 | ✅ **推荐** |
| C. 暂缓迁移，只做技术预研 | 不动代码，只写调研文档 | ⚠️ 不够：项目已有 `check_platform_compat.py` 和 paths 工具，说明已越过纯预研期；纯预研会让新增代码继续积累硬伤 |

### 结论

**采用策略 B：Windows 主线继续，逐步消除跨平台硬伤，辅以低频统信实机验证。**

这与项目现有口径一致（`00-PRD/02-当前版本计划.md`："不在本阶段强行完成统信 UOS 安装包……统信方向先按本地网页版兼容验证推进"）。本文的增量价值在于：把"逐步消除硬伤"落成可执行清单和分阶段验收。

优先验证形态排序：

```text
1. 本地网页版（FastAPI 托管 React 构建产物）—— 当前唯一近期目标
2. 浏览器直访开发版（前后端分离）—— 开发调试用
3. Tauri Linux 桌面壳 —— 暂缓，条件成熟再启动
4. deb / AppImage 安装包 —— 暂缓
```

## 二、迁移风险清单

### 2.1 Python / FastAPI 跨平台风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| Python 版本差异 | 中 | 统信 UOS（尤其定制版）系统 Python 可能偏旧（3.7/3.8），FastAPI/pydantic 新版有最低版本要求 | 明确项目最低 Python 版本并写入文档；统信侧建议用 venv + 指定版本，不依赖系统 Python |
| 依赖轮子可用性 | 中 | 内网/信创环境 pip 源可能缺 wheel，源码编译（如 pydantic-core 需 Rust）易失败 | 预研阶段在统信实机上执行一次 `pip download -r requirements.txt` 制作离线依赖包 |
| 子进程/信号差异 | 低-中 | Windows 上的进程管理习惯（如 taskkill 思路）与 Linux 信号机制不同 | 检查后端是否有 spawn/kill 逻辑；uvicorn 主链路本身跨平台 |
| 文件编码默认值 | 中 | Windows 默认 GBK，Linux 默认 UTF-8；`open()` 未显式指定 encoding 的读写行为会变 | 全库检查 `open(` 调用显式加 `encoding="utf-8"`（建议加入 `check_platform_compat.py` 检查项） |
| 大小写敏感 | 中 | Linux 文件系统区分大小写，Windows 不区分 | 已有检查脚本覆盖前端 import；建议扩展覆盖后端 Path 字面量与实际文件名的大小写一致性 |

### 2.2 React / Vite 跨平台风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 构建产物本身 | 低 | `frontend/dist` 是静态文件，跨平台无差异 | 可以在 Windows 构建、拷贝到统信运行，降低统信侧 Node 依赖 |
| 统信 Node 环境 | 中 | 如需在统信本机构建，Node 版本和 npm 源是变数 | 第一阶段直接带 dist 产物过去，绕开统信侧构建 |
| 浏览器兼容 | 中 | 统信自带浏览器基于 Chromium（版本可能偏旧），个别新 CSS/JS 特性可能缺失 | 验证时优先用统信自带浏览器测试；记录最低可用 Chromium 版本 |
| 字体回退 | 中 | 前端字体栈 Inter 优先，统信无 Inter 时回退系统字体，数字等宽（tabular-nums）效果依赖字体支持 | 见 2.10 字体风险 |

### 2.3 Tauri 2 在统信 UOS 上的风险（暂缓项，仅记录）

| 风险项 | 等级 | 说明 |
|---|---|---|
| WebKitGTK 依赖 | 高 | Linux Tauri 依赖 webkit2gtk；统信仓库版本是否匹配 Tauri 2 要求需实机确认 |
| WebView 行为差异 | 高 | 当前 `disable_drag_drop_handler()` 是针对 WebView2 的关键配置，WebKitGTK 的拖拽行为完全不同，HTML5 拖拽上传需重新验证 |
| 后端启动链 | 中 | Tauri 壳内嵌的"启动/复用后端"逻辑含 Windows 便携 Python 路径（`runtime/python/python.exe`），Linux 需另一套查找逻辑 |
| 打包生态 | 高 | deb/AppImage 打包、图标、桌面文件、签名都是新工作量 |

**结论**：Tauri Linux 全部暂缓，与现有 PRD 口径一致。本地网页版验证通过前不投入。

### 2.4 openpyxl / python-docx 风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 库本身 | 低 | 两者纯 Python，跨平台良好 | — |
| 中文文件名 zip 内部处理 | 低-中 | xlsx/docx 本质是 zip；中文文件名在 UTF-8 文件系统上正常，但需确认统信文件管理器解压/预览无乱码 | 验证清单覆盖 |
| 输出文件打开验证 | 中 | 统信侧通常用 WPS 打开输出文件；WPS 对 openpyxl 写入的批注、单元格颜色、公式缓存的呈现可能与 MS Excel 有差异 | 验证清单必须包含"WPS 打开输出 Excel：颜色/批注/公式值正确" |

### 2.5 Excel 公式重算在 Linux 下的风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 无 Excel COM | 高（认知风险） | Windows 上"打开 Excel 自动重算"的兜底在 Linux 不存在；不能假设用户装有 MS Excel | 项目已有程序侧公式兜底计算（`formula_resolver.py` 思路），这是正确路线；统信侧要把"程序兜底计算"当作主路径而非兜底 |
| 公式缓存依赖 | 中 | 上传的 Excel 若无公式缓存值，Linux 下无本地 Excel 可"打开刷新" | 明确提示用户"用 WPS 打开另存后再上传"或依赖程序兜底；验收时用无缓存样例测试 |
| WPS 重算差异 | 中 | 用户在统信用 WPS 打开输出文件重算，个别函数行为可能与 Excel 有细微差异 | 限定当前支持的公式集（项目已限定 ROUND/MIN/MAX/四则/区域求和/跨表引用），不扩大承诺 |

### 2.6 中文路径、空格路径、特殊字符路径风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 项目目录名本身 | **高** | 当前项目根目录名为 `260612-【ai大赛】-管勘智算-V2.0`，含中文全角括号【】；文件资产大量使用中文+全角符号文件名 | Linux UTF-8 文件系统本身支持，但 shell 脚本、打包脚本中未加引号的路径变量会在空格/特殊字符处断裂——`scripts/start-linux.sh` 及后续脚本必须所有路径变量加引号；验证时刻意用含空格目录测试 |
| zip 解压乱码 | 中 | Windows 下打的 zip（GBK 文件名头）在 Linux 解压可能乱码 | 交付统信的包用 `tar.gz` 或确认 zip 以 UTF-8 文件名打包 |
| 归档脚本硬编码 | 中 | `tools/archive_code.py` 含硬编码 Windows 工作流副本路径 `D:\Desktop\...` | 该脚本属 Windows 主线工具，统信侧不运行即可；在兼容检查中标记为已知 WARN，不需要改 |

### 2.7 Windows bat / ps1 脚本迁移风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 启动器不可用 | 确定 | `启动管勘智算-【codex】.bat`、`管勘智算启动器-【codex】.ps1` 等在统信全部不可用 | 已有 `scripts/start-linux.sh` 草案，作为唯一 Linux 启动入口逐步完善：venv 检查、依赖检查、端口占用提示、错误停留（对齐"启动器错误停留"的既有验收口径） |
| 双击习惯差异 | 中 | 统信桌面双击 .sh 默认可能不执行 | 提供 `.desktop` 启动文件或文档说明"终端执行 bash scripts/start-linux.sh"（.desktop 属后续阶段） |
| 快捷方式 .lnk | 低 | 根目录 .lnk 文件在 Linux 无效 | 无需处理，Windows 专用 |

### 2.8 文件权限、运行目录、输出目录风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| 执行权限 | 中 | .sh 需要 `chmod +x`；从 Windows 拷贝过去的文件权限位不保留 | 启动文档写明 `bash scripts/start-linux.sh`（不依赖执行位） |
| 运行输出目录 | 中 | 输出写入 `Codex-Temp/runtime/`，项目放在只读位置（如 /opt）时会失败 | 建议统信部署约定"项目放用户主目录下"；或后续支持环境变量指定运行输出目录（`GUANKAN_RUNTIME_DIR` 之类，作为增强项） |
| 端口权限 | 低 | 8000 非特权端口，无风险 | — |
| 防火墙/安全中心 | 中 | 统信安全中心可能拦截未签名程序联网/监听 | 验证清单包含"首次启动放行提示"记录 |

### 2.9 API Key、离线环境、内网环境风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| `.env.local` 读取 | 低 | 机制本身跨平台 | 验证一次即可 |
| DeepSeek 不可达 | 中 | 统信目标环境很可能是内网/离线，官方 DeepSeek API 不可达 | 项目已有正确口径"大模型不可用时主流程不受影响"——统信验收必须包含**无 Key / 断网场景全流程测试**；远期如需内网大模型，`llm.py` 的 base_url 可配置化是唯一改动点（已支持自定义则记录验证结果） |
| 证书链 | 低-中 | 内网代理/自签证书环境 HTTPS 调用可能失败 | 记录为已知限制，不在近期处理 |

### 2.10 字体、缩放、WebView、系统依赖风险

| 风险项 | 等级 | 说明 | 对策 |
|---|---|---|---|
| Inter 字体缺失 | 中 | 统信默认无 Inter；回退到文泉驿/思源黑体后，字宽、行高、数字对齐观感会变化 | 字体栈确认包含 `system-ui` 和常见中文字体回退；如观感差异大，考虑前端本地内嵌 Inter 字体文件（注意授权，Inter 为 OFL 可内嵌） |
| tabular-nums 支持 | 低-中 | 回退字体可能不支持等宽数字特性，表格数字列对齐效果下降 | 验证记录；必要时数字列用 `font-variant-numeric` + 回退等宽字体 |
| HiDPI 缩放 | 中 | 统信缩放设置下浏览器渲染表格密度变化，普通笔记本 1366×768 + 125% 缩放是最挤场景 | UI 验收沿用现有"普通笔记本不拥挤"口径，在统信分辨率下抽查 |
| 系统依赖 | 低（网页版） | 本地网页版仅依赖 Python + 浏览器，系统库依赖极小 | 这是优先网页版路线的核心优势 |

## 三、不建议现在做的事

1. ❌ 立即全面迁移主线到统信。
2. ❌ Tauri Linux / deb / AppImage 打包（与现有 PRD"暂缓"口径一致）。
3. ❌ 为跨平台重构核心匹配引擎或改动三数字规则（跨平台工作只碰运行环境层）。
4. ❌ 批量重命名中文资产文件为英文（破坏大量真实路径引用，收益极小；UTF-8 下中文路径本身可用）。
5. ❌ 把 `check_platform_compat.py` 挂入日常构建/测试流程（维持现有"手动专项"定位，避免拖慢 Windows 开发）。
6. ❌ 现在就做内网大模型适配（先验证"无大模型时主流程完整可用"即可）。

## 四、阶段性路线

### 阶段 0：跨平台纪律（现在起持续，零额外排期）

- 新增后端代码：路径一律 `pathlib` + 相对项目根推导；`open()` 显式 `encoding="utf-8"`；不引入 Windows 专属调用。
- 新增脚本：路径变量加引号。
- 涉及路径/启动/打包改动时手动跑一次 `python tools/check_platform_compat.py`（现有口径不变）。

### 阶段 1：统信实机最小验证（建议 1-2 天专项，需统信实机或虚拟机）

- 按 `docs/统信UOS兼容性开发说明.md` 的命令跑通本地网页版。
- 执行第六节"最小验证方案"清单。
- 产出《统信验证记录》（记录系统版本、Python/Node 版本、浏览器版本、问题清单）——建议存放到 `docs/`。

### 阶段 2：硬伤修复与离线包（验证暴露问题后排期）

- 修复阶段 1 发现的 FAIL 项。
- 制作离线依赖包（pip wheels + 前端 dist 产物 + 启动脚本 + 部署说明）。
- `start-linux.sh` 从"草案"升级为"可交付"：依赖自检、端口检查、错误停留、日志落盘。

### 阶段 3：统信交付版（有明确部署需求时再启动）

- 类比"评委绿色版"做"统信绿色版"目录（便携 venv 或系统 venv + dist + 样例 + 文档）。
- 视需求决定是否启动 Tauri Linux（默认仍暂缓）。

## 五、给 Codex 的跨平台检查清单

> 建议逐步吸收进 `tools/check_platform_compat.py`（其中部分可能已覆盖，Codex 应先读该脚本现状再增补，避免重复实现）。

**代码层**：
- [ ] 后端无新增 `C:\`/`D:\` 绝对路径字面量
- [ ] 后端无 `os.startfile` / `win32*` / `powershell` / `cmd.exe` 新增调用
- [ ] `open()` / `Path.read_text()` / `Path.write_text()` 显式 UTF-8
- [ ] 无 `os.path.join` 与手写 `\\` 混用（统一 pathlib）
- [ ] 子进程调用不依赖 shell=True + Windows 内建命令
- [ ] 前端相对 import 大小写与真实文件一致（已有）
- [ ] 前端无硬编码 `http://localhost` 差异问题（同源托管口径下确认 API base 路径相对化）

**资源层**：
- [ ] 关键资产路径均可从项目根相对推导（已有 paths.py，新增资产走同一通道）
- [ ] 运行输出仅写 `Codex-Temp/runtime/` 与任务目录
- [ ] `.env.local` 被 gitignore 和存档排除（已有）

**脚本层**：
- [ ] `scripts/*.sh` 路径变量全部加引号、以 `set -euo pipefail` 开头
- [ ] .sh 文件为 LF 行尾（CRLF 会导致 bash 报错——**这是 Windows 上写 .sh 最常见的坑**，建议加入自动检查）

**验证层**：
- [ ] 无 API Key / 断网时主流程可用
- [ ] WPS 打开输出 Excel/Word 正常（颜色、批注、公式值、模板格式）

## 七、验收标准

**阶段 1（最小验证）验收**：
1. 上表 15 项中，1-9、12、15 全部通过（10、11、13、14 允许记录问题延后）。

**阶段 2（硬伤修复）验收**：
1. 阶段 1 的 FAIL 清零，15 项全过。

3. `start-linux.sh` 具备依赖自检与错误停留。

## 八、假设与待确认项

| 假设 | 待确认 |
|---|---|
| 假设目标统信为桌面版 x86_64 | 需确认实际目标机型/架构（若 ARM/LoongArch，依赖策略需重估） |
| 假设目标环境可临时联网装依赖或接受离线包 | 需确认内网管控级别 |
