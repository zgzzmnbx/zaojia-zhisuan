## 四、P0 必须完成的产品范围

### 1. 填价工作台双视图

- 左侧一级菜单继续叫“填价工作台”，不得新增“Dashboard”一级入口。
- 在中间工作区顶部增加“项目看板｜新建填价”页签。
- 项目看板是跨项目首页；新建填价复用现有上传、列映射和开始转换页面。
- 当前任务正在处理或已有结果时，切换页签不得卸载现有状态、重复请求或重新创建任务。
- 点击“新建填价”主按钮直接切换到现有上传视图，不复制上传组件和业务状态。
- 左侧底部现有环形图继续表示当前任务，不得被全项目 Dashboard 数据替换。

### 2. 统一项目台账

建立清晰的数据层：

```text
ProjectRecord
├─ ProjectRun / Job
└─ ProjectArtifact
```

建议新增：

```text
backend/app/project_ledger.py
backend/app/project_dashboard.py
Codex-Temp/runtime/project-ledger.sqlite3
```

具体文件可根据现有代码结构合理调整，但必须满足：

- 项目数按稳定 `project_id` 去重。
- 一次重试、重算、重新生成报告或预警不得自动增加项目数。
- 一次处理任务有独立 `run_id / job_id`。
- 项目、处理任务、文件版本和复核轮次概念分开。
- 台账数据库是 Git 忽略的运行数据，不进入正式资产目录。
- 台账只保存元数据、统计快照和受控成果引用，不复制知识库、规则、经验池和报告模板。

### 3. 新建项目与现有 `/api/process` 兼容

- 用户选择 Excel 后，项目名称可从文件名自动生成，但必须允许用户修改。
- 旧客户端不传项目信息时，现有 `/api/process` 仍可运行。
- 新前端创建任务时保存真实项目关系和 Skill 快照。
- 当前勘察测量流程继续调用既有 `FillEngine`，不得复制或替换匹配引擎。
- 看板写入失败不能损坏已生成成果；错误边界要清楚，不能吞掉台账失败后伪装统计成功。

### 4. 历史数据回填

受控读取：

```text

智能协同现有 SQLite 任务
```

要求：

- 回填幂等，重复执行不重复创建记录。
- 有显式项目 ID、外部业务键或已保存关系时才绑定既有项目。
- 仅文件名相同、项目名相似或输出路径相近时，不得猜测合并。
- 没有可靠项目 ID 的记录显示为“待归类历史任务”，不混入累计项目数。
- 缺少预警结果时显示“未运行”，不能统计成零风险。
- 缺少时间或项目名时明确标记推测 / 未知来源。

### 5. Dashboard 后端 API

至少提供等价于以下能力的接口：


- 支持日期、Skill、状态、来源和关键词筛选。
- 项目列表采用后端分页。
- 前端不扫描本机目录。
- 前端不能传入任意绝对路径换取文件。
- API 不返回本机绝对路径、平台用户 ID、群 ID、文件 Key、密钥、访问令牌或连接票据。
- 项目详情和成果下载复用受控项目 / 任务 ID。

## 五、视觉质量是硬验收，不是可选美化

用户明确要求：必须做得好看、现代，达到生产级 SaaS Dashboard 的完成度。

### 1. 参考图必须真正吸收

指定参考图中重点学习：

- 标题下方紧凑的时间范围、周期对比和控制工具条。
- 三列分析模块形成的稳定网格和一致基线。
- 每个分析模块中的“小标题＋强数字＋变化提示＋趋势图”。
- 图表留白、坐标轴、图例、标签和数据周期的清晰组织。
- 模块尺寸、内边距、圆角和边框高度一致。

不得照搬：

- Shopify 品牌、图标、电商文案和业务指标。
- 黑色顶栏、绿色品牌色和大面积深色导航。
- 与造价智算规范冲突的灰底卡片海、厚阴影或固定三列像素布局。

### 2. 造价智算 UI 规范优先

最终必须使用项目已有设计语言：

- `#FFFFFF` 主工作区、`#F8FAFC` 极浅工具条 / 表头。
- `#0F172A` 强文本、`#334155` 正文、`#64748B` 辅助文字。
- `#2563EB` 主操作。
- 标准命中绿、经验提示琥珀、待复核 / 高风险红、信息 / 处理中蓝。
- `1px` 轻边框、`6px-8px` 圆角、常驻区域无阴影。
- 全局字体栈和等宽数字严格复用 UI 规范。
- 继续使用现有 `lucide-react` 图标，禁止 Emoji 充当功能图标。

### 3. Rhea preset 必须转译，禁止全量套入

视觉目标是“Rhea 风格的造价智算 Dashboard”，不是把项目改造成新的 shadcn 示例工程：

| Rhea preset 项 | 本项目实现要求 |
| --- | --- |
| `rhea` | 吸收紧凑、精确、现代的数据工作台组件比例和层级 |
| `neutral` 基底 | 映射到现有白底、极浅灰表头、细边框和低对比坐标轴 |
| `blue` 主题 | 使用现有 `--db-primary` 及 hover / focus token |
| `neutral` 图表色 | 只用于网格线、坐标轴和非业务辅助系列；业务系列继续用固定状态色 |
| `Inter` | 复用现有字体栈，不下载在线字体 |
| `Lucide` | 复用现有 `lucide-react` |
| 默认圆角 | 落在现有 `6px-8px`，不得大圆角或过度胶囊化 |
| 默认菜单＋subtle accent | 白底 / 极浅灰底、文字加深、细线或浅底的克制选中态 |

当前 `frontend/package.json` 没有 Tailwind / shadcn，项目也没有 `frontend/components.json`。因此本任务：

- 禁止直接运行 `npx shadcn@latest init`。
- 禁止直接运行 `npx shadcn@latest create`。
- 禁止运行全量 `npx shadcn@latest apply b27I2r68`。
- 禁止为了单个 Dashboard 重写全局 CSS 工具链、批量覆盖按钮 / 表格 / 导航 / 弹窗或引入第二套主题。
- 默认以 `.project-dashboard` 为根作用域，将 Rhea 的排版、尺寸、边框、控件和交互质感转译为现有 React + CSS。
- 可以借鉴 shadcn/ui 的可访问组件结构，但最终实现必须离线可运行，不依赖 preset URL。

如果确有充分技术理由必须引入 Tailwind / shadcn，先停止实施，在交付中给出影响文件、依赖体积、绿色版变化、现有样式冲突和回滚方案，等待用户明确确认；不得自行扩大技术栈。

### 4. 不允许“卡片套娃”

- Dashboard 模块可以作为主网格中的直接分析单元，但不能在灰色大卡片里再套多层小卡。
- 分析单元直接落在统一工作区，使用白底、细边框、统一圆角，无常驻阴影。
- KPI 可以做成一排有分割线的指标区，或直接主网格单元；不要每个数字套两层容器。
- 层级依靠字号、留白、边框和网格，不靠阴影、渐变、玻璃拟态和厚背景。

### 5. 响应式必须看容器宽度

- 不按 1366、1433、1440 或 1920 任一分辨率写死。
- 看板应根据中间工作区容器宽度决定 3 列、2 列或 1 列。
- 推荐使用 CSS Grid、`minmax()`、`clamp()`、`auto-fit` 或容器查询。
- 右侧问问智算展开时，中间图表可自然收敛；不得压扁文字、截断坐标或造成页面横向滚动。
- 页面主壳不横向滚动；历史表格自己的承载区允许受控横向滚动。

### 6. 动效与交互

- hover、筛选和图表强调使用 150ms-250ms 克制过渡。
- `prefers-reduced-motion` 下关闭非必要动画。
- 加载超过 300ms 显示稳定尺寸骨架。
- 图表、KPI、筛选 chip、项目行和抽屉操作都有明显 hover、focus 和 pressed 状态。
- 交互不能通过缩放卡片造成布局抖动。

### 7. 图表实现

- 当前项目没有现成图表依赖；可优先评估 Recharts，并在本地依赖中固定、进入绿色版和离线包。
- 图表模块应按页面懒加载，避免拖慢现有填价工作台首屏。
- 不允许 CDN、在线字体和在线图表脚本。
- 如果不用 Recharts，必须提供同等级的响应式、交互、无障碍和离线能力。
- 不能为了图表库大幅重构前端工具链或引入不必要的大型状态管理库。

- 不生成或覆盖 `frontend/components.json`，不把 Tailwind / shadcn 初始化作为 P0 的隐式前置条件。
- Dashboard 组件通过明确 props 和 API 获取数据，不复制现有填价状态。
- “新建填价”继续使用现有上传与列映射组件。
- 不复制 `FillEngine`、预警算法、Word 报告、知识问答或智能协同状态机。
- 后端项目聚合放入独立模块，不继续把大量逻辑堆入 `main.py`。
- 数据库访问、统计和回填使用明确事务与幂等约束。

## 八、自动化测试

### 后端专项
