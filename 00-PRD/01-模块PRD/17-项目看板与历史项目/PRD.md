# 项目看板与历史项目 PRD

## 模块目标

把现有“填价工作台”从只处理当前一张 Excel 的操作页，升级为能够看项目、开任务、查历史、看趋势、追成果的项目工作台。

用户进入“填价工作台”后，通过顶部二级页签在以下两个视图间切换：

```text
项目看板｜新建填价
```

- `项目看板`：展示累计处理项目、当前项目状态、时间趋势、风险分布、匹配质量和历史项目。
- `新建填价`：完整保留现有“上传标准 Excel—列映射—开始转换—查看工作状态”链路。

本模块的核心不是给上传页装饰几张图，而是建立真实、可追溯的“项目—任务—版本—成果”数据关系，使网页填价、对话式任务和智能协同任务可以进入同一个项目视图。

## 定位与优先级

- 模块优先级：`P0`。
- 用户入口：沿用左侧“填价工作台”，不新增左侧一级菜单。
- 默认视图：无正在处理的当前任务时默认进入“项目看板”；用户切换视图后，本次页面会话内保持最后选择。
- 当前任务保护：切换到项目看板不得卸载当前上传文件、列映射、任务结果和下载状态。
- 数据范围：P0 默认展示当前本机实例的真实项目数据；组织级、跨人员和跨实例统计属于后续生产化范围。
- 业务边界：不修改三个数字匹配、经验池预警、报告生成、人工复核和智能协同业务结论。

## 关键统计口径

### 1. 项目、任务与版本

```text
项目 Project
├─ 处理任务 ProjectRun / Job
│  ├─ 输入文件
│  ├─ 处理摘要
│  ├─ 风险结果
│  └─ 输出成果
└─ 文件版本 / 复核轮次
```

| 对象 | 定义 | 计数规则 |
| --- | --- | --- |
| 项目 | 具有唯一 `project_id` 的业务对象 | 累计项目按 `project_id` 去重 |
| 处理任务 | 一次上传、外部派发或确定性处理运行 | 每次真实运行计一次任务，但不等于新增项目 |
| 版本 | 同一项目下重新编制、退回修改或成果更新形成的业务版本 | 版本号递增，保留前版引用和生成时间 |
| 复核轮次 | 智能协同中的一次并行多人复核周期 | 不等同文件版本；退回后形成新轮次并按业务决定是否形成新版本 |

### 2. KPI 口径

| KPI | 计算方式 | 特殊说明 |
| --- | --- | --- |
| 累计项目 | 当前筛选范围内唯一 `project_id` 数 | 旧任务没有可靠项目 ID 时单独显示“待归类历史任务”，不混入项目数 |
| 本月新增 | `created_at` 位于本月的唯一项目数 | 不按任务重跑时间计算 |
| 已完成 | 项目最新有效状态为 `completed` | 已完成后生成新版本时按最新版本状态重新判断 |
| 待复核 | 最新任务存在待复核行，或协同状态处于待复核 / 已退回 | 同一项目只计一次 |
| 存在高风险 | 预警已运行且 `risk_high > 0` 的项目数 | 预警未运行不是零风险，必须单列“未运行” |
| 处理任务数 | 当前筛选范围内真实任务运行次数 | 只作为辅助指标，不替代项目数 |

## 页面入口与交互

### 1. 左侧导航

- 左侧一级菜单名称继续使用“填价工作台”。
- 副说明可由“上传与列映射”调整为“项目看板与新建填价”。
- 不新增“Dashboard”“数据驾驶舱”等新的一级菜单，避免模块列表继续膨胀。
- 左侧底部现有环形图继续表达“当前任务状态”，不得改成全项目统计。

### 2. 中间工作区顶部

```text
项目工作台
项目看板｜新建填价
```

- `项目看板` 是跨项目视图。
- `新建填价` 是当前任务操作视图，复用现有 JSX、状态和处理回调。
- 有正在处理的任务时，项目看板顶部显示轻量“当前任务”状态条，可返回新建填价或结果预览。
- 页签切换只改变呈现，不重新请求、重新上传或重新创建任务。

### 3. 看板工具条

工具条从左至右建议为：

```text
时间范围｜对比周期｜专业能力｜项目状态｜任务来源｜搜索项目｜重置筛选｜新建填价
```

- 默认时间范围为最近 30 天；用户可切换最近 7 天、本月、近 90 天、本年和自定义。
- 对比周期默认关闭；开启后可与上一周期比较新增项目和完成项目。
- “新建填价”为唯一高权重深蓝主按钮。
- P0 不实现用户自定义拖拽看板；“自定义看板”属于后续增强。

## 图表设计

| 图表 | 默认维度 | 图形 | 关键规则 |
| --- | --- | --- | --- |
| 项目处理趋势 | 最近 12 个月或当前时间范围 | 分组柱状图；数据点足够时可叠加完成率细线 | 柱状显示新增 / 完成；少于 2 个有效周期时降级为数字摘要，不画误导趋势 |
| 项目状态分布 | 最新项目状态 | Dashboard donut | 最多 5 类：处理中、待复核、已完成、已退回、失败；中心显示总项目数 |
| 风险项目排行 | 高风险数降序的前 8 个项目 | 横向堆叠柱状图 | 分段显示高风险 / 低风险；预警未运行使用灰色“未运行”标签，不画成 0 |
| 整体匹配质量 | 当前筛选范围的已处理数据行 | 100% 堆叠条形图 | 标准命中、经验提示、待复核三段；同时显示原始数量和比例 |
| 大模型请求次数 | 当前时间范围 | 浅蓝平滑面积波形图 | 只统计本机真实调用审计和可回填的历史提示词日志；明确测试模型排除；少于 2 个有效周期时显示真实空状态 |
| 请求模型种类 | 当前时间范围 | 多蓝色环形图 | 按模型标识聚合真实请求次数；中心显示模型种类数，图例常显各模型次数；不让大模型反向生成统计 |
| 项目来源分布 | 网页、对话式工作台、WeAct / 飞书、外部系统 | 小型环形图，数据成熟后显示 | 只有一个来源或样本过少时不强行显示；可作为 P0 后半阶段增强 |

图表通用要求：

- 图表点击后将筛选条件写入历史项目列表，并显示可关闭的筛选 chip。
- 所有关键数字始终可见，不把数据只藏在 hover 中。
- 线条、柱体、圆环使用项目语义色；同一状态在图表、标签和表格中颜色一致。
- 趋势对比不能只靠两种相近颜色，应同时使用实线 / 虚线或明确图例。
- 支持键盘聚焦和屏幕阅读文本；至少提供与图表同口径的文本摘要。
- 图表加载超过 300ms 时显示骨架屏；接口失败时单个图表独立失败，不让整个看板白屏。

## 历史项目列表

### 默认列

| 列 | 内容 |
| --- | --- |
| 项目名称 | 项目名称、项目编号或“待归类历史任务” |
| 来源 | 网页上传、智算助手、WeAct / 飞书、外部系统 |
| 专业能力 | Skill 中文名、版本 |
| 当前状态 | 处理中、待匹配、待复核、已退回、已完成、失败 |
| 数据规模 | 输入行数、已匹配行数 |
| 匹配质量 | 匹配率、经验提示数、待复核数 |
| 风险 | 未运行、无风险、低风险、高风险数量 |
| 版本 | 最新业务版本和复核轮次 |
| 更新时间 | 最近真实状态更新时间 |
| 成果 | Excel、Word、风险清单是否可用 |
| 操作 | 查看详情、打开预览 / 报告、下载成果 |

### 列表交互

- 默认按 `updated_at` 倒序。
- 支持项目名 / 项目编号搜索、状态 / 来源 / Skill / 时间筛选。
- 点击图表、KPI 或状态标签后，列表使用同一筛选上下文。
- 项目行点击打开详情抽屉；不强制跳离项目看板。
- 成果已被清理或文件不存在时显示“文件已失效”，不得保留可点击假按钮。
- P0 的“继续处理”只在现有状态能够安全恢复时显示；完整恢复、关联既有项目和版本管理属于 P1。

## 项目详情

P0 详情至少展示：

```text
项目基本信息
当前状态与专业 Skill 快照
处理任务时间线
输入 / 匹配 / 待复核 / 风险摘要
现有 Excel / Word / 风险清单成果
智能协同来源与当前复核状态（若存在）
```

详情只展示业务需要的数据，不回显本机绝对路径、平台用户 ID、群 ID、文件 Key、密钥、访问令牌或连接票据。

## 后端接口建议

```text
GET  /api/projects/dashboard
GET  /api/projects
GET  /api/projects/{project_id}
GET  /api/projects/{project_id}/runs
GET  /api/projects/{project_id}/artifacts
POST /api/projects/backfill
```

接口要求：

- `dashboard` 和 `projects` 接受同一套日期、Skill、状态、来源和关键词筛选。
- Dashboard 返回 KPI、趋势、状态、风险排行、匹配质量和筛选元数据。
- 列表接口使用后端分页，不把全部历史项目一次性发到前端。
- 详情与成果接口使用项目 ID 和运行 ID，不接受前端传入任意本机路径。
- 旧客户端调用 `/api/process` 时保持兼容；新增项目字段使用可选参数或先创建项目后复用既有处理接口。

## 历史数据回填

P0 回填来源：

1. `Codex-Temp/runtime/<job_id>/process-state.json`。
2. 智能协同现有 SQLite 任务记录。
3. 后续对话式智能体持久化任务记录。

回填规则：

- 能通过显式 `project_id`、外部业务键或已保存项目关系确定归属时，绑定既有项目。
- 只有文件名相同、项目名相似或输出路径接近时，不自动合并。
- 无创建时间时可使用受控文件时间作为“推测时间”，但必须标记来源，不冒充业务创建时间。
- 旧任务缺少项目名时使用输入文件名生成展示名，并标记“待归类历史任务”。
- 回填操作必须幂等；重复执行不新增重复项目、任务或成果。

- 主操作 `#2563EB`；标准命中绿色、经验提示琥珀、待复核 / 高风险红色、信息 / 处理中蓝色。
- 常驻面板无阴影，圆角以 `6px-8px` 为主。
- 图表数字使用等宽数字特性。

### shadcn/ui Rhea 视觉基准

Dashboard 的第二视觉基准为：

```text
https://ui.shadcn.com/create?preset=b27I2r8C
```

该 preset 已通过官方 CLI 解码，固定配置如下：

| 配置 | 值 | 在造价智算中的采用方式 |
| --- | --- | --- |
| `style` | `rhea` | 吸收清晰层级、紧凑控件、细边框、现代数据工作台比例 |
| `baseColor` | `neutral` | 用于工作区、分割线、表头、次级文字和图表坐标轴，不覆盖业务状态色 |
| `theme` | `blue` | 映射为现有 `--db-primary`、hover 和 focus 体系 |
| `chartColor` | `blue`（视觉复核覆盖） | 依用户浏览器批注采用默认 UI 浅蓝同色系；数值、图例和文字摘要始终常显，不仅依赖颜色表达 |
| `font` | `inter` | 复用项目既有字体栈，不新增在线字体或字体文件依赖 |
| `fontHeading` | `inherit` | 标题继承全局字体，以字号、字重和留白形成层级 |
| `iconLibrary` | `lucide` | 复用现有 `lucide-react`，不安装第二套图标库 |
| `radius` | `default` | 落到项目 `6px-8px` 圆角范围，不胶囊化、不做大圆角卡片 |
| `menuColor` | `default` | 保持白底 / 极浅灰底菜单 |
| `menuAccent` | `subtle` | 保持克制选中态，不使用高饱和大色块 |

采用原则：

- 该链接是设计 preset，不是运行时在线样式表；页面不得依赖访问该网址才能正常显示。
- 当前前端没有 Tailwind 和 `components.json`，P0 默认不执行 `npx shadcn init`、`npx shadcn create` 或全量 `npx shadcn apply`。
- 不因 Dashboard 单页引入第二套全局组件和主题系统；以 `.project-dashboard` 为根作用域，把 Rhea 的比例、控件质感和排版语言转译到现有 React + CSS。
- 可以借鉴 shadcn/ui 的组件结构和无障碍交互，但最终 CSS token、状态语义和响应式行为以项目规范为唯一裁决。
- 如果开发中认为必须引入 Tailwind / shadcn，应先形成影响清单、文件 diff、绿色版依赖变化和回滚方案，未经用户确认不得实施。

## 响应式与布局

- 看板主区域使用容器宽度而不是浏览器视口宽度决定图表列数。
- 宽容器可使用 12 列网格：趋势图占 8 列，状态图占 4 列；第二行两个图各占 6 列。
- 中等容器自动收敛为两列；右侧智算 Dock 展开导致主区变窄时，必要时改为单列。
- KPI 使用 `repeat(auto-fit, minmax(...))` 或等价网格，不能溢出或形成不均匀孤行。
- 历史项目表为主数据区，可在容器内横向滚动；关键的项目名、状态和操作列优先可见。
- 页面主壳不产生横向滚动；只有历史表格承载区允许受控横向滚动。

## 技术实现边界

- 前端建议建立独立目录，避免继续扩大 `App.tsx`：

```text
frontend/src/components/project-dashboard/
├─ ProjectDashboard.tsx
├─ ProjectDashboardToolbar.tsx
├─ ProjectMetricGrid.tsx
├─ ProjectCharts.tsx
├─ ProjectHistoryTable.tsx
├─ ProjectDetailDrawer.tsx
├─ projectDashboardUtils.ts
└─ projectDashboard.css
```

- `App.tsx` 只负责“项目看板｜新建填价”视图切换、现有任务状态桥接和模块入口。
- Dashboard 根节点使用稳定的 `.project-dashboard` 作用域；新增视觉变量优先映射现有 `--db-*` token，不污染其他业务模块。
- 当前 P0 不初始化 Tailwind / shadcn，不生成或覆盖 `components.json`，不运行全量 preset apply，不批量重写现有按钮、表格、导航和弹窗。
- 图表库如需新增，必须本地安装、进入绿色版依赖并按项目页面懒加载；不得使用 CDN。
- 可优先评估 React 生态的 Recharts；若最终不用，也必须达到同等响应式、交互、无障碍和离线能力。
- 后端建议新增独立 `project_ledger.py` / `project_dashboard.py`，不得把项目统计继续堆入 `main.py`。
- 看板只消费业务结果和统计快照，不复制或重写 `FillEngine`、预警、报告和智能协同状态机。

## 空状态、加载与失败

| 状态 | 要求 |
| --- | --- |
| 无历史项目 | 保留页面结构，显示“还没有项目”和“新建填价”主动作；不画零数据假图 |
| 只有一个项目 | 展示真实 KPI 和历史表；不强行绘制缺少时间维度的趋势 |
| 预警未运行 | 显示“未运行”，不归入无风险 |
| 正在加载 | KPI 和图表使用稳定尺寸骨架，页面不跳动 |
| 单图失败 | 当前图表显示重试和错误摘要，其他图表及历史表继续可用 |
| 台账不可用 | 新建填价主流程保持可进入，并明确说明项目历史暂不可用 |
| 成果已失效 | 显示失效状态和重新处理建议，不提供无效下载按钮 |

## 验收口径

### 功能

- 左侧仍只有一个“填价工作台”入口，内部可切换“项目看板｜新建填价”。
- 现有上传、列映射、转换、预览、预警、报告和下载链路行为不变。
- 项目数、任务数和版本数能够被明确区分；重复运行不会虚增项目。
- 网页新建的项目和智能协同任务可在同一历史列表中按来源区分。
- KPI、图表和历史列表使用同一筛选条件与同一后端口径。
- 图表点击可以筛选历史列表，清除筛选后恢复。
- 没有历史数据、预警未运行和成果失效均有真实、稳定的状态。

### 视觉

- 首屏能一眼看出这是“项目分析与历史工作台”，而不是上传页下方临时塞入图表。
- 参考图的分析网格、强数字、趋势和周期对比节奏得到体现。
- Rhea 的 Neutral 基底、Blue 主操作、Inter 排版、Lucide 图标、默认圆角和 subtle 菜单选中态得到体现，但不出现机械照搬或第二套全局主题。
- 最终色彩、字体、圆角、边框、按钮和状态语义与项目 UI 规范一致。
- 不出现灰底白卡套娃、厚阴影、玻璃拟态、大面积渐变、黑色顶栏或 Shopify 式品牌色。
- KPI、图表、历史表格和详情抽屉对齐整齐，数字、小数点、坐标轴和图例清晰。
- 在普通笔记本、宽屏、左右栏折叠 / 展开和浏览器缩放下无大片空白、主区压扁或页面横向溢出。

### 数据与安全

- 前端不扫描本机目录，不接收或回显本机绝对路径、平台用户 ID、群 ID、文件 Key 和密钥。
- Dashboard 只展示当前实例授权范围内的数据。
- P0 的统计异常不改变专业计算结果，不破坏已有输出文件。
- 旧任务回填可重复执行且不产生重复记录。

### 验证

- 项目台账、去重、筛选、聚合、回填和文件失效状态有后端专项测试。
- 图表数据转换、筛选联动、空状态和响应式工具函数有前端专项测试。
- `python -m pytest backend/tests -v` 通过。
- `npm run frontend:build` 通过。
- 内置浏览器完成真实运行态验收，并保存普通笔记本、参考图相近宽度、宽屏和右侧 Dock 展开态截图。
- `python tools/check_prd_consistency.py --strict` 与 `git diff --check` 通过，或只保留已确认的历史巡检问题。

## 功能边界

- 不做组织级跨人员排行榜和绩效评价。
- 不做在线数据仓库或远程 BI 平台。
- 不根据文件名相似度自动合并历史项目。
- 不自动删除、移动或重命名历史运行目录和成果文件。
- 不用大模型计算 KPI、统计项目数或判断项目状态。
- 不为了看板重构三个数字匹配、预警算法、报告模板和智能协同状态机。
- 不把规划中的第二专业 Skill 伪装成已有项目分布。
- 不引入在线字体、在线图标、在线图表脚本或 CDN。
- 不把 shadcn preset URL 当作在线 CSS 依赖，不在 P0 全量引入 Tailwind / shadcn 或覆盖现有全局样式。
