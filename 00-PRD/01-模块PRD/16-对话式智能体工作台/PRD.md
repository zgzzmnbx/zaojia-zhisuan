# 对话式智能体工作台 PRD

## 模块目标

在左侧功能菜单设置独立一级入口“智算助手”，作为对话式智能体工作台的正式主入口；现有右侧“问问智算”继续作为随行 Dock，并提供“进入智算助手”快捷入口。用户从任一入口进入后，只保留左侧功能导航，将其余区域切换为宽屏会话区；用户可以在底部选择专业能力包、上传文件、下达任务、查看执行进度、确认关键动作并接收 Excel、Word、风险清单等成果。

本模块追求“对话优先”，但不删除现有专业页面。主流任务应能只通过聊天完成；复杂字段映射、宽表逐行复核、规则详情和报告精查仍可从会话中的结构化卡片进入原专业工作台，避免为了追求聊天感而牺牲确定性、审计和表格效率。

对外表达：

> 造价智算既可以作为专业工作台使用，也可以作为对话式造价智能体使用。用户选择专业能力包、上传资料并说明目标，智能体负责调用确定性专业流程、反馈进度、组织复核并交付成果。

## 当前完成度（2026-07-21）

阶段结论：**模块 P0 已完成，已经达到“可演示、可真实执行业务、可交付成果”的程度；模块 P1 / P2 尚未完成，当前不能表述为可恢复的多 Skill 智能体平台。**

- 已完成真实闭环：选择当前已上线 Skill、点击或拖拽上传 Excel、开始转换、批量匹配、经验池预警、风险清单、风险报告、Excel / Word 成果和专业页面接力均复用现有确定性业务链路。

## 可行性判断

### 结论

技术和产品上均可行，且适合造价智算现有架构，因为当前右侧 Dock 已具备消息流、知识问答、快捷指令、任务跳转和文件成果入口，现有 React 主壳也已经支持模块切换和左右栏折叠。新增左侧入口只负责切换到同一会话工作台状态，不需要新建第二套聊天系统。

真正的难点不是“全屏 CSS”，而是让聊天中的每个动作都对应一个确定性的专业 API、任务状态和 Skill 运行时上下文。如果只做视觉放大，用户仍然只能聊天，不能真正通过聊天完成工作。

### 为什么值得做

- 降低入口复杂度：用户无需先理解所有左侧模块，可以直接上传文件并说目标。
- 强化智能体感知：系统主动组织步骤、反馈进度、请求确认和交付成果，而不是只回答问题。
- 适合演示：能直观看到“选专业能力—交任务—执行—复核—拿成果”的完整链路。
- 保留专业可信度：计算继续由结构化规则和现有业务引擎完成，聊天只负责编排和解释。
- 有利于扩展：第二个真实 Skill 上线后，同一会话入口可以切换专业能力而不复制整套界面。

### 不建议做成绝对纯聊天的原因

工程造价存在宽表、合并单元格、字段映射、逐行依据、公式和大量人工复核点。聊天适合发起任务、解释过程、确认关键动作和接收成果，不适合替代所有密集表格核对。因此产品口径应是：

> 主流程可以只靠聊天完成，复杂核对始终可以一键进入专业页面。

## 交互模式

### 当前已落地的会话 UI 基线

- 视觉语言采用白色主背景、黑灰正文、浅蓝辅助强调和细分割线，借鉴主流大模型会话的克制布局，但不复制外部品牌标识。
- 助手消息不套大面积卡片，用户指令右对齐显示浅色气泡；每条用户指令开始一个新会话分段，解决连续助手回复缺少边界的问题。
- 输入区固定在会话底部，文本区激活后不显示额外灰色框线；附件、任务操作和 Skill 包位于同一工具行，任务操作默认折叠。
- Skill 包使用自绘 Popover 向上展开；当前项以浅蓝选中态和勾选标识呈现，未上线项保留真实状态但不可选择，不再调用 Windows 原生 `<select>` 菜单。
- 任务操作和 Skill 菜单只允许同时展开一个；点击外部或按 `Esc` 关闭，避免浮层重叠。

### 模式一：随行 Dock

- 保留当前左侧导航、中间专业工作区和右侧问问智算。
- 用于解释当前行、查依据、触发快捷动作和随行复核。
- 默认宽度继续由 `zhisuanWindow` 配置控制。

### 模式二：智能体专注模式

- 左侧功能菜单设置独立一级入口“智算助手”，作为常驻、可发现的正式入口。
- “智算助手”位于专业 Skill 选择器下方、传统业务模块上方；不使用“全屏化智算助手”作为用户侧名称。
- 用户也可以点击右侧问问智算顶部“进入智算助手”，进入完全相同的工作台状态。
- 左侧功能导航保持不变，原中间工作区暂时隐藏但不卸载、不清空。
- 会话区占满左侧导航之外的全部可用空间。
- 顶部提供当前会话、当前任务、运行状态、退出专注模式和新建会话。
- 中部为宽度受控的消息流；普通文字保持易读行宽，表格、进度和成果卡可以适当放宽。
- 底部为固定输入区，包含附件、专业能力、任务上下文、文本输入、发送 / 停止。

模式切换只改变表现形式，不创建新会话、不重置消息、不重新调用接口。用户点击左侧其他专业模块时退出专注布局并显示对应页面，但会话、任务和输入草稿继续保留；再次点击“智算助手”应无损恢复。

## 底部输入区

### 组成

1. `＋` 附件按钮：上传当前 Skill 支持的 Excel、Word 或其他允许文件。
2. 专业能力选择器：显示当前 Skill、状态、版本和锁定状态。
3. 上下文提示：显示当前项目、任务、Sheet / 行或“新任务”。
4. 多行输入框：支持自然语言任务和追问。
5. 发送 / 停止按钮：运行中允许停止模型生成或取消尚未提交的动作。

### Skill 选择规则

- 数据必须来自后端 Skill Registry，不在前端维护第二份名称和状态。
- `active` 可创建任务；`planned`、`disabled` 只可查看，不能执行。
- 没有任务时可以选择专业能力。
- 文件已上传但任务尚未创建时，切换 Skill 必须重新校验文件。
- 任务创建后以任务 Skill 快照为准，输入区显示锁定图标；要切换必须新建任务。
- 在真实 Skill 运行时路由完成前，选择器只能诚实显示当前勘察测量能力，不能声称已经动态装载不同计价库和规则。

## 会话驱动的工作流程

### 标准流程

```text
新建会话
→ 选择专业能力
→ 上传文件
→ 输入目标
→ 校验输入规范
→ 智能体提出确定性执行计划
→ 用户确认关键选项
→ 调用专业 API 创建任务
→ 展示进度和阶段结果
→ 展示风险与待复核项
→ 用户在会话确认或进入专业页面处理
→ 生成并返回 Excel / Word 成果
→ 留存任务、Skill 和动作审计
```

### P0 必须支持的确定性动作

| 用户表达 | 系统动作 | 反馈形式 |
| --- | --- | --- |
| “帮我编制这个控制价”＋Excel | 检查 Skill 和输入，创建待匹配任务 | 文件卡＋执行计划＋确认按钮 |
| “开始匹配 / 批量匹配” | 调用现有批量匹配接口 | 阶段进度＋匹配摘要 |
| “检查风险 / 运行经验池预警” | 调用现有预警接口 | 风险摘要＋可定位风险卡 |
| “这行为什么待复核” | 携带任务、Sheet、行号查规则和证据 | 解释气泡＋依据卡＋去处理 |
| “生成报告” | 调用现有报告链 | 生成状态＋Word 成果卡 |
| “下载结果” | 返回当前任务已生成成果 | Excel / Word 下载卡 |
| “重新开始” | 新建会话或新任务，保留旧任务只读历史 | 二次确认＋新上下文 |

这些动作必须映射到后端白名单，不让大模型根据文本直接执行任意方法。

## 消息与卡片类型

- 普通消息：解释、追问、提示和总结。
- 知识证据卡：来源、标题、适用范围、正式 / 项目记忆标签。
- 文件卡：文件名、类型、大小、校验状态和移除动作。
- 执行计划卡：准备调用的 Skill、步骤、输入和关键选项。
- 确认卡：开始处理、覆盖当前草稿、撤销知识等需要确认的动作。
- 进度卡：阶段、进度、已完成步骤、当前动作和取消状态。
- 风险卡：等级、Sheet、行号、原因、依据和“去处理”。
- 成果卡：Excel、Word、风险清单和报告预览入口。
- 错误卡：说清失败环节、是否已写入、可重试动作和专业页面兜底。

文字消息采用易读窄行宽；结构化卡片可以使用更宽区域，但仍保持白底、细边框、低圆角和无常驻阴影，不做卡片套娃。

## 智能体执行边界

### 路由原则

```text
确定性快捷指令与状态机
＞ 已确认的专业动作
＞ 知识库证据问答
＞ 普通大模型解释
```

- 前端可以做即时意图提示，后端必须再次校验。
- 大模型可以把自然语言整理为候选意图和参数，但不能直接调用任意代码。
- 所有专业动作通过白名单 Action ID 映射到现有 API。
- 写操作必须校验会话、任务、Skill 快照、任务状态和用户确认。
- 价格、系数、风险等级和最终成果仍由结构化引擎生成。
- 大模型不可用时，已定义的确定性指令、上传、处理、下载和专业页面仍可使用。

## 会话上下文模型

每个智能体会话至少维护：

```text
session_id
skill_snapshot
project_key
job_id
active_module
active_sheet
active_row
attachments
messages
pending_action
action_history
artifact_references
created_at
updated_at
```

边界：

- `skill_snapshot` 和 `job_id` 是业务上下文，不允许只保存在显示文本里。
- 聊天消息、动作审计、项目知识记忆是三类不同数据，不混用同一状态。
- 普通聊天不会自动进入知识记忆；只有用户明确保存并通过既有治理流程后才能沉淀。
- 会话异常不能破坏 Excel、Word、规则、经验池和知识记忆数据库。
- P0 可以只保存当前会话；P1 再提供历史会话、命名、搜索和保留策略。

## 与专业 Skill 的真实依赖

本模块不能建立在“只记录 Skill 标签”的假能力上。聊天驱动专业任务前，模块 15 至少需要补齐 `SkillRuntimeContext`：

```text
SkillRuntimeContext
├─ input_profile
├─ processor_id（代码白名单）
├─ knowledge_base_path
├─ rule_assets
├─ risk_profile
├─ knowledge_sources
├─ report_template_path
└─ validation_profile
```

会话选择 Skill 后，所有专业动作必须使用任务创建时冻结的运行时上下文。当前 `DEFAULT_KB_PATH`、默认规则和默认报告模板仍是全局固定实现，v5.11.0 只对已上线的单一勘察测量 Skill 复用现有真实业务链路；在 P0-B 运行时路由完成前，不得把底部 Skill 选择包装为真实多专业执行。

## 技术实现建议

### 前端

建议增量拆分：

```text
frontend/src/components/agent-workspace/
├─ ConversationalAgentWorkspace.tsx
├─ AgentComposer.tsx
├─ AgentMessageStream.tsx
├─ AgentActionCard.tsx
├─ AgentProgressCard.tsx
├─ AgentArtifactCard.tsx
└─ agentWorkspace.css
```

- `App.tsx` 只保留模式状态和现有任务桥接，不继续堆全部会话组件。
- 复用现有 `ChatMessage`、快捷指令、等待状态、知识问答和 `ProfessionalSkillSelector` 数据源。
- 模式切换使用同一 React 状态树；不要同时挂载两套输入框并双向同步。
- 专注模式使用布局状态类切换，不使用覆盖整个页面的 Modal。

### 后端

建议在真实会话动作增加后拆出：

```text
backend/app/agent_orchestrator.py
```

职责：

- 管理白名单 Action ID 和状态前置条件。
- 把已确认动作调用到现有处理、预警、风险、报告和下载服务。
- 保存会话动作审计和任务引用。
- 不复制 `FillEngine`、报告生成、知识检索和 WeAct 业务逻辑。

接口可以结合现有 API 收敛，建议方向：

```text
POST /api/agent/sessions
GET  /api/agent/sessions/{session_id}
POST /api/agent/sessions/{session_id}/messages
POST /api/agent/sessions/{session_id}/attachments
POST /api/agent/sessions/{session_id}/actions/{action_id}/confirm
POST /api/agent/sessions/{session_id}/actions/{action_id}/cancel
```

P0 若不保存历史会话，可以先复用现有接口和前端状态，但消息 / 动作数据结构必须为后续会话恢复预留稳定 ID，不得从气泡文字反向解析状态。

## UI 规范

- 左侧新增一级入口名称固定为“智算助手”，不使用“全屏化智算助手”“智能体专注模式”等技术形态名称作为菜单文案。
- “智算助手”放在专业 Skill 选择器下方、传统业务模块上方，使用与现有导航一致的图标、字号、间距和克制选中态；它是统一任务入口，不与“问问智算知识库”混同。
- 左侧功能导航在专注模式中继续保留，并保持现有宽度和折叠能力。
- 会话区背景为白色或极浅灰，主消息列建议限制在易读宽度；动作卡和成果卡可更宽。
- 顶部栏高度保持紧凑，显示当前会话、任务、状态和退出动作，不做营销 Hero。
- 输入区固定底部并与消息区用 `1px` 边框分隔；允许轻微浮层感，但不使用厚阴影和大面积渐变。
- 用户消息可以右对齐浅蓝，智能体消息左对齐浅灰 / 白色；专业动作卡不强制套进气泡。
- Skill 选择器使用紧凑按钮或 Chip，显示中文名称，`Skill` 为辅助标签。
- 正在执行时明确显示“正在解析 / 校验 / 匹配 / 预警 / 生成报告”，不能只显示旋转图标。
- 普通笔记本、宽屏浏览器和 Tauri 窗口缩放下，左侧不被挤没、输入区不溢出、成果卡不横向撑破。
- 支持键盘焦点、`Enter` 发送、`Shift+Enter` 换行、`Esc` 退出专注模式、附件和 Skill 选择的可访问名称。

## 非功能要求

- 模式切换应为纯前端轻量操作，不重新请求大文件或重建 Word 预览。
- 长任务刷新频率受控，不因聊天界面持续轮询拖慢表格处理。
- 附件大小、类型和数量由当前 Skill 输入规范及后端共同校验。
- 会话和动作日志不得记录 API Key、App Secret、访问令牌、临时下载票据或完整异常堆栈。
- 关闭 / 刷新前存在未提交附件、输入草稿或待确认动作时提供合理提示。
- 绿色版、浏览器和 Tauri 使用同一前端能力；不依赖外部 CDN 和在线聊天服务。

## 验收口径

- 左侧专业 Skill 选择器下方、传统业务模块上方存在一级入口“智算助手”，点击后进入宽屏会话工作台。
- 右侧问问智算可以通过“进入智算助手”进入同一工作台；两个入口不创建两套会话。
- Dock 与专注模式共用同一消息、草稿、等待状态、任务和当前行上下文。
- 模式切换和退出不丢失当前 Excel 处理结果，不重复创建任务。
- 底部可以查看和选择真实专业能力；规划中能力不可执行；任务创建后 Skill 锁定。
- 用户可以在会话上传指定样例、说明目标、确认处理、运行批量匹配和预警、生成并下载 Excel / Word。
- 每个专业动作都显示真实阶段、结果或错误，不用普通模型文本伪造“已经执行”。
- 待复核项可以在会话查看摘要，并进入结果预览定位处理后返回。
- 大模型不可用时，确定性动作和专业页面仍能使用。
- 当前 100 行关键样例通过 Dock / 专注模式调用时，输出与既有专业页面逐工作表、逐单元格一致。
- 未知、规划中、版本冲突或运行时资产不完整的 Skill 不能创建会话任务。
- 前端生产构建、后端专项、全量回归、键盘操作和浏览器运行态验收通过；已知历史失败必须与本模块新增回归分开说明。

- 不在 P0 建设类似 ChatGPT 的公开插件市场、模型选择市场或无限工具调用。
- 不照搬 ChatGPT / Codex 品牌、图标、颜色和产品导航，只吸收通用会话工作台模式。
