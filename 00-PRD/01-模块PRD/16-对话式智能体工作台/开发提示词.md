## 一、任务目标

在造价智算左侧新增正式一级入口“智算助手”，把现有右侧“问问智算”扩展为一个保留左侧导航的宽屏对话式智能体工作台。

最终应形成两种互补形态：

1. 右侧“问问智算”Dock：服务当前专业页面，适合随手咨询、解释当前行和触发快捷动作。
2. 左侧“智算助手”：完整对话工作台，除左侧导航外占满其余区域，用户可以选择专业 Skill、上传文件、用自然语言发起任务、查看真实进度并获得成果。

两个入口必须复用同一消息流、输入草稿、当前任务、当前文件、当前行、等待状态和 Skill 快照。不得创建第二套互不相通的聊天记录。

## 三、P0 必做范围

### 1. 左侧正式入口

- 在左侧专业 Skill 选择器下方、传统业务模块上方新增一级入口“智算助手”。
- 菜单名称固定为“智算助手”，不要显示“全屏化智算助手”“智能体专注模式”等技术形态名称。
- 使用与现有左侧导航一致的图标、字号、间距、悬浮态和克制选中态。
- 建议扩展现有 `DaweibaModuleId`，使用清晰、稳定的模块 ID；不要建立独立路由体系或复制主壳。
- 点击左侧其他模块时退出宽屏布局并展示相应专业页面；再次点击“智算助手”时恢复原会话和任务状态。

### 2. 右侧快捷入口

- 在现有右侧问问智算顶部身份区增加“进入智算助手”动作。
- 它与左侧入口进入同一个工作台状态，不能创建新会话或复制消息数组。
- 右侧 Dock 保持原有折叠、宽度、知识问答、快捷指令和当前行解释能力。

### 3. 宽屏工作台布局

- 进入“智算助手”后保留左侧导航，隐藏但不卸载原中间专业工作区。
- 不再单独显示右侧 Dock；同一个消息流成为左侧以外的主工作区。
- 不使用覆盖全页面的 Modal，不复制一份 `chatMessages`，不靠两个输入框双向同步。
- 顶部紧凑展示“智算助手”、当前专业能力、当前任务或未创建状态、新会话和退出动作。
- 中部消息区独立滚动；普通文本保持易读行宽，现有动作按钮、风险、成果和错误信息允许更宽。
- 底部 Composer 固定，不能随消息滚出屏幕。
- `Esc` 返回进入前的专业模块；存在输入草稿、附件或待确认动作时不得清空。

### 4. 底部专业能力与上下文

- Composer 至少包含：附件、专业能力、任务上下文、多行输入、发送 / 停止。
- 专业能力数据必须复用现有后端 Skill Registry 和当前前端 Skill 状态，禁止在新组件中维护第二份静态 Skill 清单。
- `active` Skill 可使用；`planned`、`disabled` 只可查看、不可执行。
- 当前已有任务继续使用任务创建时的 Skill 快照；切换 Skill 只影响新任务。
- 当前专业 Skill P0-B 运行时路由尚未全部完成，因此界面只能诚实声明当前勘察测量能力可执行，不能把规划中的通用服务类包装成已经可运行。

### 5. P0 可用闭环

本任务不能停留在“左侧多了一个菜单、页面看起来像聊天”。至少跑通当前勘察测量 Skill 的以下真实闭环：

```text
进入智算助手
→ 看到与右侧 Dock 相同的历史消息
→ 选择当前真实 Skill
→ 通过附件入口复用现有 Excel 上传逻辑
→ 输入或点击现有确定性指令开始转换 / 批量匹配
→ 显示真实等待状态和后端返回结果
→ 可以进入结果预览、运行预警、生成报告或下载已有成果
→ 切回专业页面继续处理
→ 再回智算助手时上下文不丢失
```

- 优先复用现有 `sendChatMessage`、上传处理函数、快捷指令、任务状态和成果动作。
- 不允许仅用模型文字假装“已经处理”“报告已生成”。只有收到真实后端响应或任务状态后才能显示完成。
- 如果某项现有能力尚无可复用接口，应明确显示不可执行原因，并在 PRD 中保持 `[待开发]`，不得伪造结果。

### 6. 组件边界

不要继续把所有新 JSX 堆进 `frontend/src/App.tsx`。可以按现有工程风格建立：

```text
frontend/src/components/agent-workspace/
├─ ConversationalAgentWorkspace.tsx
├─ AgentComposer.tsx
├─ AgentMessageStream.tsx
└─ agentWorkspace.css
```

组件名称可根据真实结构微调，但必须遵守：

- `App.tsx` 负责现有全局状态、模块切换和业务回调桥接。
- 工作台组件通过明确 props 接收同一消息、任务、Skill 和动作，不自己复制业务状态。
- 不复制 `FillEngine`、知识问答、报告生成、经验池或飞书逻辑。
- 不为了本功能引入新的大型状态管理库、聊天 UI 库或在线 CDN。

## 四、不可突破的业务边界

- 不修改三个数字匹配规则及输出口径。
- 不修改结构化计价库、规则表、经验池母版、正式知识资料和报告模板。
- 不让大模型直接写 Excel、裁决价格、调整系数或风险等级。
- 不把聊天记录自动写入项目知识记忆；继续使用既有“知识候选＋人工确认”治理流程。
- 不删除现有填价、结果预览、经验池、工作量抓取、Word 报告、知识库问答、智能协同和数字化项目助手页面。
- 不破坏右侧问问智算 Dock，也不建立第二套消息、任务或 Skill 状态。
- 不顺手修复与本任务无关的历史问题。

## 五、视觉与交互要求

- 遵循大尾巴主题：白底、浅灰分区、黑灰文字、1px 轻边框、低圆角、无常驻阴影、少量蓝色主操作。
- 不照搬 ChatGPT 或 Codex 的品牌、图标和配色，只吸收“消息居中、底部输入固定、附件与上下文靠近输入、任务进度留在会话流”的布局规律。
- 左侧“智算助手”是统一任务入口，应在信息层级上高于传统业务模块，但不能用高饱和大色块抢占视觉。
- 普通消息继续左灰右蓝；专业动作、进度、风险、成果和错误作为结构化行展示，不做卡片套娃。
- 至少验证普通笔记本宽度、宽屏窗口和窄窗口缩放；不得写死只在某个分辨率成立的像素布局。
- 键盘要求：`Enter` 发送、`Shift+Enter` 换行、`Esc` 退出；按钮和选择器具有可访问名称和可见焦点。

## 六、验证要求

开发前记录基线，开发后至少执行：

```powershell
npm run build
npm run test:report-preview
```

如果修改后端，再执行相关专项测试和：

```powershell
python -m pytest backend/tests -v
```

必须进行浏览器运行态验收，固定使用项目约定端口 `8000` 和 `5174`，检查：

1. 左侧“智算助手”位置、名称、选中态正确。
2. 左侧入口和右侧快捷入口进入完全相同的会话工作台。
3. Dock 消息与宽屏消息一致，切换前后没有重复欢迎语、重复请求和状态丢失。
4. 切换填价、预览、报告等模块后，原页面状态不被卸载或清空。
5. 底部 Skill、附件、上下文、输入和发送 / 停止在不同宽度下不溢出。
6. 当前真实 Excel 样例可以从会话入口走现有确定性处理链，结果与专业页面一致。
7. 规划中 Skill 不可执行，任务创建后 Skill 锁定。
8. DeepSeek 不可用时，确定性动作和专业页面仍可使用。

运行：

```powershell

git diff --check
```
