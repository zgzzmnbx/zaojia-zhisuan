# 问问智算 AI 助手 PRD

## 模块目标

在主流程旁提供随行式智能问答、行级解释、风险报告、知识库依据解释和可治理的通用/项目知识记忆，让用户在不离开填价页面的情况下完成复核理解，并将明确认可的经验持续用于后续问答。

后续通过“对话式智能体工作台”扩展为 Dock / 专注双形态：仍复用本模块同一消息流、知识问答和治理边界，不新建第二套聊天；文件、专业 Skill、任务动作、进度和成果卡由模块 16 承载。

参赛口径中，本模块承担“把翻文件变成问依据”“把全表人肉核对变成风险聚焦”的能力表达。它服务新人快速理解业务口径，也服务骨干人员快速调取依据和复核建议。

## 问答模式与路由规则

“问问智算”不是单一自由聊天框，而是右侧 AI Dock 内的智能调度入口。用户在同一个输入框中提出问题或点击快捷指令后，系统应先判断请求类型，再送入对应能力。

### 当前模式

| 模式 | 触发方式 | 系统行为 | 边界 |
| --- | --- | --- | --- |
| 快捷指令模式 | 点击快捷按钮，或在聊天框输入任何包含“批量匹配”“经验池预警分析”“输出风险报告”“输出excel表格”“输出word报告”等动作指令的文本 | 复用现有批量匹配、预警、风险报告和下载能力，并在聊天窗口显示执行进展 | 不交给大模型自由裁决，不绕过原业务流程 |
| 强制知识库模式 | 输入以 `查库：`、`查库:`、`@知识库`、`#知识库` 开头的问题 | 去掉触发前缀后，强制检索本地知识库、规则说明和结构化规则卡片，再基于证据组织回答 | 无证据时固定提示人工复核，不允许模型凭常识补依据 |
| 自动知识库问答模式 | 用户询问“依据、标准、来源、为什么、0.22、第二层、待复核、预警、不能连乘”等问题 | 自动走本地知识库问答路径，必要时携带当前行上下文 | 不承诺所有普通问题都查库，但依据类问题应优先查证据 |
| 普通自由问答模式 | 不命中动作指令、强制查库和依据类问题的普通表达 / 总结 / 润色请求 | 调用大模型进行解释、总结或文本组织 | 不裁决最终价格、系数或规则命中结果 |
| 行级 AI 复核模式 | 用户点击表格行级 AI 按钮，或在行级上下文存在时追问当前行 | 只填入行级复核问题或使用当前行上下文回答，由用户手动发送 | 不自动发送，不自动改写行结果 |
| 风险报告模式 | 输入或点击“输出风险报告”“生成 AI 审查摘要”等 | 汇总转换结果、预警、费用和知识库依据，生成审查报告文本，并照常写入 Word | 风险报告只做审查说明，不修改 Excel 填价结果 |

### 路由优先级

同一条输入应按以下顺序识别，避免动作指令、查库请求和普通聊天互相误触发：

1. 强制知识库前缀优先。只要以 `查库：`、`查库:`、`@知识库`、`#知识库` 开头，就必须进入强制知识库模式。
2. 其次识别动作类快捷指令。用户点击快捷按钮，或聊天框输入文本中包含批量匹配、经验池预警、风险报告、Excel 下载或 Word 下载等已定义动作指令时，都必须执行对应现有动作，而不是进入普通聊天。
3. 再识别依据类 / 解释类问题，进入自动知识库问答模式。
4. 最后才进入普通自由问答模式。

示例：用户输入 `查库：输出风险报告怎么生成？` 时，应按强制知识库模式处理，而不是触发“输出风险报告”动作。

### 统一反馈要求

- 长任务应在聊天窗口显示阶段状态，例如检索依据、整理证据、生成回答、正在导出等。
- 可动虚拟形象可以同步表达长任务阶段，但必须与文字阶段提示并存；形象动作只做辅助反馈，不能作为唯一状态提示。
- 知识库问答应尽量展示依据来源；强制知识库模式必须明确显示“已调用知识库”或同类状态。
- 大模型不可用时，应提示“智算辅助暂不可用”或明确错误原因，Excel 填价、预警、下载和 Word 主流程继续可用。
- 所有模式都必须遵守“规则裁决、模型解释、人工兜底”：大模型不得反向修改三个数字，不得自动写回正式知识库。

## 强制知识库模式

### 触发词

当用户在“问问智算”输入框中以以下任一前缀开头时，进入强制知识库模式：

- `查库：`
- `查库:`
- `@知识库`
- `#知识库`

触发词只作为模式开关，不作为最终问题内容。程序应去掉触发词和紧随其后的空白，再把剩余问题用于检索和回答。

### 行为要求

- 强制知识库模式必须先检索本地知识库、规则说明、结构化规则卡片和必要的当前行上下文，再调用大模型组织回答。
- 前端和后端都应能识别触发词；前端用于即时切换 UI 和请求参数，后端用于兜底，避免其他入口绕过规则。
- 回答界面需要明确显示“已调用知识库”或同类状态，降低用户对回答来源的误解。
- 如果没有检索到有效证据，回答必须明确说明“当前知识库未找到明确依据，需要人工复核”，不得让大模型凭常识补数字、补规则或补标准出处。
- 非触发词问题保持现有普通助手体验；可以做解释、总结、写材料或调用已有快捷指令，但不承诺一定查库。

### 设计边界

- 强制知识库模式只影响“问问智算”问答路径，不改变 Excel 填价、三数字匹配、经验池预警、工作量抓取和 Word 报告生成逻辑。
- 大模型仍只做依据解释和表达组织，不裁决最终单价、基价或两个调整系数。
- 不自动把问答记录写回正式知识库；后续如需沉淀高频问答，应另设人工确认流程。

## `@知识库` 回答质量三层优化

### 第一层：功能类知识卡

当前知识库不能只收标准、规则和价格依据，也要收“造价智算自己的功能说明”。这些功能类知识卡用于回答用户对系统运行机制的提问，例如：

- “输出风险报告怎么生成？”
- “行级 AI 复核会不会自动改结果？”
- “输出 Excel / Word 是否经过大模型改写？”
- “问问智算有哪些模式？”
- “大模型不可用会不会影响主流程？”

功能类知识卡必须进入 `backend/app/knowledge_qa.py` 的本地检索源，但只能解释系统行为，不反向驱动业务逻辑。

### 第二层：行级上下文增强

行级 AI 复核不能只携带当前预览中肉眼可见的列。点击表格末列 AI 按钮时，应尽量把本行关键诊断字段一起送入“问问智算”上下文，包括：

- `匹配状态`
- `匹配说明`
- `候选数量`
- `预警参数`
- `预警细节`
- `基价 / 单价`
- `实物工作费调整系数`
- `技术工作费调整系数`

这些字段只用于解释“为什么待复核、为什么预警、依据在哪里”，不得自动改写行结果，也不得绕过用户手动发送。

### 第三层：检索和反馈优化

知识库检索需要覆盖业务依据词和功能操作词两类表达。除 `0.22`、`第二层`、`不能连乘`、`待复核`、`预警` 等业务词外，还应补充风险报告、审查摘要、导出、下载、行级 AI、快捷指令、问问智算模式等功能词。

检索结果应尽量标注合适模块。若问题包含 `不能连乘`、`附加调整系数` 或 `实物工作费`，应优先归入实物工作费调整系数；若问题涉及 `问问智算`、`@知识库`、强制知识库或大模型模式，应归入问问智算。

当没有有效证据时，仍固定输出“当前知识库未找到明确依据，需要人工复核。”，不让模型凭常识补标准、补价格或补系统功能。

### 第四层：结构化计价库优先和模糊候选

当用户在 `@知识库` 中询问“多少钱、单价、基价、价格、收费”等价格类问题时，应优先把 `03-知识库-二维数据库制作/【数据库】【导入】.xlsx` 作为结构化计价库候选池检索。用户条件足够明确时，回答应优先引用结构化计价库精确行，说明 `序号`、`要素1-5`、`单位`、`基价`、`实物工作费调整系数`、`技术工作费调整系数` 等字段。

当用户问题比较模糊，例如只问“控制测量 GPS E级多少钱”而没有给出复杂程度、单位、场景或比例尺时，不应直接回答“知识库未找到依据”。系统应尽量列出相似候选项供用户选择，并提示需要补充哪些条件。候选项应来自结构化计价库或标准资料，不允许模型自行编造价格。

典型验收问题：

- `@知识库，地形图测绘（地形测量） 山岭隧道（洞身） 复杂 1:2000 一般单价多少？` 应优先命中结构化计价库序号 118，基价 `14244`。
- `@知识库，控制测量 首级控制测量 GPS测量E级 中等 多少钱？` 应优先命中结构化计价库序号 2，基价 `3203`。
- `@知识库，控制测量 GPS E级 多少钱？` 属于模糊问题，应返回若干 `GPS测量E级` 相关候选，让用户按首级控制测量、场景、简单 / 中等 / 复杂等条件选择。

## 可治理的知识记忆（P0）

### 产品目标

知识记忆不是永久保存全部聊天，也不是让大模型自行训练或修改知识库。P0 建设“知识候选池＋通用/项目知识记忆＋可治理审核”三部分，把用户明确认可的通用经验、项目口径、已解决问题和复核结论沉淀为有来源、有范围、可撤销的知识。

P0 支持本实例内的通用知识和项目内部知识延续。企业级组织发布、跨用户授权、跨部署共享、案例推荐和自动知识抽取留到后续阶段。

### 三层结构

| 层级 | 内容 | 是否参与问答 | 管理要求 |
| --- | --- | --- | --- |
| 知识候选池 | 用户主动保存的问答结论、复核意见、项目口径和系统使用经验 | 只有已确认内容参与；候选和待确认内容不参与 | 全部保留审核接口、版本和审计；低风险通用类型可配置自动通过，敏感类型和冲突项人工确认，不提供硬删除 |
| 通用知识记忆 | 不限某一项目的已确认操作经验、规则解释和问题结论 | 可以参与所有项目问答 | 使用保留范围 `通用知识`；回答标注“通用知识”、来源和确认人 |
| 项目知识记忆 | 经授权人员确认、仅对当前项目有效的知识 | 可以参与当前项目问答 | 回答必须标注“项目记忆”、来源、确认人和适用条件 |
| 正式知识库与规则资产 | 标准资料、结构化计价库、正式规则和功能知识卡 | 按现有知识库路径优先参与回答 | P0 不允许项目记忆自动写入或覆盖正式资产 |

### 可沉淀的知识类型

- 项目口径：当前项目采用的复杂程度、工作内容归类、材料边界和特殊处理原则。
- 已解决问题：问题是什么、采用了什么结论、依据是什么、适用于哪些条件。
- 复核结论：多人复核中已经闭环、具有复用价值的意见和处理说明。
- 系统使用经验：造价智算操作方式、常见报错原因和已确认的解决方法。

以下内容不得直接沉淀为项目知识记忆：

- 未经人工确认的大模型推断。
- App Secret、访问令牌、连接票据、个人敏感信息和无关群聊内容。
- 单个项目的价格或系数直接作为跨项目通用规则。
- 没有适用条件、来源或责任人的口头结论。
- 已被正式标准、规则或新版本资料明确替代的旧结论。

### 知识记忆卡片

每条候选和项目知识至少包含：

- 标题、问题描述和确认结论。
- 适用范围：通用知识、当前任务、当前项目；默认为通用知识。P0 的“通用”是本实例内可用，不等于企业级组织发布。
- 适用条件、例外情况和失效条件。
- 来源类型及定位：问答消息、项目任务、文件、Sheet、行、风险项或复核意见。
- 证据摘要和可访问的来源入口。
- 提交人、确认人、创建时间、确认时间和最近复核时间。
- 状态：候选、待确认、已确认、已驳回、已撤销、疑似失效。
- 知识版本；更新时保留旧版本和变更说明，不覆盖历史。

### P0 用户流程

1. 用户在问问智算获得回答，或在复核流程中形成明确结论。
2. 用户可点击“保存为知识候选”核对预填内容，也可直接输入“记住这一条”保存最近一条带结构化候选的回答。
3. 默认范围为通用知识，无需选择项目；如果结论只适用某项目或任务，用户在弹窗改选范围并填写项目信息。
4. 系统确定性识别知识类型，并在同范围内检查重复、相似项和不同结论；不调用大模型裁决冲突。
5. 操作方法、一般解释等低风险通用知识按配置自动确认；项目口径、价格 / 调整系数、正式标准和冲突项转人工确认。项目/任务候选始终人工确认。
6. 已确认通用知识参与所有项目问答；已确认项目/任务知识只在同一 `project_key` 内参与。回答展示范围、确认人和来源。
7. 用户可说“把上一条改成……”生成更正版本，或说“忘掉上一条”撤销；更正版本确认前旧版继续有效，确认后旧版转疑似失效。
8. 已确认项目知识可申请提升为通用候选；提升不改变原项目知识状态，并重新按通用知识类型策略审核。

### 问答检索优先级

当前项目中的依据类回答按以下优先级组织：

```text
正式标准与规则
＞ 结构化计价库
＞ 已确认的通用与同项目知识记忆
＞ 仅作参考的历史案例
＞ 大模型通用解释
```

- 项目/任务范围的知识候选、待确认和已驳回内容不得作为正式回答依据；通用候选仅在系统自动确认后参与。
- 项目知识记忆与正式标准冲突时，正式标准优先，并把项目记忆标记为“疑似失效”，进入人工复核。
- 项目知识记忆只能在所属项目和授权用户范围内检索，不能跨项目、跨群聊或跨用户泄露。
- 回答同时命中正式依据和项目记忆时，应分区展示，不能把项目口径伪装成国家、行业或企业正式标准。

### 界面与交互

- 继续使用现有右侧“问问智算”聊天流，不新增独立一级菜单或第二个聊天窗口。
- 在有价值的知识库回答和行级解释下保留“保存为知识候选”；识别“记住这一条”“只在当前项目记住”“把上一条改成……”“忘掉上一条”“查看刚保存的知识”等确定性口令，普通闲聊不自动保存。
- 在问问智算中提供轻量“知识记忆”入口，展示当前项目已确认知识、待确认候选和疑似失效项。
- 按钮方式保存前展示即将保存的内容、范围和来源，默认为本实例通用知识；项目/任务范围需用户显式改选。
- 回答使用项目记忆时，用户可以查看来源、提出纠错、撤销记忆或进入候选确认记录。
- P0 的人工确认优先在网页完成；WeAct 后续可以发送候选待确认提醒和安全链接，但不在消息中直接批准正式知识。

### v5.9.2 实现状态与实际边界

- 后端已使用 SQLite 建立 `knowledge_items`、`knowledge_versions`、`knowledge_audit` 三张表，覆盖候选、待确认、已确认、已驳回、已撤销和疑似失效状态。
- 前端在右侧“问问智算”保留按钮保存，并支持通用/项目保存、查看、更正和撤销口令；默认范围仍为通用知识，无需选项目。
- 后端按操作方法、一般解释、项目口径、价格/系数、正式标准五类执行审核策略；默认只有前两类通用知识可自动通过，配置位于 `knowledgeMemory.autoApproveTypes`。
- 同范围内容创建前执行确定性去重、相似项和冲突检查；完全重复复用已有记录，明显不同结论强制人工确认。正式资料检索只提供确认线索，不由模型自动裁决。
- 更正已确认知识会创建父子版本，新版本确认后才把旧版转为疑似失效；项目知识提升通用会创建 `promotion` 关系，不让原项目知识失效。

- 项目记忆数据库不可用时，接口降级为原有正式知识库问答，不影响 Excel 填价、经验池预警、Word 报告和其他专业主流程。
- 当前“操作人”和“确认角色”是本地试点字段，用于流程演示和审计留痕，不构成企业身份认证。跨用户、跨组织的真实授权仍需后续接入企业登录或 WeAct 身份映射。

### P0 验收口径

- 用户可以从一次知识库问答或复核结论点击按钮创建知识候选，并看到范围与来源自动回填。
- 用户输入“记住这一条”时，系统保存最近一条带结构化候选的回答；没有可保存回答时不创建空知识或闲聊记录。
- 低风险通用候选可自动确认，审计包含 create、submit、confirm；敏感通用候选和项目/任务候选未经人工确认时不参与问答。
- 完全重复内容不创建第二条知识；冲突内容必须进入人工确认并返回可核对的既有知识。
- 更正版本确认前旧版保持有效，确认后旧版停止检索；提升通用后原项目知识仍保持已确认。
- 授权人员确认后，同一项目内的相似问题可以命中该项目知识，并明确标注“项目记忆”。
- 其他 `project_key` 不能检索该项目知识；跨用户、跨组织授权由后续企业身份能力补齐。
- 项目记忆与正式规则发生冲突时，回答采用正式规则，并将冲突项转为疑似失效。
- 用户可以撤销已确认知识；撤销后不再参与检索，历史确认和使用记录仍可追溯。
- 知识记忆功能不可用时，现有知识库问答、Excel 填价、风险预警、Word 报告和 WeAct 任务主流程继续可用。

## 功能边界

- 大模型只解释、不裁决。
- 知识库问答先由本地程序检索证据，再让大模型基于证据回答。
- 快捷指令只触发已有功能，不让大模型绕过业务流程。
- 智算聊天流中的风险摘要跳转只调用前端预警定位能力，不新增独立聊天窗口，不改变预警或匹配结果。
- 结构化风险清单 MVP 只整合现有待复核、预警和规则说明线索；不得反向改写行结果或经验池。
- 标准依据轻量追溯 MVP 只返回匹配说明、经验池来源和规则说明入口；它不等同于 Claude 建议的“规则资产依据列 / 依据出处映射”完整标准。
- 标准依据追溯优先并入辅助填价弹窗，而不是新增主页面常驻模块：第一步服务基价 / 单价，第二步再服务实物工作费调整系数和技术工作费调整系数。
- 辅助填价后续扩展以 `09-辅助填价与人工复核闭环/PRD.md` 为主入口；本模块只维护问问智算在候选解释、标准依据追溯和 AI 不越权方面的边界。
- 标准依据完整出处映射进入 P2 可信底座增强；实施时应优先补规则资产出处字段或独立映射表，再由行级详情 API 展示，不能让大模型临场编造标准出处。
- 可动虚拟形象已作为 v5.5.0 体验增强落地；它属于右侧智算侧栏的身份和状态层，不参与路由判断、知识库检索、风险评级、价格 / 系数候选生成或任何写回动作。
- 智算窗口设置统一由 `config/project-default-settings.json` 的 `zhisuanWindow` 提供；设置页临时调整不写入浏览器 localStorage，欢迎语不再通过独立静态文件维护。
- API Key 不写入前端代码和代码存档。
- P0 允许用户通过按钮或明确口令创建知识候选；通用范围当前自动确认，项目/任务范围人工确认。任何候选或知识记忆都不得自动改写正式知识库、规则表、结构化计价库或经验池。

## 验收口径

- 行级 AI 按钮只填入问题或展开助手，不自动发送。
- 行级 AI 复核完成后的“详细情况”按钮只打开辅助填价详情弹窗，不在问问智算对话框或聊天气泡中承载候选选择或写值动作。
- 本地知识库无明确依据时，提示需要人工复核。
- 大模型不可用时，不影响 Excel、预警、下载和 Word 主流程。
- `zhisuanWindow` 必须覆盖聊天区高度、Dock 宽度、纵向高度偏好、欢迎语、快捷指令、外观风格和显示项；默认 Dock 宽度为 `400px`，绿色版和云端发布均必须携带该文件。
- 大模型设置项保存后重开页面仍有效。
- 回答应尽量说明依据来源、匹配逻辑、异常原因、经验偏离和建议处理方式。
- 同一条输入同时像动作指令和查库问题时，应按路由优先级处理；强制知识库前缀优先于快捷指令。
- 快捷指令、知识库问答、普通问答、行级复核和风险报告应在同一个“问问智算”聊天流中反馈，不新增独立聊天窗口。
- 用户在聊天框输入包含任一已定义动作快捷指令的文本后按 Enter，应与点击对应快捷按钮执行同一动作；`查库：`、`查库:`、`@知识库`、`#知识库` 等强制查库前缀仍优先于动作指令识别。
- 虚拟形象验收：待机、检索、思考、命中、预警、完成等状态至少有 4 类可区分动作或表情；关闭 / 降低动效后不影响问答和快捷指令；在普通笔记本宽度下不挤压主表格、不遮挡输入框和消息流。
- 高风险 / 低风险摘要明细中的“跳到表格”应能直接切到结果预览并闪烁目标行，行为与经验池预警模块的跳转按钮一致。
- 结构化风险清单至少应包含风险类型、等级、sheet、行号、指标、说明、建议动作和证据线索；风险等级必须来自待复核状态或经验池偏离规则，不由大模型判断。
- 标准依据轻量追溯能展示匹配说明、经验池来源和项目规则说明入口即可；完整出处映射验收另按 P2 需求执行，必须能展示规则资产维护的标准依据或“暂无标准出处映射”。
- 价格追溯并入辅助填价验收：点击书本图标后，候选列表和依据追溯在同一 Dialog / Modal 内展示；追溯不得挤占主页面，不得伪装成完整标准条款。
- 以 `查库：`、`查库:`、`@知识库`、`#知识库` 开头提问时，必须走知识库问答路径，并在界面中显示已调用知识库。
- 强制知识库模式无有效证据时，必须输出固定的未找到依据提示，不得输出看似确定的标准条款、价格或系数。
