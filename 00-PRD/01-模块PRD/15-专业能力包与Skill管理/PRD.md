# 专业能力包与 Skill 管理 PRD

## 模块目标

把“造价智算”从单一勘察测量场景的软件原型，升级为可装载不同造价专业能力的智能体底座。用户开始业务前选择“专业能力（Skill）”，系统据此确定本次任务使用的输入规范、结构化计价库、规则、风险检查、知识资料、报告模板和验收口径。

当前长输管道勘察测量最高投标限价编制能力作为首个已验证的专业 Skill。v5.10.0 已完成 P0 能力选择入口、Skill 清单、安全校验和任务运行上下文，未移动现有正式资产、未改变现有计算结果；P1 再以“通用服务类造价测算”验证第二个真实专业 Skill。

对外产品口径：

> 造价智算不是一个勘察测量填价工具，而是一套可装载专业 Skill 的工程造价智能体底座；勘察测量最高投标限价编制，是首个经过完整验证的专业 Skill。

## v5.10.0 实现状态与实际边界

- P0 已于 2026-07-20 完成。前端在上传之前加载能力卡片和详情，主工作区顶部持续展示当前选择或已锁定任务快照。
- 后端新增独立 Skill Registry，校验 Manifest 完整性、状态、项目内相对路径、资产存在性、路径越界、可执行入口和运行秘密；未知、规划中、停用、版本不匹配或不完整能力均不能创建真实任务。
- `process-state.json` 固化 Skill ID、版本、Manifest SHA-256 摘要、创建时间和资产标识；处理、预览、预警、风险、报告和下载按该任务快照追溯。
- 现有勘察测量结构化计价库、三数字匹配引擎、规则表、经验池和报告模板未改动。真实 100 行样例的旧请求与显式 Skill 请求已逐工作表、逐单元格对比一致。
- 绿色版构建显式携带 `business-skills/`；Tauri 发布复用绿色版阶段目录，因此使用同一份非秘密能力清单。

## 产品概念与命名

### 用户看到的名称

- 一级名称：`选择专业能力`。
- 辅助标签：`Skill`。
- 当前状态：`当前专业能力：勘察测量最高投标限价编制`。
- 不建议把入口直接命名为“选库”，因为用户选择的不只是数据表。
- 不建议只展示英文“Skill”，避免普通业务人员不理解。

### 产品内部定义

一个专业 Skill 是完整的业务能力包：

```text
专业 Skill
＝业务场景
＋输入规范
＋结构化计价库
＋规则包
＋字段映射
＋风险检查
＋知识资料
＋报告模板
＋标准流程
＋验收用例
```

### Skill、工具与资产的边界

| 对象 | 产品定位 | 是否直接称为 Skill |
| --- | --- | --- |
| 勘察测量最高投标限价完整业务闭环 | 用户可选择的专业能力包 | 是，首个专业 Skill |
| 通用服务类完整测算闭环 | 第二个横向扩展能力包 | 是，P1 专业 Skill |
| 结构化计价库 | 专业数据资产 | 否，由专业 Skill 引用 |
| 确定性匹配引擎 | 通用业务工具或服务 | 否，由专业 Skill 编排调用 |
| 调整系数规则表 | 可审计规则资产 | 否，由专业 Skill 引用 |
| Word 模板 | 成果资源 | 否，作为 Skill 资产 |
| 问问智算与知识记忆 | 跨专业共享智能能力 | 可作为通用能力 Skill，但不取代正式依据 |
| 智能协同与 WeAct | 协同流程加外部连接能力 | 更接近 Skill 与连接器组合的 Plugin |

## 功能边界

| 范围 | 本模块负责 | 本模块不负责 |
| --- | --- | --- |
| 用户入口 | 专业能力选择、详情查看、状态展示和当前能力标识 | 重做现有填价、预警、报告和问答页面 |
| 能力登记 | 通过 Manifest 登记 Skill 元数据、资产引用、版本和验证状态 | 在自然语言描述中维护可执行价格规则 |
| 任务上下文 | 把 Skill ID、版本和关键资产标识固化到新任务 | 切换或改写已经创建任务的专业上下文 |
| 资产治理 | 引用现有结构化计价库、规则表、知识资料和模板 | P0 搬迁、复制或重组现有正式资产 |
| 横向扩展 | 为第二个真实专业 Skill 提供统一装载协议 | 在通用服务类业务闭环未验证前伪装成已上线能力 |
| 平台生态 | P0 预留可扩展结构，P2 再建设管理中心 | P0 安装第三方包、运行任意脚本或开放能力市场 |

## 总体架构

```text
造价智算通用底座
├─ Excel 解析、字段映射和任务状态
├─ 确定性规则执行与人工复核
├─ 经验池、风险清单和依据追溯
├─ 问问智算、知识记忆和报告生成
├─ 智能协同与成果分发
└─ 专业 Skill 注册、选择和隔离
   ├─ 勘察测量最高投标限价 Skill（P0 已上线）
   ├─ 通用服务类造价测算 Skill（P1）
   └─ 其他造价专业 Skill（P2 储备）
```

通用底座保持稳定；不同专业 Skill 只装载自己的专业输入、数据、规则、模板和验收资产。共享能力可以复用，但不得用共享经验替代专业正式规则。

## 专业能力包结构

P0 推荐新增业务能力包目录，不立即迁移现有真实资产：

```text
business-skills/
└─ survey-measurement-limit-price/
   ├─ manifest.json
   ├─ SKILL.md
   ├─ references/
   ├─ assets/
   └─ evals/
```

- `manifest.json`：造价智算程序确定性读取的运行清单，保存 ID、版本、状态、能力开关和现有资产相对路径。
- `SKILL.md`：面向智能体和开发人员的工作说明，描述适用场景、所需输入、执行步骤、输出要求、边界和检查项。
- `references/`：按需放置专业说明或对现有正式资料的引用说明。
- `assets/`：只放该 Skill 新增且不存在稳定外部路径的模板或静态资产；P0 不复制现有结构化计价库和正式规则。
- `evals/`：保存专业 Skill 的样例清单、预期行为和回归说明。

P0 的 `SKILL.md` 参考 Agent Skills 开放格式，但程序不得把 Markdown 自然语言当成价格和系数裁决依据。所有机器运行参数必须来自结构化 Manifest、正式规则和确定性代码。

外部规范参考：

- [Agent Skills Specification](https://agentskills.io/specification)
- [OpenAI Skills](https://help.openai.com/en/articles/20001066)
- [OpenAI Plugins in Codex](https://help.openai.com/en/articles/20001256-plugins-in-codex/)

## P0 详细方案：选择专业能力

### P0 目标

P0 只解决“产品架势、能力清单、运行上下文和兼容隔离”，不在本阶段同时开发第二套通用服务类业务规则。

P0 完成后，评委和用户应明确感知：

1. 造价智算是一套可扩展底座，不是只服务一张勘察测量表。
2. 当前真正上线的是勘察测量最高投标限价 Skill。
3. 未来新增专业场景时，通过新增能力包扩展，而不是复制整套前后端。
4. 每次任务明确记录使用了哪个 Skill、哪个版本和哪些专业资产。

### 入口位置

主入口放在“填价工作台”开始区域、文件上传之前：

```text
选择专业能力                                      [查看能力详情]

[勘察测量最高投标限价编制]     [通用服务类造价测算]
 已上线 · 默认                    规划中
 规则裁决 / 风险复核 / 报告       下一项横向验证能力

当前专业能力：勘察测量最高投标限价编制 · v1.0
```

交互要求：

- 只有一个可用 Skill 时默认选中，但选择入口仍可见。
- 用户点击已上线能力卡后查看详情并确认。
- 规划中能力卡不可进入业务主流程，只展示规划说明。
- 用户选定后，在主工作区顶部显示紧凑的“当前专业能力”状态，不占用左侧一级菜单。
- 不要求用户在每个页面重复选择；同一任务生命周期保持不变。
- 切换专业能力前，如已有未完成任务，必须提示“仅影响后续新任务”，不得改变已创建任务。

### 能力详情

P0 能力详情至少展示：

- Skill 名称、ID、版本、状态。
- 适用业务和不适用范围。
- 支持的输入文件类型和关键模板特征。
- 已启用能力：字段映射、三数字匹配、经验池预警、报告、问答、协同。
- 已加载专业资产数量与名称摘要。
- 最后验证时间、验证样例和已知限制。
- “规则裁决、模型解释、人工兜底”边界说明。

不得展示 App Secret、API Key、服务器内部路径或其他运行秘密。

### 专业能力卡状态

| 状态 | 行为 | 展示要求 |
| --- | --- | --- |
| 已上线 | 可选择并创建任务 | 显示版本、验证状态和主要能力 |
| 内测中 | 仅授权试用或样例验证 | 明确不是正式生产能力 |
| 规划中 | 不可创建真实任务 | 只展示方向和预计包含的资产，不伪装为已实现 |
| 已停用 | 禁止创建新任务 | 历史任务仍可按原 Skill 版本追溯 |

## P0 Manifest 设计

P0 Manifest 至少包含：

```json
{
  "id": "survey-measurement-limit-price",
  "displayName": "勘察测量最高投标限价编制",
  "version": "1.0.0",
  "status": "active",
  "domain": "工程造价/勘察测量",
  "description": "长输管道勘察测量最高投标限价编制专业能力",
  "inputProfile": {
    "extensions": [".xlsx"],
    "templateHints": []
  },
  "capabilities": {
    "pricing": true,
    "workloadCapture": true,
    "experienceWarning": true,
    "knowledgeQa": true,
    "knowledgeMemory": true,
    "wordReport": true,
    "collaboration": true
  },
  "assets": {
    "knowledgeBase": "现有结构化计价库相对路径",
    "rules": [],
    "reportTemplate": "现有报告模板相对路径"
  },
  "validation": {
    "status": "verified",
    "sample": "当前100行空单价输入表",
    "updatedAt": "待实现时填写"
  }
}
```

实际字段可在开发前结合现有后端收敛，但必须满足：

- 全部资产路径使用项目相对路径或跨平台配置，不写死本机盘符。
- Manifest 不保存密钥和用户隐私。
- Manifest 读取失败时给出明确错误，不静默加载半套规则。
- `status` 不是装饰字段，后端必须禁止使用规划中或停用 Skill 创建任务。
- P0 不允许 Manifest 覆盖任意 Python 模块或执行外部命令。

## P0 运行流程

```text
进入填价工作台
→ 读取专业 Skill 清单
→ 默认选择勘察测量 Skill
→ 用户查看或确认
→ 上传 Excel
→ 前端把 skill_id 和 skill_version 传入任务请求
→ 后端校验 Skill 状态和 Manifest
→ 生成任务级 Skill 快照
→ 现有勘察测量处理链按原逻辑运行
→ 预览、报告、下载和协同沿用同一任务 Skill 上下文
```

任务创建后必须以任务快照为准。即使默认 Skill 后续升级，历史任务仍显示创建时的 Skill ID 和版本，不被新版本静默改写。

## P0 兼容与隔离

### 向后兼容

- 旧前端或 WeAct 未传 `skill_id` 时，后端使用项目默认的勘察测量 Skill。
- 默认 Skill 的现有输入、输出、匹配、预警和报告结果必须保持一致。
- P0 不修改三个数字的匹配顺序、结构化计价库内容或规则表内容。
- P0 不移动正式资产，不批量重命名历史“管勘智算”路径。

### 隔离要求

- 专业 Skill 只决定后续新任务的资产装载和能力开关。
- 已创建任务不能因用户切换 Skill 被改变。
- 知识记忆中的项目范围与专业 Skill 范围是两个维度；P0 不自动把某个项目知识提升为跨 Skill 正式规则。
- 通用知识可跨专业辅助解释，但不得覆盖当前 Skill 的正式规则和结构化计价库。
- 经验池、任务缓存、报告模板和未来专业知识库必须具备 Skill 维度，第二个 Skill 上线前完成串用防护。

## P0 非功能要求

- 首屏能力清单读取应为轻量本地操作，不因读取大量知识库拖慢页面。
- 能力选择失败不影响用户查看现有页面，但必须阻止使用未知或不完整 Skill 开始专业计算。
- 所有新增接口返回结构化错误，不向用户暴露绝对路径和堆栈。
- 绿色版、Tauri 和云端均从同一份非秘密 Skill 清单读取能力状态。
- UI 延续大尾巴主题和 Navattic 极简 SaaS 风格，不增加高饱和“应用商店”视觉。
- 选择入口支持键盘操作、焦点状态和无障碍标签。

## P1：第二个真实专业 Skill

“通用服务类造价测算”作为第二个横向验证能力，不能只复制一张卡片。P1 至少需要：

1. 独立业务定义和适用范围。
2. 独立输入模板与字段模型。
3. 独立计价依据、结构化数据和确定性规则。
4. 独立风险检查与人工复核点。
5. 独立 Excel / Word 成果模板。
6. 独立样例、预期答案和回归测试。
7. 与勘察测量 Skill 的资产隔离验证。
8. 使用同一选择入口、任务框架、问答、知识记忆、报告和协同底座。

只有最小闭环真实跑通后，卡片状态才能从“规划中”改为“内测中”或“已上线”。

## P2：Skill 管理与生态

P2 再考虑：

- Skill 安装、启用、停用和卸载。
- 版本升级、兼容性检查和回滚。
- 组织、角色和专业授权。
- Skill 签名、发布审批和供应链安全。
- Agent Skills 格式导入导出。
- 单位内部专业能力市场。
- 根据文件特征、用户角色和项目类型推荐 Skill。
- Skill 使用统计、命中率、人工复核率和质量评估。

P2 不允许用户上传任意 Skill 后直接执行其中脚本。外部包必须经过白名单、签名、权限和人工发布审核。

## 验收口径

以下为 P0 阶段验收口径：

- 填价工作台上传文件前可以看到“选择专业能力”，并显示 `Skill` 辅助标签。
- 勘察测量最高投标限价编制显示为“已上线”且默认选中；通用服务类未实现前显示为“规划中”。
- 用户能打开能力详情，看到适用范围、版本、能力、资产摘要、验证状态和边界。
- 当前专业能力在任务生命周期内持续可见，切换只影响后续新任务。
- 前端任务请求、后端任务状态和下载 / 报告元数据可追溯 Skill ID 与版本。
- 未传 Skill 的旧请求继续运行当前勘察测量流程，100 行关键样例输出与改造前逐行一致。
- 规划中、停用、未知、Manifest 不完整或路径越界的 Skill 不能创建任务。
- P0 不移动、不覆盖、不批量重命名现有知识库、规则表、经验池、模板和样例。
- 问问智算、知识记忆、智能协同、绿色版、Tauri 和云端行为不因 P0 退化。
- `SKILL.md` 只承载工作说明；任何价格、系数和风险等级仍由结构化规则和正式资产确定。

## 开发前置与验证要求

进入 P0 代码开发前：

1. 先记录当前 100 行样例、知识检索和项目样例的真实基线。
2. 先确认现有两个回归问题是资料漂移还是代码问题，新增 Skill 框架不得扩大失败范围。
3. 先检查 `frontend/src/App.tsx`、`backend/app/main.py` 和当前工作区 diff，采用增量式接入。
4. 优先新建独立的 Skill Registry / Manifest 模块，避免把大量清单解析逻辑继续堆入 `main.py`。

至少验证：

```powershell
python -m pytest backend/tests -v
npm run frontend:build
python tools/check_prd_consistency.py --strict
git diff --check
```

新增专项测试至少覆盖：Manifest 合法性、默认兼容、状态禁用、路径越界、任务快照、切换隔离、未知 Skill、绿色版携带和当前 100 行样例不回归。

## 本阶段不做

- P0 不开发通用服务类真实计价规则，只保留诚实的规划状态。
- P0 不建设 Skill 商店、在线下载、任意脚本执行或外部包自动安装。
- P0 不重构或搬迁现有勘察测量结构化计价库、规则表、经验池和报告模板。
- P0 不允许大模型根据 `SKILL.md` 自由生成价格、系数、规则或风险等级。
- P0 不把功能点助手业务代码纳入本专业 Skill 目录；数字化项目助手继续保持独立子项目边界。
- P0 不把“选择专业能力”新增为左侧一级功能页；它是开始任务前的上下文选择能力。
