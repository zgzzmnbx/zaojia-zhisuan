# 造价智算项目战略评估与未来功能路线

> 评估日期：2026年7月10日
>
> 文档性质：项目策划分析与未来功能储备，不直接替代当前版本计划和各模块 PRD
>
> 评估范围：产品总览、当前版本计划、现行模块 PRD、空模块规划及飞书协同方向

## 一、核心结论

造价智算已经不是单纯的“自动填价工具”，但还没有真正完成向“场景智能体”的跃迁。

当前产品最强的是结构化计算、风险解释和成果输出，最弱的是任务、人员、复核、审批、交付、归档这条组织流程。下一阶段不应继续单纯增加左侧菜单或堆叠孤立功能，而应补出一个贯穿全局的“造价任务中心”。

建议将产品进一步定义为：

> **造价智算——面向工程造价任务的可信协同智能体。**
>
> 围绕一项具体造价任务，完成资料接收、规则计算、风险识别、人工复核、成果审批、交付归档和经验沉淀。

产品长期北极星不是“自动填了多少价格”，而是：

> 用户把一项造价任务交给造价智算后，系统能在不越权的前提下，帮助用户收齐资料、执行规则、暴露风险、组织复核、生成成果、完成审批归档，并把可复用经验安全地沉淀下来。

## 二、项目总体评价

当前项目已经形成“规则引擎 + 经验池预警 + 大模型解释 + Word/Excel 成果输出”的造价数字员工雏形，产品定位也已经从单点脚本升级为“工程造价辅助智能体”。

从高级策划视角给出以下主观评价：

| 维度 | 评价 |
| --- | --- |
| 业务真实性 | 9/10，是真实、高频、责任明确的业务问题 |
| 可信 AI 边界 | 9/10，“规则裁决、模型解释、人工兜底”的方向非常正确 |
| 单场景执行能力 | 8/10，已经有可运行、可验证、可交付的完整原型 |
| 人工复核闭环 | 6/10，已经起步，但处理状态、责任人和复核完成仍不完整 |
| 组织协同能力 | 3/10，当前主要还是单机、单人、单次任务 |
| 平台化与治理 | 4/10，权限、用户、项目台账、规则版本和审计尚未成体系 |
| 横向复制能力 | 5/10，有方法论，但能力底座和新业务场景还没有真正解耦 |

### 2.1 当前项目最值得肯定的地方

#### 1. 找到了真实且适合结构化的业务切口

长输管道勘察测量最高投标限价编制具有规则明确、表格化程度高、重复劳动多、专业责任重等特点，非常适合作为造价智能体的第一个落地场景。

#### 2. 没有掉进“大模型直接猜价格”的陷阱

当前明确坚持：

- 基价、单价和两个调整系数由结构化规则、二维知识库和经验提示处理。
- 大模型只做解释、问答、报告润色和候选说明。
- 出现多候选、无候选和字段冲突时进入待复核。
- 人工确认值不自动污染正式知识库和规则资产。

这是产品可信性的核心，也是未来进入企业真实业务流程的基础。

#### 3. 已经形成较完整的单场景能力链

当前已经覆盖：

- 标准 Excel 读取和列映射。
- 原始工作量抓取。
- 三个核心数字匹配。
- 结果预览和人工修改。
- 辅助填价候选。
- 经验池预警和治理。
- 结构化风险清单。
- 知识库依据问答。
- Word、Excel 成果输出。
- Windows 绿色版和桌面版交付。

#### 4. 已经积累了真正有壁垒的业务资产

当前最有价值的资产不只是代码，还包括：

- 结构化规则和二维知识库。
- 规则、经验和大模型之间清楚的责任边界。
- 样例、答案表和回归测试口径。
- 待复核、经验偏离和依据解释的业务判断方式。
- Word、Excel 正式成果链路。
- 项目过程中沉淀的规则说明、术语归并和异常处理经验。

这些资产决定了造价智算有机会成为真正可推广的业务产品，而不是一次性比赛 Demo。

## 三、目前最需要改进的地方

### 3.1 产品仍然以“功能模块”为中心，还不是以“造价任务”为中心

当前左侧模块回答的是“软件有什么功能”：填价、结果预览、经验池、工作量抓取、Word 报告、知识问答。

但业务人员真正关心的是：

- 我现在负责哪个项目？
- 材料齐不齐？
- 当前使用的是哪一版资料？
- 计算到哪一步了？
- 还有多少风险没有处理？
- 谁负责编制、谁负责复核？
- 哪一版是最终成果？
- 是否已经批准、发送和归档？

因此，最缺的不是一个新算法，而是一个统一的“造价任务”对象：

```text
项目任务
├─ 基本信息与场景参数
├─ 输入资料及版本
├─ 计算运行记录
├─ 风险与待复核项
├─ 人工确认记录
├─ 审批与意见
├─ 最终成果
└─ 归档与经验回流
```

建议统一任务状态：

`待收件 → 材料待补充 → 待计算 → 计算中 → 待复核 → 复核中 → 待审批 → 已发布 → 已归档`

任务中心一旦建立，现有模块才能真正串成一个智能体。

### 3.2 “问问智算”更像随行聊天框，还没有完全成为数字员工

现有 AI Dock、结构化风险清单、知识库强制检索和快捷指令已经具备较好基础。

但真正的场景智能体不能只等待用户提问，还应主动完成：

- 发现材料缺失并追问。
- 发现规则冲突并提示。
- 计算完成后自动形成待复核队列。
- 到期未复核时提醒责任人。
- 复核完成后准备审批材料。
- 审批通过后生成交付包。
- 新制度发布后提示哪些规则和历史任务可能受影响。

下一阶段应把“聊天助手”升级成“任务编排员 + 随行复核员”。

### 3.3 可信底座还有核心缺口

当前版本计划把知识库质量检测和人工复核闭环列为 P1，方向正确。建议进一步明确四项底座工作：

1. 知识库只读质量检测。
2. 实物工作费调整系数第一层规则资料建设；没有权威来源时继续保持待复核，不强行补规则。
3. 标准依据完整出处映射。
4. 异常样例回归集：多候选、单位冲突、表头变化、版本变化和公式异常。

这些不是普通后台功能，而是产品可信度的地基。

### 3.4 人工复核闭环还差“责任和完成”

当前辅助填价已经规划了两个系数候选、自定义值备注、历史人工确认、处理状态和风险入口联动。

下一步不应只把两个系数放进弹窗，还应同时补齐：

- 复核责任人。
- 接受、修改、挂起、退回四类动作。
- 修改前值、修改后值和人工依据。
- 待重算、待二次复核、已关闭等状态。
- 操作人和操作时间。
- 未处理高风险能否阻止成果发布。
- 复核完成率和未完成清单。

否则“人工兜底”仍然只是理念，没有形成可管理的流程闭环。

### 3.5 PRD 目录混合了不同层级

当前模块 PRD 中同时存在：

- 核心业务能力：匹配、预警、工作量抓取。
- 交互载体：结果预览、AI 助手。
- 交付渠道：桌面端、绿色版。
- 比赛事项：参赛展示与申报材料。
- 跨模块闭环：辅助填价与人工复核。

这些内容并不处于同一层级。当前需求状态虽然比较完整，但部分需求在多个 PRD 中重复引用，不能简单用需求数量衡量独立能力数量。

建议以后逐步调整为：

```text
00-产品战略
01-核心能力PRD
   材料解析 / 规则计算 / 风险 / 复核 / 成果 / AI
02-场景PRD
   勘察测量限价 / 结算审核 / 估概算审核 / 数字化项目
03-协同与渠道PRD
   本地桌面 / 飞书 / 平台API
04-治理PRD
   权限 / 规则版本 / 数据质量 / 审计 / 脱敏
05-UI与体验规范
```

### 3.6 三个空模块的处理建议

#### `10-结算审核助手模块`

建议保留，但移动到“场景 PRD”。结算审核是一个新业务场景，不是普通功能模块。进入开发前至少应明确输入资料、审核对象、输出成果、审减边界、人工责任和回归样例。

#### `11-数字化项目功能点法计算模块`

建议作为独立场景包，知识库和规则与勘察测量完全隔离。第一阶段宜优先做材料检查、功能点拆分、计算过程复核和依据解释，不宜直接让大模型自由估价。

#### `12-报告预览模块`

不建议独立成一级业务场景。宜与 Word 报告合并为“成果中心”，统一承载报告预览、Excel 成果、风险清单、依据清单、附件检查和正式发布。

### 3.7 产品价值还缺少持续量化

建议从下一轮试用开始记录以下指标：

- 自动匹配率。
- 待复核率。
- 候选人工采用率。
- 高风险误报率和漏报率。
- 单项目编制时长。
- 单项目复核时长。
- 人工修改次数和集中出现的字段。
- 知识库冲突数和规则缺口数。
- Word、Excel 成果一次通过率。

这些数据既能指导产品迭代，也能成为参赛、汇报和后续推广的真实价值证据。

## 四、三种产品发展路线

| 方向 | 优点 | 问题 | 结论 |
| --- | --- | --- | --- |
| 继续做深填价工具 | 风险最低、最快形成专业壁垒 | 用户范围和组织价值有限 | 可以继续夯实，但不应成为最终形态 |
| 做“飞书连接的场景智能体” | 能把现有能力嵌入真实组织流程，推广价值最高 | 需要任务模型、用户权限和持续运行服务 | **推荐路线** |
| 直接做完整造价平台 | 想象空间最大 | 复杂度过高，容易稀释当前可信优势 | 远期方向，不宜现在直接推进 |

推荐路线是：

> 内部仍以现有造价智算引擎为核心，飞书负责人员、消息、任务、审批和触达。先做“飞书连接的场景智能体”，以后再平台化。

这与当前“先规则后模型、先助手后平台、先闭环后扩张、先可评估后规模化”的原则一致。

## 五、飞书全过程造价智能体设想

### 5.1 基本判断

飞书适合承担造价智算的“组织协同层”，不应成为价格计算引擎。

飞书自建应用可以提供机器人、网页应用、消息、文件、审批、多维表格、任务和事件回调等能力。现有造价智算继续负责规则计算、知识库检索、风险分析和成果生成，飞书负责人员连接、任务触达、流程流转和审批。

### 5.2 推荐架构

```mermaid
flowchart LR
    A["飞书发起任务 / 上传资料"] --> B["造价任务中心"]
    B --> C["现有规则引擎与知识库"]
    C --> D["结果、风险和候选"]
    D --> E["造价智算网页工作台复核"]
    E --> F["飞书派单与审批"]
    F --> G["成果发送与归档"]
    G --> H["经验池、规则治理和项目复盘"]
```

### 5.3 一项任务的完整流程

1. 编制人在飞书中给“造价智算”机器人发送“新建限价编制任务”，上传 Excel 和委托资料。
2. 机器人创建任务编号，检查资料是否齐全，并返回任务卡片。
3. 后台调用现有造价智算引擎，完成读取、匹配和风险生成。
4. 飞书消息卡片显示进度、命中率、待复核数和高风险数。
5. 复杂的 Excel 表格复核不放在聊天卡片中，而是点击进入现有网页工作台。
6. 待复核项按人员派发，处理结果同步回任务台账。
7. 复核完成后在飞书发起“成果发布审批”。
8. 审批通过后，自动发送 Excel、Word、风险摘要和依据清单。
9. 项目归档；人工确认结果先进入“经验候选池”，经过治理和审批后才允许转成正式资产。

### 5.4 飞书各能力的建议分工

| 飞书能力 | 在造价智算中的作用 |
| --- | --- |
| 机器人与消息 | 新建任务、进度通知、异常提醒、成果发送 |
| 消息卡片 | 展示任务状态、风险摘要、进入工作台、确认简单动作 |
| 网页应用 | 承载复杂表格预览、候选选择和专业复核 |
| 多维表格 | 承载任务台账、风险台账和管理视图，不存放全部核心价格资产 |
| 审批 | 成果发布、规则更新、知识入库等人工审批流程 |
| 云文档与文件 | 交付文件、报告和附件的受控共享 |
| 任务与日历 | 复核派单、截止时间和逾期提醒 |
| 事件回调 | 审批状态、消息交互和任务变化的自动同步 |

### 5.5 本地试点与生产环境边界

企业自建应用可以通过飞书 SDK 建立 WebSocket 长连接。本地服务只需能够访问公网，不一定需要公网 IP、域名或内网穿透，因此可以先在当前 Windows 电脑上做小范围试点。

但生产环境不能长期依赖某个人的电脑：

- 应部署到持续运行的企业内服务器或统一数字平台。
- 需要管理员创建自建应用并审批权限。
- 需要建立用户身份、任务权限、文件权限和操作日志。
- 敏感知识库、真实价格和完整历史文件不建议全部同步到多维表格。
- 飞书只保存必要的任务摘要、状态和受控链接，核心业务资产留在受控存储中。
- 飞书卡片适合提醒、确认、派单和审批；复杂表格编辑继续留在造价智算工作台。

## 六、未来功能发散清单

### 6.1 任务开始前

- 智能收件：读取委托书、合同、招标文件和工作量表，自动分类。
- 材料完整性清单：缺少标准、工作量和项目参数时主动追问。
- 项目参数抽取：项目名称、地区、规模、比例尺、复杂程度和计价时点。
- 适用依据推荐：提示可能适用的国家、集团、公司和合同类依据。
- 一键建档：自动建立任务编号、目录、责任人和计划节点。
- 资料版本比较：新旧工作量表、合同和控制价变化自动标红。
- 场景识别：判断任务属于限价编制、估概算审核、结算审核还是变更签证。
- 委托范围检查：识别合同、委托书和工作量表之间的范围冲突。

### 6.2 编制过程中

- 规则适用性问卷：用少量业务问题判断规则前提是否满足。
- 计算影响沙盘：切换某个系数或候选值，查看费用影响但不直接写入。
- 规则冲突雷达：发现同条件多价格、单位冲突、依据冲突和历史经验分歧。
- 工作量抓取前候选确认：解决一对多只能预警和跳过的问题。
- 多版本差异分析：比较本轮与上一轮的增减项、单价变化和总价影响。
- 自动追问模式：条件不全时先提问，不直接返回无依据或猜测结果。
- 项目级上下文：智算知道当前任务、当前版本、当前风险和当前责任人。
- 费用组成解释：按项目、专业、费用类型解释总价构成。
- 规则试算对比：展示“采用规则 A”和“采用规则 B”的差异，最终由人工决定。
- 批量异常聚类：把大量待复核行按原因归类，减少逐行重复查看。

### 6.3 复核与审批

- 待复核工作队列。
- 风险派单与责任人。
- 接受、修改、挂起和退回。
- 复核意见、证据附件和讨论串。
- 二次复核与关闭条件。
- 未关闭高风险禁止发布。
- 飞书审批发布。
- 原值、建议值、确认值、依据和人员的完整追踪。
- 复核进度和超期提醒。
- 审核意见自动草拟，但必须标明“草稿，需人工审定”。
- 复核抽样策略：低风险抽样、高风险全查，策略由管理人员配置。

### 6.4 成果与归档

- 成果中心：Word 在线预览、Excel、风险清单和依据清单统一管理。
- 一键生成交付包。
- 自动检查成果命名、日期、金额一致性和附件完整性。
- 可配置报告模板库。
- 一键生成领导摘要、业务复核版和技术审查版。
- 项目电子卷宗。
- 数据脱敏交付版。
- 项目复盘报告：统计人工修改最多的字段和最容易待复核的规则。
- 成果水印、版本号和发布状态。
- 交付后撤回与重发机制，避免多个“最终版”同时流转。

### 6.5 知识和治理

- 规则只读查看器。
- 规则版本、发布时间、生效范围和修改人。
- 新制度发布后的规则影响分析。
- 知识库冲突检测和质量评分。
- 经验池按地区、年份、项目类型和合同类型分层。
- 人工确认记录转“知识候选”，由规则维护人审批后才能入库。
- 高频问答候选库。
- 标准原文—规则—代码—结果行四级追溯。
- 规则变更后自动运行相关答案表回归。
- 规则生效日期与历史项目重算提示。
- 标准文件失效、替代和补充关系维护。
- 经验池样本量、离散度和代表性评分。

### 6.6 管理驾驶舱

- 每个项目的进度、风险、复核和交付状态。
- 自动匹配率、待复核率和人工采用率。
- 高风险误报率和漏报率。
- 平均编制时间和平均复核时间。
- 规则命中频次和争议频次。
- 不同项目同类费用指标对比。
- 个人和团队待办，用于工作分配和风险提醒，不直接做简单绩效排名。
- 项目组合风险视图。
- 规则缺口排行榜。
- 知识库和经验池健康度趋势。

### 6.7 新场景扩展

建议优先顺序：

1. **结算审核助手**：最适合复用风险清单、比对、意见草拟和人工复核。
2. **估概算审核**：核心不是回填，而是申报值与规则参考值比对。
3. **变更签证预算审核**：重点处理合同内外、单价一致性和高频小额风险。
4. **服务类项目限价与结算**：监理、检测、咨询等场景使用独立知识库。
5. **数字化项目功能点法**：独立规则包，先做材料检查、功能点计算和依据解释。
6. **多项目造价指标分析**：形成企业级指标库和异常基准。
7. **政策变更影响分析**：新文件发布后识别受影响规则、项目和历史成果。

## 七、建议的实际开发顺序

### 第一阶段：把当前场景彻底闭环

- 知识库质量检测。
- 实物工作费第一层规则资料建设。
- 两个系数辅助填价。
- 自定义值必填人工依据。
- 风险处理状态。
- 标准依据完整出处映射。
- 工作量抓取前候选确认。
- 异常回归样例集。

### 第二阶段：建立造价任务中心

- 任务编号和基本信息。
- 输入、输出、版本、风险和复核记录归集。
- 统一操作日志。
- 成果中心和报告预览。
- 任务状态机。
- 用户、角色和权限预留。

### 第三阶段：飞书轻量试点

- 飞书自建应用和机器人。
- 新建任务、上传材料和进度通知。
- 结果摘要和网页工作台跳转。
- 待复核提醒。
- 成果文件发送。

### 第四阶段：飞书组织闭环

- 复核派单。
- 多维表格任务台账。
- 飞书审批。
- 权限和责任留痕。
- 管理驾驶舱。
- 规则更新审批。

### 第五阶段：扩展新场景

先结算审核，再估概算审核，最后再考虑数字化项目功能点法和其他服务类场景。

## 八、最终建议

下一步不要急着把造价智算做成“功能更多的平台”，而要先让一项真实造价任务能够从进入系统开始，一直走到复核、审批、交付和归档结束。

这条主线一旦成立，飞书、结算审核、数字化功能点法和管理驾驶舱都会自然接入；否则产品容易变成越来越多的页面、模块和按钮。

建议遵循以下长期原则：

1. 先完成任务闭环，再扩展业务场景。
2. 先形成可信底座，再增加自动化程度。
3. 先把飞书作为协同入口，不急于把全部业务搬进飞书。
4. 新场景使用独立知识库和规则包，不污染现有勘察测量规则。
5. 大模型继续只做解释、组织和草拟，不直接裁决价格、系数或审减结论。
6. 人工确认结果先进入候选层，经过治理和审批后再成为正式知识资产。
