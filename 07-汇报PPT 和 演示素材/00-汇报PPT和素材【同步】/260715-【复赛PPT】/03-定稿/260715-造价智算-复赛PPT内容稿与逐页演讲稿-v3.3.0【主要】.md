## 一、核心叙事判断

本稿不把“造价智算”只讲成一个网页工具，而是讲成：

> 一位刚刚完成第一项真实岗位任务的造价数智员工。

整场只讲清一条主线：

> 一个人面对一张复杂表格，为什么会陷入反复查找和机械核验；造价智算怎样依靠规则、经验、模型和人工边界完成这项任务；它又怎样从一个网页工具走进 WeACT 协同入口，开始具备数字员工的工作形态；最后，这个小切口如何扩展为可复制的造价能力体系。

叙事遵循以下原则：

1. 用一项真实岗位任务开场，让领导和评委先看到“它接了什么工作”。
2. 用连续的识别、规则裁决、人工复核、成果交付和 WeACT 协同闭环建立可信度。
3. 用业务链、数据链、角色链和 Task—Skill—Tool—Experience 说明能力如何沉淀。
4. 用组织价值、治理机制和成长路径说明这不是一次性工具，而是可持续运营的数智员工。
5. 不使用未经统一样本、统一口径和最终验收的数据制造结论。

## 新第 05 页｜造价智算不猜价格，让规则自己跑起来

**本页定位**：全场第一句技术价值主张。不是口号停顿页，而是用路线对比、执行链和责任边界证明“为什么可信”。

**大赛模块**：02 解决方案｜核心原则

### 屏幕文案
造价智算不猜价格  
它让规则自己跑起来

能确定的，给出依据  
不能确定的，标记待复核

最终判断与责任仍由专业人员承担

路线对比

- 大模型自由生成：靠语义猜测｜结果难以回查｜边界容易模糊｜不作为价格裁决路径
- 规则驱动执行：字段硬校验｜规则确定命中｜依据可回查｜作为专业计算主路径

执行链路

工程表输入 → 结构识别 → 字段硬匹配 → 规则命中 → 证据绑定 → 结果输出

- 确定项：结果 + 来源
- 非确定项：待复核 + 原因

证据资产：结构化计价库｜标准规则｜经验提示｜操作记录

规则负责可确定的部分｜模型负责解释与表达｜人员负责边界与结果
### 画面建议

- 上半区左侧是核心原则，右侧是“自由生成 / 规则驱动”的路线对比。
- 中部用六步执行链证明这不是一句口号，并明确确定项与非确定项如何分流。
- 下部放证据资产和责任分工；版面信息密，但用四个分组保持呼吸感。

### 现场讲稿

大模型很容易生成一个“听起来很像”的答案，但造价业务需要的是“能够负责”的答案。

所以我们的原则只有一句：**不猜价格，让规则自己跑起来。**自由生成靠语义猜测，适合表达，不适合作为价格裁决路径；造价智算走的是字段硬校验、规则确定命中、证据绑定和结果输出。能确定的给结果和依据，不能确定的明确待复核，最后的判断和责任仍然留给专业人员。这条原则决定了后面所有技术选择。

### 表达重点

- 本页讲 30—35 秒，重点讲路线差异和责任边界。
- 大字出现后先停顿，再说“不猜价格”。
- 这是评委最容易记住的一句话。

### 转场

> 原则明确以后，下一步要看它依靠什么工程架构，把规则、模型、数据和人工稳定地组织起来。

## 新第 06 页｜一套混合 AI 工程架构，支撑可复核的专业判断

**本页定位**：全场技术细节页。不是罗列技术名词，而是说明每一种技术如何服务专业规则、证据链和责任边界。

**大赛模块**：02 解决方案｜技术架构与工程实现

### 屏幕文案

交互与协同层

React + TypeScript + Vite 工作台｜WeACT / 飞书消息入口｜Tauri 2 桌面壳

Python 服务编排层

FastAPI 接口服务｜任务编排｜文件管理｜状态查询｜成果回传

专业处理层

- openpyxl：工作表探测、表头识别、列映射、合并单元格和 Excel 样式保留
- Python 规则引擎：字段归一、要素1—5与单位硬匹配、规则优先级、异常分流
- python-docx：根据模板生成正式 Word 报告

知识与证据层

结构化计价库｜标准规则表｜经验池｜标准资料｜报告模板｜SQLite 知识记忆与审计

大模型协同层

DeepSeek + 证据检索：知识问答｜规则解释｜待复核说明｜报告表达优化  
明确边界：不直接裁决最终价格和调整系数

完整技术链路

## 七、评委追问的统一口径

### 1. “这和普通 Excel 自动化有什么区别？”

建议回答：

> 普通自动化通常假定表结构固定，并把重点放在批量写值。造价智算处理的是规则、经验和责任边界：它先理解非标准表结构，再依据结构化计价库和规则判断三类关键数字，对不确定项进入人工复核，最后形成 Excel 和 Word 正式成果，并能通过 WeACT 接收任务、反馈进度和返回结果。它数字化的是岗位任务，不只是 Excel 操作。

### 2. “为什么不用大模型直接判断价格？”

建议回答：

> 造价业务需要可解释、可复核、可追责。大模型适合语言理解和解释，但不适合在缺少确定依据时自由生成最终价格。因此我们采用“规则裁决、模型解释、人工负责”，把概率能力放在它最擅长的位置，把专业裁决留给确定规则和专业人员。

### 3. “数字员工现在到底做到哪一步？”

建议回答：

> 目前已经跑通网页端的识别、匹配、复核、解释和 Excel / Word 成果闭环，也跑通了 WeACT 中的任务接收、有序处理、状态查询、风险查询和成果返回。身份权限、风险分派、多维任务台账、审批归档以及生产监控审计，是下一阶段规模化运营要补齐的能力。

### 4. “能不能推广到其他造价业务？”

建议回答：

> 可以复制方法，但不能把一个场景的规则直接套到所有场景。我们要复制的是标准任务包的建设方法：先定义岗位任务，再梳理业务链、数据链和角色链，把专业知识形成规则、技能、工具和经验资产，经过样例验收后再上线。这样可以逐步扩展到估算、概算、招标控制价、过程核查和结算辅助等任务。

### 5. “准确率和效率是多少？”

建议回答：

> 当前项目把关键样例逐行一致作为持续回归目标，但仍在处理已知回归差异，因此本次不使用未经最终验收的准确率和效率百分比。我们目前能够确定的是：完整任务链已经跑通，三类关键数字都有分层处理和复核边界，两类正式成果可以交付。下一阶段会在统一样本、统一口径和统一运行环境下形成可审计的性能与准确率指标。

### 6. “会不会替代造价人员？”

建议回答：

> 它替代的是重复查找、机械匹配和过程记录，不替代专业责任。系统负责把证据和风险送到人面前，专业人员负责定义规则、判断边界、处理风险和确认成果。我们设计的目标是增强专业岗位，而不是取消专业岗位。

- 第 3—4 页，让听众理解为什么要从岗位和标准任务包重新设计这项工作。
- 第 5—10 页，让听众相信它有规则、有技术、能复核、能交付。
- 第 11—12 页，让听众看到它已经开始以数智员工方式工作，并有真实闭环证据。
- 第 13—16 页，让听众理解它的组织价值、成长空间、治理路线和建设团队。

全场不靠夸张数据制造掌声，而靠三种力量抓住评委：

1. **真实任务的代入感。**
2. **真实闭环的可信度。**
3. **从小切口走向造价数字劳动力的想象力。**

最终要让领导和评委带走的，不是“这个系统有很多功能”，而是下面这句话：

> 造价智算已经证明，企业造价知识可以被训练成一名有规则、有边界、有交付、可管理的数智员工；今天跑通的是一张表，未来可以复制的是一整套造价能力。
